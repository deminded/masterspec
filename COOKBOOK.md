# masterspec — кукбук

Положи данные → запусти → получи → проверь. 👤 запускает человек, 🤖 работает внутри (субагенты).

## Папки
```
masterspec/  _input/  01-requirements/  02-specifications/  04-decisions/
  changes/<name>/  change.md  .research/  new/  route-run-<ts>.md
```

## Кто что запускает
👤 ЧЕЛОВЕК: `explore`, `derive`, `evolve`, `recover`, `apply-change` (через merge PR), `archive-change`.
🤖 ВНУТРИ (не дёргаешь): `gen` (элемент-генерация), создание change (шаг 0 внутри `evolve`), приёмщик, негатив-антагонист.
↔ `verify` — двойная роль: автоматически внутри `derive`/`evolve` И напрямую человеком (аудит готового слоя, миграция старой спеки).
Кодоген (`impl-plan`/`implement`) — за границей набора.

Флаги: `pass=linear|parallel` · `verify=core|full`.

## Решения
На развилке с реальным выбором узел-решение записывает решение: `adr-` (сквозное архитектурное, в `04-decisions/`) или `dr-` (локальное, рядом с артефактом на его слое). Запись нужна там, где был реальный выбор, на тривиальном — нет.

**«Немые» ситуации — `verify` помечает их как недоведённые (блок на ревью):**
- *немое решение* — был выбор из ≥2 вариантов, но decision record не записан;
- *немой вердикт* — узел в зоне влияния изменения не правлен И не помечен «не затронут, потому что…» (scope-fence);
- *немой подъём* — бизнес-эффект поднят флагом, но каскад вверх (правка верхнего узла + приёмочный критерий) не материализован.

## Кейс 1 — Генерация с нуля
1. 👤 Положи запрос + материалы в `_input/`.
2. 👤 Если фабрика поверх существующего кода — `explore <factory> roots=<пути>` → 🤖 разведчики → `.research/`. С нуля без кода шаг пропускается: `derive` соберёт контекст из запроса.
3. 👤 `derive <factory> layer=req pass=linear verify=core` → 🤖 gen (as/fn/nfr/rules/cdm) + приёмщик + verify. Выход: `01-requirements/` + `route-run-<ts>.md`.
4. 👤 Проверь `route-run-<ts>.md`: блокеры осей O4 (динамика), O5 (негатив). Закрой, ответь на дозапросы.
5. 👤 Согласуй: ревью по git-diff, merge PR = перевод `status: draft → actual`. `apply-change` тут НЕ нужен — derive писал прямо в дерево.
6. 👤 `derive <factory> layer=spec pass=parallel verify=full` → 🤖 gen (cmp/scn/api/data) + приёмщик + verify (O0/O6/O7). Выход: `02-specifications/`.
7. 👤 Согласуй (codegen_ready?) → кодоген (вне набора).

## Кейс 2 — Добавление функции
1. 👤 Опиши новую функцию (бизнес-запрос — текст или файл в `_input/`). `change.md` заведёт сам `evolve` на шаге 0; класть его руками не нужно.
2. 👤 `evolve <factory> entry=req --root=new:<новая-fn>` → 🤖 ДОБАВЛЕНИЕ узла: forward-каскад вниз (порождает scn/api/data/tc-acc) + gen + scope-fence + приёмщик + verify scope=change.
3. 👤 Получишь в `new/`: узел + рёбра + AC + вердикты соседей.
4. 👤 Проверь (нет немых вердиктов; инвариант переиспользуемых) → согласуй.

## Кейс 3 — Исправление поведения
1. 👤 Опиши изменение правила (бизнес-запрос). `change.md` заведёт `evolve` (шаг 0).
2. 👤 `evolve <factory> entry=rule --root=<rule-slug>` → 🤖 внутри.
3. 👤 Получишь правки + AC + материализованные подъёмы + реестр развилок.
4. 👤 Проверь (нет немых подъёмов; развилки в узел-решение с записью, не молча) → согласуй.

## Кейс 4 — Исправление спецификации смежника
1. 👤 Опиши новый внешний контракт (что прислал смежник). `change.md` заведёт `evolve` (шаг 0).
2. 👤 `evolve <factory> entry=ext --root=<api-slug>` → 🤖 внутри.
3. 👤 Получишь правки + AC обеих сторон (вход + успех + ОТКАЗ) + развилку техническое/бизнес с тестом + условный вердикт подъёма.
4. 👤 Проверь (ответ-отказ; узел-владелец нового значения; критерий подъёма = доменный инвариант) → согласуй.

## Кейс 5 — Восстановление и миграция
**Из документов или кода:**
1. 👤 `recover <factory> source=docs` (из ТЗ/презентаций/переписки) или `source=code roots=<пути>` (из репозитория) → 🤖 черновики артефактов с пометкой источника + «Белые пятна».
2. 👤 `verify` → закрой блокеры, ответь на вопросы → доведи через `derive`/`evolve`.

**Миграция спеки со старого формата:** структура та же (3 слоя), новое — в полноте (состояния, оракулы AC, квалификаторы, обе стороны контракта, записанные решения).
1. 👤 `verify scope=req`, затем `verify scope=spec` на существующей спеке → отчёт = backlog дыр с severity (пустые матрицы состояний, AC-не-оракулы, односторонние контракты, немые решения).
2. 👤 Закрывай точечно через `evolve`/`derive` — дополняй, не пересоздавай. Сначала блокеры (то, что мешает codegen_ready), остальное по мере касания.

## Как согласовать (на каждом гейте)
route оставляет результат на ветке/в карантине (сам он не коммитит, не пушит и не открывает PR — это делает человек или CI). При git-хостинге — человек открывает **PR**; без хостинга — достаточно ветки и ревью `git diff`. Аналитик ревьюит (`route-run-<ts>.md`: метрики, немые вердикты/подъёмы) и согласует (merge или отметка статуса) — агент свой результат сам не согласует. Что дальше — зависит от потока:
- **Генерация (`derive`):** черновики уже в дереве со `status: draft`. Merge PR = аналитик переводит `draft → actual` и перестраивает индекс (полная перегенерация подхватит `actual`-маркеры). `apply-change` НЕ вызывается.
- **Изменение (`evolve`):** черновики в `changes/<name>/new/`. После merge — `apply-change` вливает `new/` в дерево, артефакты → `actual`, индекс обновлён.

Ключ — разделение прав: согласование всегда за человеком (merge). Кто физически проставляет `actual` — зависит от потока: генерация — человек сменой статуса; изменение — `apply-change` при вливании уже согласованного change. Агент без предшествующего merge `actual` не ставит.

## На выходе (отчуждаемо от содержания)
`route-run-<ts>.md`: точность, полнота, немые вердикты/подъёмы, codegen_ready, остаток по типу. Без содержания — для приёмки и отчёта.
