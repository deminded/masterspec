# masterspec — кукбук

Положи данные → запусти → получи → проверь. 👤 запускает человек, 🤖 работает внутри (субагенты).

## Папки
```
masterspec/  00-source-data/  01-requirements/  02-specifications/  03-codemap/  04-decisions/
  changes/<name>/  change.md  .research/  new/  route-run-<ts>.md
```
Входящие материалы — в `00-source-data/` (нормализованный `src-<id>.md` + `_index.md` с provenance; сырьё рядом как есть). НЕ `_input/`.

## Кто что запускает
👤 ЧЕЛОВЕК: `explore`, `pack` (собрать guardrail-пакет из корп-документа — ДО derive, чтобы intake-чек-листы работали), `derive`, `evolve`, `recover`, `apply-change` (через merge PR), `codemap`, `archive-change`, `expose` (спроецировать фабрику-библиотеку в usage-контракт «как пользоваться, не устройство»).
🤖 ВНУТРИ (не дёргаешь): `gen`, создание change (шаг 0 внутри `evolve`), приёмщик, негатив-антагонист.
↔ `verify` — двойная роль: автоматически внутри `derive`/`evolve` И напрямую человеком (аудит слоя, миграция).
Кодоген (`impl-plan`/`implement`, либо внешний openspec design+plan) — за границей набора masterspec.

Флаги: `pass=linear|parallel` · `verify=core|full`.

## Решения
На развилке с реальным выбором узел-решение записывает: `adr-` (сквозное, в `04-decisions/`) или `dr-` (локальное, рядом с артефактом). На тривиальном — нет.
**«Немые» ситуации — `verify` помечает недоведёнными (блок на ревью):** немое решение (выбор ≥2 без record); немой вердикт (узел в зоне влияния не правлен и не помечен scope-fence); немой подъём (бизнес-эффект поднят, но каскад вверх не материализован).

---
**Опциональная подготовка (любой кейс):** если есть корпоративный стандарт/политика — сначала `pack <doc> factory=<factory>` собирает guardrail-пакет (manifest + карточки правил) и вписывает путь в `masterspec-config.yaml`. Тогда `intake` внутри derive/evolve прогоняет чек-листы этих правил, а `verify` добавляет их как оси. Без пакета intake вырождается в классификацию (тоже полезно).

Первый вопрос аналитика — **есть ли SSoT (спецификация)?** Кейсы разложены по этой оси.

## Кейс 0 — Генерация с нуля (нет ни кода, ни SSoT)
1. 👤 Положи запрос + материалы в `00-source-data/`.
2. 👤 `derive <factory> layer=req pass=linear verify=core` → 🤖 intake (классификация I/O, OE-грани для external, N/A-строка для internal) + gen (as/fn/nfr/rules/context/cdm/tc-acc) + приёмщик + verify. Молчание ≠ N/A: только `N/A — причина`, неизвестное = OPEN, блокирует spec_ready. Выход: `01-requirements/` + `route-run-<ts>.md`.
3. 👤 Проверь route-run (OE metrics, OPEN=0, без AC/tc-acc=0, оси O4/O5, репрезентативность fixtures). Закрой дозапросы.
4. 👤 Согласуй: ревью git-diff, merge PR = `draft → actual` (apply-change НЕ нужен — derive писал в дерево).
5. 👤 `derive <factory> layer=spec pass=parallel verify=full` → 🤖 gen (cmp/scn/alg/api/data/lp/tc-flt/tc-int), каждая APPLICABLE OE материализована. Выход: `02-specifications/`.
6. 👤 Согласуй (codegen_ready?) → кодоген (вне набора).

## Кейс 1 — Восстановление SSoT из кода и/или confluence (есть код, нет SSoT)
1. 👤 Собери источники в `00-source-data/`: выгрузку из Confluence (предпочтительнее статичной выгрузкой, чем ходить туда через MCP в рантайме) и/или укажи репозитории (в проекте их может быть несколько).
2. 👤 `recover <factory> source=code roots=<пути-репозиториев>` и/или `source=docs` (из confluence-выгрузки/ТЗ/переписки) → 🤖 черновики артефактов с пометкой источника (`provenance: docs:/code:`) + «Белые пятна» (§7: missing/negative/contract-gap).
3. 👤 `verify` → закрой блокеры, ответь на вопросы → доведи слои через `derive`/`evolve`.
4. 👤 (для полного code-слоя) `codemap` — трассы OE-граней → реальный код (file:symbol), `untraced_in_code=0`.

## Кейс 2 — Доработка без SSoT (частичное восстановление затронутого + доработка)
Спеки нет, а доработать надо — восстанавливаем SSoT ТОЛЬКО затронутого функционала, не всю фабрику.
1. 👤 Оконтурь затронутый функционал; источники (код нужных модулей, confluence по теме) → `00-source-data/`.
2. 👤 `recover <factory> source=code roots=<пути затронутого>` точечно → 🤖 черновики затронутых узлов + пятна.
3. 👤 `verify scope=...` на восстановленном → закрой блокеры.
4. 👤 Далее как Кейс 3 — доработка на (теперь частично готовом) SSoT.

## Кейс 3 — Доработка на готовом SSoT (полный цикл до archive)
1. 👤 Опиши изменение (бизнес-запрос — текст/файл в `00-source-data/`). `change.md` заведёт сам `evolve` (шаг 0), руками класть не нужно.
2. 👤 `evolve <factory> entry=req|rule|ext --root=<slug>` → 🤖 внутри: forward-каскад (entry=req: новая fn порождает scn/api/data/tc-acc), правка правила (entry=rule), новый внешний контракт (entry=ext, обе стороны+ОТКАЗ) + gen + scope-fence + приёмщик + verify scope=change. Выход: `changes/<name>/new/` + `change.md`.
3. 👤 Проверь (нет немых вердиктов/подъёмов; развилки записаны; инвариант переиспользуемых) → согласуй: merge PR.
4. 👤 `apply-change` → вливает `new/` в дерево, артефакты → `actual`, индекс обновлён. (Это отвечает на «после evolve — что дальше»: сначала apply, а НЕ сразу план.)
5. 👤 План реализации кода: `openspec design` + `openspec plan`, либо `masterspec-iml-plan` — удобным тебе способом → исполни план (кодоген/implement).
6. 👤 `codemap` — обнови трассы: изменённые/новые грани OE → реальный код (file:symbol), `untraced_in_code=0`.
7. 👤 `archive-change` — заархивируй завершённый change.

## Кейс 4 — Миграция (SSoT старого формата → 3 слоя полноты)
Структура та же (3 слоя), новое — в полноте: состояния, оракулы AC, OE-контракты, квалификаторы, обе стороны контракта, записанные решения.
1. 👤 `verify scope=req`, затем `verify scope=spec` на существующей спеке → отчёт = backlog дыр с severity (отсутствующие/OPEN OE, синтетика без репрезентативности, пустые матрицы состояний, AC-не-оракулы, односторонние контракты, немые решения).
2. 👤 Закрывай точечно через `evolve`/`derive` — дополняй, не пересоздавай. Сначала блокеры (мешающие codegen_ready), остальное по мере касания.
3. 👤 (замыкание code-слоя) `codemap` до `untraced_in_code=0`, штамп `meta_model_version` по достигнутому scope.

## Как согласовать (на каждом гейте)
route оставляет результат на ветке/в карантине — сам НЕ коммитит, НЕ пушит, НЕ открывает и НЕ мержит PR. Это делает **человек** (или CI): при git-хостинге человек открывает и мержит **PR**; без хостинга — ветка + ревью `git diff`. Аналитик ревьюит `route-run-<ts>.md` (метрики, немые вердикты/подъёмы) и согласует — агент свой результат сам не согласует.

**Кто физически ставит `draft → actual` — зависит от потока:**
- **Генерация (`derive`):** черновики уже в дереве со `status: draft`. Merge PR = человек переводит `draft → actual` (по-артефактно сменой статуса) и перестраивает индекс. `apply-change` НЕ вызывается.
- **Изменение (`evolve`):** черновики в `changes/<name>/new/`. После merge — `apply-change` вливает `new/` в дерево И сам проставляет `actual` (по каждому артефакту вручную ходить не надо).

Ключ — разделение прав: согласование (merge) всегда за человеком; агент без предшествующего merge `actual` не ставит.

## На выходе (отчуждаемо от содержания)
`route-run-<ts>.md`: точность, полнота, немые вердикты/подъёмы, codegen_ready, остаток по типу — для приёмки и отчёта.
