# metadata

- **Название:** masterspec
- **Автор:** Парфёнов
- **Версия:** 2.0.0
- **Версия мета-модели (контракт фабрик):** 3.0 — контракт соответствия, которому подчинено ОПИСАНИЕ фабрики (OE «Контракт живой эксплуатации» + tc-стандарт). Отдельная ось от версии пакета выше (та версионирует инструмент). Фабрика получает штамп `meta_model_version` как выход-сертификат при сертификации (`apply-change §13`); см. `references/meta-model-version.md` и `references/migration.md`.
- **Описание:** Набор скиллов для описания фабрик (автоматизированных систем) по трёхслойной мета-модели: требования → спецификации → кодовая база. Обеспечивает сквозную трассировку «что хотим → как устроено → где в коде» без дублирования между слоями, и строгую генерацию спецификаций, пригодных для перехода к коду.
- **Применимость:** Используйте, когда нужно спроектировать новую фабрику, восстановить описание из существующих документов или кода, провести вычитку полноты описания, внести изменение в описанную фабрику. Подходит для работы с паспортами АС/ФП, функциями, НФТ, компонентами, сценариями, алгоритмами, API, схемами данных, диаграммами, профилями нагрузки, справочниками, тест-кейсами и решениями (ADR).
- **Состав:** kernel-справочник `masterspec` (мета-модель, шаблоны, дисциплина слоёв, паттерны процесса) и скиллы-операции: `explore` (разведка кодовой базы), `derive` (породить слой; layer=req|spec), `verify` (вычитка по осям O0–O7; scope=req|spec|change), `gen` (сгенерировать один артефакт; type=…), `evolve` (изменить фабрику; entry=req|rule|ext), `recover` (восстановить из документов/кода; source=docs|code), `apply-change`/`archive-change` (влить/архивировать изменение).
- **Использование:** Положите директорию набора в каталог скиллов вашего агента (например `~/.claude/skills/`). Вызывайте через естественный запрос («опиши фабрику X», «проверь слой требований», «внеси изменение Y») либо явно укажите скилл и параметр («derive layer=req», «verify scope=spec»). Скилл сам подберёт шаблоны из `templates/`, мета-модель из `meta_model.md` и дисциплину слоёв из `references/layer-discipline.md`.
- **Зачем:** Без дисциплины слоёв описания расползаются — функциональные требования смешиваются с именами компонентов, спецификации с именами классов, трассировка до кода теряется. Набор вводит жёсткие границы слоёв, направление ссылок снизу вверх, capabilities (`cap-*`), промышленный tc-стандарт, критичностно-взвешенное покрытие, каталоги отказов (`tc-flt-`) и verify-report с телеметрией стоимости/возраста/автоматизации.
