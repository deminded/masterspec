# Реестр нотаций сценария

`notation` — атрибут фронтматтера `scn-*.md`, задаёт форму поведенческой части сценария: разделы
«Последовательность шагов»/«Ветвления» шаблона `templates/tpl-scenario.md` (для `yaml-graph`) либо
диаграмма/таблица правил (для остальных нотаций) — буквальной секции «Поведение» в шаблоне нет.
Строка валидируется по этому реестру, а не по зашитому `enum` в схеме фронтматтера. Это позволяет
добавлять новые нотации через ADR без правки схемы (тот же приём открытого списка, что у
`boundary-registry` — реестр значений `api`, `references/boundary-registry.md`).

> **Реестр ЗНАЧЕНИЙ, не реестр ЭКЗЕМПЛЯРОВ.** Этот файл перечисляет допустимые *значения* атрибута
> `notation` — он не заводит параллельный список сценариев-файлов и потому не конфликтует с
> запретом «не плодить реестры экземпляров» (`patterns/verification-axes.md`, детерминированный
> pre-gate: `expected` берётся из дерева/индекса, не из нового реестра — иначе риск O0). Экземпляры
> по-прежнему резолвятся из дерева и индекса.

## Зачем реестр, а не единственная нотация

Не всякое поведение описываемо сценарием-сиквенсом. Вытягивающая (pull) логика на параллельных
процессах не описывается ни последовательностью вызовов, ни стейт-машиной — нет глобального
порядка, есть только локальные правила реакции участников + инварианты потока. Насильно натянутый
единый формат («всегда граф шагов») в таком случае не описывает систему, а искажает её — поэтому
вариативность: набор нотаций, из которых под конкретный сценарий выбирается подходящая.

Выбор ДОПУСТИМЫХ нотаций под конкретный проект (guardrails: что разрешено/предпочтительно в данной
фабрике) — задача **Фазы 5**. Инфраструктура guardrails в 3.0 уже заведена (паттерн
`patterns/guardrails.md`, параметр `guardrails=` в `derive`/`gen`/`evolve`/`verify`), но per-project
рестрикция набора нотаций через неё — ещё не обкатана (ROADMAP). До неё `gen` выбирает нотацию по
таблице «тип поведения → нотация» ниже.

## Связь с машинным сайдкаром

Наличие машинного сайдкара у сценария определяется **нотацией**, а не «природой контракта» (как для
`api`/`data` в `patterns/sidecar-formats.md`):

- `yaml-graph`, `pull-rules` — тело компаньона несёт всю информацию инлайн (нумерованный список +
  ссылки / таблица правил); отдельный машинный сайдкар для этого **не заводится** — резолвинг ссылок
  и проверка полноты идут prose/LLM-разбором (см. Verify у `yaml-graph` ниже), не парсингом формальной
  схемы.
- `sequence`, `workflow` — тело несёт диаграмму как fenced-блок; при потребности в инструментальной
  проекции она выносится сайдкаром (`sidecar_format` + `sidecar`, см. `patterns/sidecar-formats.md`).
- `bpmn` — тело НЕ инлайн: машинный XML живёт только сайдкар-парой (обязательно), по образцу
  schema-first `api`/`data`.

Формат сайдкара фиксируется во фронтматтере компаньона теми же полями, что для `api`/`data`:
`sidecar_format: <id>` и `sidecar: <slug>.<ext>` (единый механизм пары — `patterns/sidecar-formats.md`).
Отдельного `behavior:`-блока (как было в v2) сценарий не заводит — форма унифицирована с api/data.

## Семантический инвариант — поверх ЛЮБОЙ нотации

Независимо от формы диаграммы или таблицы, каждый участник и каждый шаг сценария обязаны
резолвиться в `-> cmp-<slug>/cap-<name>` (и `-> api-<slug>` для внешних вызовов):

- для **`yaml-graph`** и **`pull-rules`** это нативно — ссылка стоит прямо в строке шага/правила;
- для **`sequence`**, **`workflow`**, **`bpmn`** — обязательна секция «Участники» в теле `scn-`
  (см. `templates/tpl-scenario.md`) с явным маппингом «имя на диаграмме → `-> cmp-<slug>/cap-<name>`».

Диаграмма без резолвинга участников — иллюстрация, а не спецификация: она не проходит **O2** (все
ссылки `-> slug` ведут на существующий артефакт + двусторонний cross-check `scn↔api`) и не годится
как вход для следующего слоя.

**Правило расширения:** новая нотация добавляется через ADR (обоснование: какой класс поведения не
покрывают существующие нотации), затем регистрируется здесь.

## Допустимые значения

### `yaml-graph` — дефолт для новых сценариев

- **Зона применимости:** поведение с чётким порядком шагов; безопасный выбор по умолчанию и при
  сомнении в выборе нотации.
- **Форма в теле scn:** разделы «Последовательность шагов» + «Ветвления» шаблона
  `templates/tpl-scenario.md` — нумерованный markdown-список, где каждый шаг = узел с инлайн-ссылкой
  `-> cmp-<slug>/cap-<name>` (опц. `-> api-<slug>` для внешнего вызова), а «Ветвления» = развилки с
  ≥2 явными исходами. Это НЕ отдельный YAML-блок `nodes`/`edges` — имя «yaml-graph» условное (граф
  резолвится из нумерации + ссылок). Шаг может передавать управление другому сценарию
  (`-> scn-<slug>` как узел-переход) — допустимая ссылка.
- **Verify:** резолвинг ссылок и cross-check `scn↔api` — **O2**; полнота ветвлений (у каждой
  развилки ≥2 исхода, нет молчаливой else-ветки) — **O4a**. Обе проверки — prose/LLM-разбор
  нумерованного markdown-списка (формальной схемы `nodes`/`edges` для машинного парсинга нет);
  сайдкар не нужен.

### `sequence` — PlantUML / Mermaid sequence diagram

- **Зона применимости:** запрос-ответные взаимодействия с ГЛОБАЛЬНЫМ порядком — важно не только
  «что вызывается», но «в каком порядке относительно всех участников».
- **Форма в теле scn:** fenced-блок ```` ```plantuml ```` (`@startuml`/`@enduml`) или
  ```` ```mermaid ```` (`sequenceDiagram`); участники — `participant`/`actor`; сообщения —
  синхронные/асинхронные вызовы. При выносе в сайдкар: `sequence`-формат
  (`<slug>.sequence.mmd`/`.puml`), см. `patterns/sidecar-formats.md`.
- **Verify:** парсимость синтаксиса диаграммы (структурная валидность — как для любого машинного
  файла, `masterspec-gen` §структурная валидность); секция «Участники» с маппингом на `cmp/cap` —
  **O2** (см. семантический инвариант выше).

### `workflow` — Mermaid flowchart

- **Зона применимости:** ветвящийся поток управления с решениями, где важна логика «если — то»,
  а не обмен сообщениями между акторами.
- **Форма в теле scn:** fenced-блок ```` ```mermaid ```` (`flowchart`/`graph`); узлы-решения —
  ромб; каждый исход ветвления — подписанное ребро. Сайдкар — `<slug>.flow.mmd` при потребности.
- **Verify:** парсимость Mermaid-синтаксиса (структурная валидность); секция «Участники» — **O2**;
  полнота ветвлений — у каждого узла-решения все подписанные исходы, без молчаливой else-ветки —
  **O4a**.

### `bpmn` — BPMN 2.0

- **Зона применимости:** сложный бизнес-процесс (pools/lanes, промежуточные/граничные события,
  компенсации, несколько ролей), когда BPMN уже принят в организации как язык описания процессов.
- **Форма в теле scn:** машинный XML — НЕ инлайн, а **сайдкар-парой** рядом с компаньоном `.md`
  (`<slug>.bpmn` + `scn-<slug>.md`), по образцу schema-first `api`/`data`. Фронтматтер компаньона:
  `notation: bpmn`, `sidecar_format: bpmn-2.0`, `sidecar: <slug>.bpmn`. Компаньон несёт
  семантический маппинг (lane/pool/task → `cmp/cap`), а не повтор процесса словами — повтор был бы
  второй нормативной редакцией.
- **Verify:** пара компаньон+сайдкар = одна нормативная редакция (нет дублирующего описания
  процесса словами) — **O0**; синтаксис BPMN 2.0 XML (well-formed + схема — структурная валидность
  сайдкара); секция «Участники» с маппингом lanes/pools/tasks на `cmp/cap` — **O2**.

### `pull-rules` — вытягивающая/параллельная логика без глобального порядка

- **Зона применимости:** параллельные процессы, где каждый участник тянет работу по своему
  локальному правилу реакции — общего порядка вызовов нет, и вводить его было бы искажением
  (pull-based пайплайны, очередь/воркеры, реактивные потоки). Ни sequence, ни стейт-машина такое
  не описывают: нет единого «кто за кем».
- **Форма в теле scn:** таблица правил реакции:

  | `rule-id` | Участник (`-> cmp/cap`) | Тянет что | Откуда | При каком условии | Производит что | Куда | Конкуренция |
  |---|---|---|---|---|---|---|---|
  | `pr-01` | `-> cmp-<slug>/cap-<name>` | ... | ... | ... | ... | ... | `exclusive` \| `multi-ok` |

  плюс секция «Инварианты потока»: WIP-лимиты, гарантии доставки/порядка (там, где они есть),
  условия завершения потока, арбитраж пересекающихся правил (см. ниже).

  **Полнота условий** — обязательна: для каждого правила явно описано, что происходит, когда
  условие НЕ выполнено (ждёт / пропускает / таймаут). Правило без этого — дыра, как пустая ячейка
  матрицы состояний.

  **Идентификация и арбитраж конкуренции** — обязательны:
  - каждое правило получает уникальный `rule-id` (например `pr-01`);
  - каждое правило несёт атрибут конкуренции: `exclusive` (один work item обслуживает ровно один
    потребитель) | `multi-ok` (допустима мультиобработка одного work item несколькими правилами;
    тогда обязательна пометка требований идемпотентности потребителя — что повторная/параллельная
    обработка одного и того же item безопасна);
  - **инвариант арбитража:** если условия ≥2 правил пересекаются над одним источником («Откуда») —
    пересечение ОБЯЗАНО объявить разрешение конфликта явно: приоритет между `rule-id` (кто тянет
    первым), эксклюзивность (`exclusive` снимает пересечение отбором «первый взял — остальные
    пропускают») или мультиобработку (`multi-ok` + идемпотентность). Пересечение условий над одним
    источником без объявленного арбитража — семантическая дыра, даже если каждое правило по
    отдельности полно по условиям «что если не выполнено».
- **Verify:** резолвимость участников — нативно, ссылка в столбце «Участник» (**O2**); наличие и
  непустота секции «Инварианты потока» + присутствие `rule-id`/атрибута конкуренции у каждого
  правила — структурная полнота элемента (**O1**); полнота условий «что если не выполнено» и
  объявленный арбитраж для каждой пары пересекающихся правил — прогон содержания (**O4a**); машинно
  форма нативна в теле, сайдкар не заводится.

## Таблица выбора: тип поведения → нотация

Используется `gen` до обкатки guardrails-рестрикции (Фаза 5) — как источник выбора нотации, если
она не задана явно в плане/фокус-наборе:

| Тип поведения | Нотация |
|---|---|
| Глобальный порядок, запрос-ответ между участниками | `sequence` или `yaml-graph` |
| Ветвящийся поток управления (решения, условия) | `workflow` или `yaml-graph` |
| Параллельная вытягивающая логика без общего порядка | `pull-rules` |
| Сложный бизнес-процесс, BPMN уже принят в организации | `bpmn` |
| Дефолт / сомнение в выборе | `yaml-graph` |

## Как добавить новую нотацию

1. Открыть ADR с обоснованием: какой класс поведения не описывается существующими нотациями,
   почему нельзя выразить через `yaml-graph`/`pull-rules`.
2. После принятия ADR — добавить запись в раздел «Допустимые значения» (зона применимости, форма,
   verify-оси) и строку в таблицу выбора.
3. Обновить guardrails-конфигурацию (Фаза 5) — когда per-project рестрикция нотаций будет обкатана.
