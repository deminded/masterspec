# Реестр значений boundary

`boundary` — атрибут фронтматтера `api-*.md`, задаёт уровень границы вызова и через него профиль
безопасности. Строка валидируется по этому реестру, а не по зашитому `enum` в схеме фронтматтера —
как `notation` у scn (`scenario-notation-registry.md`). Новый уровень добавляется через ADR без
правки схемы.

> **Реестр ЗНАЧЕНИЙ, не реестр ЭКЗЕМПЛЯРОВ.** Перечисляет допустимые *значения* атрибута `boundary`,
> не заводит параллельный список api-файлов — не конфликтует с запретом «не плодить реестры
> экземпляров» (`patterns/verification-axes.md`, pre-gate покрытия). Экземпляры резолвятся из дерева.

## boundary ≠ scope

`scope` (`internal | external`) — про РАЗМЕЩЕНИЕ: каталог `04-apis/internal|external` и срез индекса.
`boundary` — про УРОВЕНЬ ГРАНИЦЫ вызова и требуемый профиль безопасности. Оси **ортогональны** и не
выводятся одна из другой: внутренний по размещению api может быть `inter-factory` (service-to-service
между фабриками), а не `intra`. Поэтому `boundary` — отдельный атрибут, не производная от `scope`.

## Допустимые значения

| Значение | Смысл | Профиль безопасности (ось применимости guardrails) |
|---|---|---|
| `intra-factory` | Между компонентами одной фабрики. | Relaxed: внешняя аутентификация не требуется (доверенный периметр процесса/фабрики). |
| `inter-factory` | Между фабриками внутри организации. | Service-to-service auth: mutual TLS / service-mesh identity. |
| `perimeter` | Внешний контур — API вне организации. | OAuth2/mTLS, rate-limiting, полный OWASP API Security Top 10. |

> **Протоколы в столбце «Профиль безопасности» — ИЛЛЮСТРАТИВНЫЙ пример** типового уровня строгости
> для интуиции (mTLS, OAuth2, OWASP Top 10 — то, что ОБЫЧНО стоит за каждым уровнем), а не
> нормативное правило kernel. Единственное, что задаёт kernel, — качественную ось применимости и
> монотонность строгости `intra-factory → inter-factory → perimeter`. Конкретные протоколы, версии и
> пороги для ДАННОГО проекта приходят security-guardrail-пакетом (`masterspec-pack`), не отсюда — это
> единственное место, где такие протоколы вообще упоминаются (`tpl-api.md`, `patterns/verification-axes.md`
> ссылаются сюда, а не повторяют перечень).

## Как boundary применяется — через guardrails, не зашитым правилом

Профиль безопасности НЕ зашивается в kernel (по `patterns/guardrails.md`: корпоративные нормы —
ИБ-стандарт, OWASP, API-стандарт — в kernel не входят, kernel даёт только механизм). Разделение:

- **kernel даёт механизм:** этот реестр (ось применимости) + атрибут `boundary` во фронтматтере api +
  ключ `boundary` в словаре селекторов `applies_to` (`patterns/guardrails.md`, `masterspec-pack`) —
  чтобы security-правила резолвились по уровню границы + нормативную связь «глубина проверки O5
  растёт `intra-factory` → `inter-factory` → `perimeter`».
- **нормы поставляет пакет:** конкретные правила-карточки (OWASP API Top 10, требования mTLS,
  rate-limit пороги) собираются `masterspec-pack` из корпоративного ИБ/API-стандарта в guardrail-пакет
  с `applies_to: {element_type: api, boundary: perimeter}` и подключаются per-project. Без пакета
  реестр задаёт только ось (значение атрибута) и дифференцирует глубину O5 качественно, не буквой
  стандарта.

Точки внедрения (штатные для guardrails): `gen` — conformance-пункты активных правил в приёмочные
критерии api; `verify` — чек-листы активных правил как дополнительные оси (глубина O5 по `boundary`);
машинная проекция — Spectral-рулсет по `boundary` над sidecar `.yaml` (ROADMAP, Фаза 3).

## Как добавить новый уровень

1. Открыть ADR с обоснованием: чем новый уровень границы отличается от существующих (какой профиль
   безопасности он несёт и почему не покрывается intra/inter/perimeter).
2. После принятия ADR — добавить строку в таблицу «Допустимые значения» (смысл + профиль).
3. Если под уровень нужны конкретные нормы — собрать/дополнить guardrail-пакет (`masterspec-pack`),
   селектор `applies_to.boundary: <новый уровень>`.
