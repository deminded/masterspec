# Формат change — diff-блоки vs файлы в new/

Канонический формат change.md и выбор формата для каждой правки. Читают: `evolve` (шаг 0 — заполнение change.md) и `apply-change` (мерж).

change.md состоит из шапки (статус, фабрика) и 8 секций: §1 цель, §2 таблицы (2.1 MODIFIED / 2.2 ADDED / 2.3 REMOVED), §4 diff-блоки, §5 список new/, §6 обоснования REMOVED, §7 влияние на ссылки, §8 критерии приёмки.

---

## 1. Правило выбора формата правки

| Характеристика правки | Формат |
|-----------------------|--------|
| Одна строка/bullet (AC, правило, строка таблицы, описание в компоненте) | diff-блок §4, тип `modify-bullet` |
| Добавление одного bullet в существующий список | diff-блок §4, тип `modify-bullet` (`ДО:` = якорный соседний bullet) |
| Замена раздела (`## ...` / `### ...`) целиком | diff-блок §4, тип `replace-section` |
| Новый подраздел (`### cap-<имя>`) внутри существующего раздела | diff-блок §4, тип `add-subsection` |
| Новый артефакт целиком (fn-, cmp-, scn-, alg-, nfr-, rules-, cdm-, dict-, adr-, dr-, data-, api-, tc-acc-, tc-int-, ui-view-, nav-) | файл `new/<slug>.md` + строка в §2.2 |
| Новый артефакт кодового слоя (repo-map, cmap-, trace-, dmap-) | тот же `new/<slug>.md` + строка §2.2, фронтматтер `generated: true`. Обычно codemap не заводят через change вручную — он порождается `recover source=code`/`derive`; через change — только если правится осознанно. |
| Удаление артефакта целиком | строка в §2.3 + обоснование §6; файл не трогаем до apply |
| Правка `00-masterspec-index.md` | **не вручную** — `apply-change` делает полную перегенерацию §3–§6 по файлам фабрики (`index-canonical.md`). Diff-блоки по §3–§6 индекса отклоняются на ревью. Исключение — §1 «Паспорт» / §7 «Белые пятна» через `modify-bullet`/`replace-section`. |

Три типа diff-блока (`modify-bullet`, `replace-section`, `add-subsection`) покрывают все правки существующих артефактов. Большая замена — это `replace-section`; если правок много — разбей change на несколько.

## 2. Структура diff-блока

Каждый блок в `change.md §4`:

```markdown
### 4.N. <slug> — <что меняем одной строкой>

**Файл**: <путь относительно masterspec/>
**Раздел**: <заголовок: `## ...` → `### ...`>
**Тип правки**: modify-bullet | replace-section | add-subsection

ДО:
```
<точный текст из целевого файла, который будет заменён>
```

ПОСЛЕ:
```
<новый текст>
```
```

### 2.1. modify-bullet
Замена точной строки (bullet списка или строка текста). `apply-change` ищет текст из `ДО:` в указанном разделе строгим совпадением (после нормализации leading whitespace); не найден — конфликт (`merge-workflow.md`).

```
**Файл**: 01-requirements/02-functions/fn-send-notification.md
**Раздел**: ## Критерии приёмки
**Тип правки**: modify-bullet

ДО:
- AC-03. Система возвращает статус доставки в течение 30 секунд.

ПОСЛЕ:
- AC-03. Система возвращает статус доставки в течение 30 секунд.
  При таймауте внешнего канала — ретрай до 3 раз с backoff 1s/5s/15s.
```

### 2.2. replace-section
Замена содержимого раздела целиком. `apply-change` находит раздел по заголовку, удаляет всё между ним и следующим заголовком того же уровня, вставляет `ПОСЛЕ:` (без самого заголовка). `ДО:` — можно оставить пустым или продублировать старое для истории ревью.

```
**Файл**: 01-requirements/03-nfr/nfr-delivery-latency.md
**Раздел**: ## Целевые показатели
**Тип правки**: replace-section

ПОСЛЕ:
- p50: 500 ms
- p95: 2000 ms
- p99: 5000 ms
- timeout: 10000 ms
```

### 2.4. set-frontmatter
<!-- Единственная операция, меняющая ФРОНТМАТТЕР. Заведена потому, что переход владения
     (imported → authored) иначе недостижим: остальные типы правят только тело. -->

Меняет ОДНО поле фронтматтера существующего артефакта. Разрешённые поля — только те, что выражают
СМЕНУ ВЛАДЕНИЯ или ЗРЕЛОСТИ, и каждое требует основания:

| Поле | Когда допустимо | Обязательное основание |
|---|---|---|
| `contract_origin: imported → authored` | внешний владелец исчез / контракт стал нашим legacy | решение `dr-` в том же change (акт присвоения) |
| `contract_source` | ре-импорт новой ревизии внешнего контракта | новый `{uri, revision, sha256}`, дайджест сходится с файлом |

```
#### 4.N. api-<slug> — присвоение контракта (imported → authored)
- **Тип:** set-frontmatter
- **Поле:** contract_origin
- **Было:** imported
- **Стало:** authored
- **Основание:** -> dr-adopt-<slug> (внешний реестр выведен из эксплуатации 2026-07-01)
```

Прочие поля фронтматтера через change НЕ меняются: `status` ставит lifecycle (`apply-change`/`accept`),
`slug`/`type`/`factory` неизменны, `updated` проставляется при применении.

### 2.3. add-subsection
Добавление нового подраздела в существующий раздел. `apply-change` находит раздел, вставляет `ПОСЛЕ:` в конец раздела (после последней подсекции, перед границей). Прежде всего — новые `cap-*` внутри существующего `cmp-`.

```
**Файл**: 02-specifications/01-components/cmp-delivery-router.md
**Раздел**: ## Возможности
**Тип правки**: add-subsection

ПОСЛЕ:
### cap-retry — ретрай неудачных отправок

**Вход**: неуспешный исход отправки (-> scn-retry-flow).
**Поведение**: повторить отправку через тот же канал с backoff.
**Ограничения**: максимум 3 попытки; backoff 1s/5s/15s.
**Ошибки**: `retry-budget-exhausted` при исчерпании попыток.
```

## 3. Файлы в `new/` (новый артефакт целиком)

- Имя файла = slug (с префиксом: `fn-`, `cmp-`, `scn-`, …).
- YAML-фронтматтер обязателен. `status` нового артефакта в `new/`:
  - артефакты слоёв (fn/cmp/scn/api/data/…) — `status: draft`;
  - новое решение `adr-` — `status: proposed` (фаза предложения; в `accepted` переводит человек после принятия, не при создании); локальное `dr-` — `status: accepted` сразу (фазы предложения нет).
- Содержимое — по шаблону `masterspec/templates/tpl-*.md`, HTML-комментарии удалить.
- В change.md §2.2 — строка с этим slug; в §5 — продублирован список для ревью.

### 3.1. Машинный сайдкар (режим компаньона) — api/data и scn/alg по нотации
Контракт `api`/схема `data` (OpenAPI/AsyncAPI/JSON Schema), а также `scn`/`alg` в диаграммной или
таблично-решающей форме (`sequence`/`workflow`/`bpmn`/`decision-table`; реестр —
`scenario-notation-registry.md`, форматы — `patterns/sidecar-formats.md`) могут существовать в
машинном виде, нужном внешним процессам (кодоген, валидация, mock, диаграммеры). Тогда артефакт —
**ПАРА файлов с одним slug, рядом в одной директории**:
- `<slug>.<ext>` — сама машинная спека; **формат и имя объявлены во фронтматтере компаньона** полями
  `sidecar_format: <id>` и `sidecar: <slug>.<ext>`. Расширение НЕ зашито в `.yaml`/`.json`: это может
  быть `.openapi.yaml`, `.asyncapi.yaml`, `.schema.json`, `.bpmn`, `.dmn`, `.mmd`, `.puml`, … — весь
  lifecycle читает поле `sidecar:`, а не гадает по расширению;
- `<slug>.md` — **компаньон, обязателен**: masterspec-фронтматтер (`type`, `slug`, `factory`,
  `status`, для api — `scope`, плюс `sidecar_format`/`sidecar`), назначение, ограничения, **для api —
  маппинги полей** (логические правила наполнения «поле ← источник», в OpenAPI/AsyncAPI не
  выражаются, живут в компаньоне — single-source O0) и ссылка на сайдкар.

> **Нормативное владение фрагментами пары.** У `api`/`data` и у `bpmn` — schema-first: машинный
> сайдкар нормативен по СТРУКТУРЕ (операции, схемы, типы; для bpmn — процесс), компаньон нормативен по
> СЕМАНТИКЕ (маппинги полей «поле ← источник», SLA/идемпотентность/ретраи, для bpmn — маппинг
> lane/pool/task → `cmp/cap`; плюс провенанс и связи). Повтор структуры словами в компаньоне = вторая
> нормативная редакция, нарушение O0. Полное правило и `contract_origin` (imported/authored) —
> `patterns/sidecar-formats.md §Нормативное владение`.
>
> У `sequence`/`workflow`/`decision-table` тело машинно-парсимо и живёт В компаньоне; сайдкар —
> производная проекция для инструментов, правится тело.

Правила:
- В `new/` кладутся **ОБА файла**. `apply-change` копирует пару в целевую директорию.
- Индексация, `verify` и определение статуса идут по **компаньону** (`.md` несёт фронтматтер); машинный файл метаданных masterspec не содержит.

**ДОБАВЛЕНИЕ новой пары** (артефакта ещё нет): строка в §2.2 ADDED, оба файла в `new/`; `apply-change` копирует пару, проверяя отсутствие коллизии ОБОИХ целевых путей (`<slug>.md` И объявленного `<sidecar>`); `sidecar:` — только локальный basename рядом.

**ИЗМЕНЕНИЕ существующей пары** (артефакт уже есть):
- *машинный сайдкар* меняется ЗАМЕНОЙ ЦЕЛИКОМ — строка в §2.1 MODIFIED (slug + путь = объявленный `sidecar:` компаньона, любой формат), новая версия кладётся в `new/<sidecar>`. `apply-change` перезаписывает существующий сайдкар (целевой ДОЛЖЕН существовать — это MODIFIED, а не ADDED с проверкой коллизии). Diff-блоки §4 по машинному сайдкару (любого формата — yaml/json/xml/текст-нотация) НЕ применяются — частичный merge структурированных форматов запрещён намеренно (риск некорректного слияния);
- *компаньон `.md`* меняется обычными diff-блоками §4 (или, при крупной переработке, `replace-section`).

При мерже `apply-change` копирует/заменяет файл(ы) в целевой директории (по `type:` через `artifact-routing.md`) и проставляет финальный статус: артефакты слоёв → `actual` (merge PR уже был согласованием); `adr-`/`dr-` сохраняют свой решенческий статус.

## 4. Запреты

**Лаконичность (анти-размазывание).** Одна суть изменения НЕ пересказывается в каждом затронутом артефакте:
- каноническая формулировка поведения живёт в ОДНОМ месте слоя (поведенческое правило → rules-артефакт; ссылка `-> RULE-<id>` из других артефактов легитимна);
- важное решение → `dr-*`; обычное изменение описано в самом change (§1), мотивация — в route-run;
- АРТЕФАКТЫ НЕ ССЫЛАЮТСЯ НА CHANGE и не носят тикет-рефов в bullet'ах: после наката эта информация в спеке не нужна — трасса восстанавливается через git blame (коммит → change + run внутри коммита);
- MODIFIED-bullet = чистая локальная дельта своего измерения (state-машине — переход, глоссарию — значение), без повторения условий и мотивации;
- пояснения хода — в route-run, не в артефактах.
Нарушение ловит ось `D-DUP` (`patterns/verification-axes.md`): одинаковые фразы длиной >8 слов в ≥3 артефактах одного change — дефект.

- **Не теряй фокус change.** Правка > 5 артефактов в разных слоях — подумай о разбиении. Исключение: согласованное архитектурное решение (ADR + несколько `cmp-`/`scn-`).
- **Не добавляй новый артефакт diff-блоком** — только через `new/<slug>.md`.
- **Не нарушай дисциплину слоёв в теле diff-блока** (`masterspec/references/layer-discipline.md`).
- **Не правь YAML-фронтматтер целевого файла diff-блоком** (миграция типа/slug/`criticality` — через remove + new). `updated:` `apply-change` обновляет сам.

### 4.1. Migration-класс

Change, переводящий фабрику на новую версию мета-модели (добавить обязательные поля/секции: `criticality`, `Контракт живой эксплуатации` и т.п.), помечается в шапке `> **Класс**: migration`. Особенности:
- обогащение СУЩЕСТВУЮЩИХ артефактов без смены slug (фронтматтер + новая `##`-секция) не выражается diff-блоками, а remove + new того же slug конфликтует со smoke-check; путь применения зависит от зрелости фабрики (`references/migration.md §3`): `draft` → прямо в дерево + сертификация; `actual` → in-place artifact-rewrite (расширение) или временное понижение в draft;
- при apply migration-класс **сохраняет объявленный `status:`** из `new/` (не форсит `actual`) — миграция меняет форму, не зрелость (`apply-change merge-workflow.md §5.5`);
- версия проставляется отдельно, сертификацией по зелёным детекторам (`apply-change §13`, `meta-model-version.md`, `migration.md`) — не полем в change.md.

## 5. Валидация change.md перед сохранением

`evolve` (шаг 0) перед записью проверяет:
1. Шапка заполнена, статус = `На согласовании`.
2. §1 заполнен (цель + приоритет + слои).
3. §2 — хотя бы одна непустая таблица.
4. Для каждой строки §2.1: если путь `.md` — есть diff-блок в §4; если путь = объявленный `sidecar:` компаньона (любой формат) — diff-блока НЕТ, есть файл-замена `new/<sidecar>` (sidecar-replace, §3.1).
5. Для каждой строки §2.2 — файл `new/<slug>.md` создан, YAML-фронтматтер валиден.
5a. Для машинной пары (api/data/scn/alg с sidecar) — компаньон `<slug>.md` И объявленный сайдкар (`sidecar:`) присутствуют в `new/`, и сайдкар парсится по своему `sidecar_format` (yaml/json — полноценно; xml для bpmn/dmn — well-formed; текстовые нотации mermaid/plantuml — непустота без markdown-огородок). Битый/отсутствующий сайдкар ловится здесь, на создании change, а не в конце мержа.
6. §7 заполнен или `Нет изменений.`
7. §8 критерии приёмки на месте.
8. Дисциплина слоёв проверена для каждого MODIFIED и ADDED.

Нарушение — не сохранять, показать проблемы пользователю.
