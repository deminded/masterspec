# Машинные сайдкары: форматы по типу артефакта

## Зачем
Артефакт api/data (а где полезно — scn/alg) существует как **пара**: человекочитаемый компаньон
`<slug>.md` (истина, провенанс, метаданные) + машинный сайдкар `<slug>.<ext>` (та же информация в
стандартной нотации, пригодной для инструментов: генераторов клиентов, валидаторов, диаграммеров).
Компаньон — источник истины; сайдкар — производная проекция. При расхождении правится компаньон,
сайдкар перегенерируется.

## Формат по типу (дефолт + расширяемость)
Формат выбирается по ПРИРОДЕ контракта, а не жёстко по типу. Выбранный формат фиксируется во
фронтматтере компаньона: `sidecar_format: <id>` и `sidecar: <slug>.<ext>`.

| Тип | Дефолт | Когда | Файл |
|---|---|---|---|
| `api` (sync/REST) | **OpenAPI 3.x** | запрос-ответ, HTTP-ресурсы | `<slug>.openapi.yaml` |
| `api` (event-driven) | **AsyncAPI 2.x** | очереди, вебхуки, стриминг, поллинг (Telegram, IMAP/SMTP, брокеры) | `<slug>.asyncapi.yaml` |
| `data` | **JSON Schema 2020-12** | схема сущности/конфига | `<slug>.schema.json` |

Таблица покрывает `api`/`data` — их формат выбирается по природе контракта. У `scn`/`alg` сайдкар
устроен иначе (см. следующий раздел).

## `scn` / `alg` — форму задаёт нотация, не «природа контракта»

У сценария и алгоритма наличие и формат сайдкара определяются атрибутом формы во фронтматтере:

- **`scn`** — атрибут `notation` (`../scenario-notation-registry.md`): `yaml-graph`/`pull-rules`
  машинно-читаемы в теле → сайдкар НЕ заводится; `sequence` → `<slug>.sequence.mmd`/`.puml`;
  `workflow` → `<slug>.flow.mmd`; `bpmn` → `<slug>.bpmn` (сайдкар-пара обязательна).
- **`alg`** — атрибут `form`: `procedural` — проза в теле, сайдкара нет; `decision-table` — таблица
  решений в теле (машинно-парсима), при потребности в инструментальном DMN-движке — сайдкар
  `<slug>.dmn` (`sidecar_format: dmn-1.x`).

Когда сайдкар есть, поля фронтматтера те же (`sidecar_format` + `sidecar`).

**Дефолтов может не хватить — это норма, не повод втискивать.** Если контракт по природе не
ложится в дефолт — выбери корректный стандарт под него и запиши в `sidecar_format`:
- SOAP → **WSDL** (`<slug>.wsdl`); gRPC → **protobuf** (`<slug>.proto`); GraphQL → **SDL**
  (`<slug>.graphql`); табличные данные → **JSON Schema** или **Avro**; иное — по обоснованию.
Список открыт. Не найдя точного стандарта — НЕ выдумывай нотацию: оставь только компаньон `.md` и
пометь белым пятном «машинная проекция: формат под контракт не определён, к владельцу».

## Правила формы (проверяются до отдачи)
- Машинный файл ПАРСИТСЯ без ошибок (валидный yaml/json/схема нотации) и НЕ содержит
  markdown-ограждений (```). Это проверка ФОРМЫ (см. `masterspec-gen/SKILL.md` §структурная валидность).
- Компаньон `.md` во фронтматтере несёт `sidecar_format` и `sidecar: <файл>`; в теле — провенанс и
  ссылки вверх/вниз как у обычного артефакта.
- Сайдкар несёт ту же информацию, что компаньон, без нового содержания «из головы»: провенанс
  наследуется от компаньона.

## Кто создаёт
- `recover source=code` — при наброске api/data (проход 1) создаёт и сайдкар по этой таблице
  (природа контракта видна из кода: REST-роуты → OpenAPI, поллинг/вебхуки/брокер → AsyncAPI).
- `derive layer=spec` — при генерации api/data (через `gen target_mode=sidecar`) — так же.
- `gen target_mode=sidecar` — механизм пары (компаньон + машинный); формат берёт отсюда.
- Догенерация к готовой спеке (нет сайдкаров) — отдельным migrate-проходом.
