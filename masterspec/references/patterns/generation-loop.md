# Generation-loop: агентная генерация элемента с приёмкой

## Зачем
Генерация элемента спецификации — не «шаблон + один проход», а цикл с явными приёмочными критериями и слепой проверкой. Параметр `loop=on|off` у `gen` (дефолт off до обкатки).

## Цикл (5 шагов)
1. **collect** — исходные данные элемента (в lean — по файловому контракту `context-isolation.md`).
2. **resolve** — активные guardrails элемента (`patterns/guardrails.md`).
3. **criteria** — приёмочные критерии ЭЛЕМЕНТА: чек-лист шаблона + conformance-пункты guardrails + специфика данных. Критерии рождаются ДО документа и сохраняются как артефакт прогона (рабочая зона) — иначе проверка деградирует в «нравится/не нравится».
4. **generate** — документ по шаблону.
5. **blind-verify** — ОТДЕЛЬНЫЙ агент: на входе только критерии + документ (без контекста генерации). Каждый дефект получает severity: critical | major | minor. Пункты, непроверяемые машинно, помечены `judge:llm` и попадают в выборочный человеческий аудит.

## Гейт (severity-политика, не скор)
- critical, major → доработка обязательна (возврат на шаг 4 с дефект-листом);
- minor → не более 2 итераций доработки; остаток фиксируется в route-run, не вылизывается;
- после 2 полных циклов с незакрытыми critical/major — эскалация человеку с расхождениями (не тихий третий круг).
Политика — данные фабрики/паттерна, НЕ guardrail-пакета и НЕ мнение верификатора (анти-«судья-как-рента»).

## Аудит-след
В route-run по элементу: критерии (путь), итерации, дефекты по severity, что осталось minor. Ложная зелёная галочка — главный риск: слепость верификатора + критерии-до-документа + human-gate на merge.
