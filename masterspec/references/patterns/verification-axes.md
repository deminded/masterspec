# 5 осей верификации (предмет скилла verify)

Вычитка артефактов как набор проверок-данных. Слой задаёт конфигурацию (verify scope=req|spec|change), оси общие. Метки: O1, O2, O3, O4 (с под-осями O4a и O4b), O5 — общие 5 осей; O0, O6, O7 и O_T1–O_T6 каталога отказов — дополнительные для scope=spec. Критерий spec_ready (scope=req) = нет блокеров O1, O4b, O5, включая hard gates контракта живой эксплуатации (`../operational-envelope.md`); codegen_ready (scope=spec) = O0 пройден, O6/O7, O_T и OE-воплощение полны, нет блокеров. Оси 4-5 (динамика и негатив) находят дефекты, которых не дают статические оси 1-3.

## O1 — элемент изолированно
Чек-лист по типу артефакта: все обязательные разделы заполнены — литеральная заглушка (TBD/TODO/placeholder/«…») в теле обязательного раздела считается НЕзаполненным = блокер, как пустой раздел (не только в O7); критерии приёмки проверяемы (число или факт, не «удобно/корректно»); НФТ с числовой метрикой и квалификаторами (пиковая/средняя нагрузка, окно измерения, условие); cdm/data с разделом «состояния и переходы». Для каждого `fn-` заполнено поле `Внешний инициатор/канал`. External-I/O функция имеет ровно `{OE-LOAD, OE-INPUT, OE-EVIDENCE, OE-SOURCES, OE-SECURITY, OE-RESILIENCE, OE-DELIVERY, OE-CONTROL}` с полями статус/контракт/основание/проверка; internal-only — только `OE: N/A — internal-only`, без скелета. `N/A` имеет предметную причину, `OPEN` = blocker `spec_ready`, `APPLICABLE` имеет минимум один AC. Форма по нотации (`../scenario-notation-registry.md`): `scn` с диаграммной нотацией (`sequence`/`workflow`/`bpmn`) обязан иметь секцию «Участники» (маппинг на `cmp/cap`); `scn` `pull-rules` — секцию «Инварианты потока» + `rule-id` и атрибут конкуренции у каждого правила; `alg` `decision-table` — обязательную Hit policy.

## O2 — внутрислоевая консистентность
Каждая ссылка `-> slug` ведёт на существующий артефакт; нет дублирования (один факт в одном месте, повтор → ссылка); все упомянутые сущности покрыты; нет конфликтующих правил; при конфликте НФТ и критерия приёмки — требовать развязку, не оставлять висячее противоречие. **Cross-check scn↔api (двусторонний):** каждый внешний вызов сценария ведёт на существующий `api-` (прямое); каждый `api-` привязан хотя бы к одному `scn-` (обратное — осиротевший контракт без сценария-потребителя = дыра). Сводится детерминированным set-diff (см. pre-gate покрытия).

## O3 — согласованность с эталоном и между слоями
Термины определены в глоссарии; ссылки из спецификаций на требования корректны и не фиктивны (срабатывают при прогоне); на каждое правило есть алгоритм либо явная пометка «тривиально»; каждый критерий приёмки покрыт приёмочным тестом. Для req: `APPLICABLE OE` минус OE-ссылки в AC/`tc-acc-` = пусто. Для spec: `APPLICABLE OE` минус union таблиц «Реализация контракта живой эксплуатации» связанных `scn-` = пусто; каждый `OE-LOAD` имеет `lp-`; каждый применимый `OE-DELIVERY` имеет участок в `context-` и внешний `api-`; у каждой OE назван владелец механизма, а каждый OE-владелец `scn-` проверяется связанным `tc-int-` без копирования OE-таблицы в тест.

## O4 — динамика (прогон)
**O4a — прогон содержания:** все альтернативные и исключительные ветки покрыты; полная матрица «состояние × событие» (пустая ячейка = блокер); у каждого перехода состояния задан триггер, исполнитель и момент (при чтении / по расписанию / при записи); трасса «событие → данные» с побочными эффектами на каждой ветке, включая ошибочные; выбранная реализация не стирает различие, требуемое контрактом (например, «не найдено» ≠ «истекло»). Для `scn` `pull-rules` — арбитраж каждой пары пересекающихся над одним источником правил объявлен явно (приоритет / эксклюзивность / идемпотентность); для `alg` `decision-table` — все комбинации входов покрыты строкой или явной default-строкой (непокрытая комбинация = дыра, аналог пустой ячейки матрицы).
**O4b — годность для следующего слоя:** каждый критерий приёмки выражен как проверяемый сценарий given-when-then; нет неопределённого «обрабатывает корректно»; для вычисляемых величин задан единый источник (например, единый источник текущего времени). Для каждой APPLICABLE OE `tc-acc-` задаёт воспроизводимый профиль/fixture, provenance/соответствие реальному классу, fidelity и границу наблюдения. На MVP/spec_ready synthetic или stub допустимы с моделью соответствия и явным residual risk; production-like/live-e2e и закрытие этих рисков обязательны только к codegen_ready. OE-LOAD различает общую популяцию, активное подмножество, прирост, частоту и backlog — один RPS не считается полным конвертом периодической функции.

## O5 — состязательный прогон (негатив), отдельным проверяющим
Обязательный чек-лист безопасности и приватности: подделка серверных запросов (SSRF), нормализация ввода, коллизии генерируемых идентификаторов, перебор идентификаторов и наличие ограничения частоты, политика хранения персональных данных, граничные значения. Глубина чек-листа для `api` растёт по `boundary` (`../boundary-registry.md`): `intra-factory` (relaxed) → `inter-factory` (service-to-service auth) → `perimeter` (строгий внешний профиль). Конкретные обязательные пункты и протоколы по уровню (иллюстративный пример — `boundary-registry.md`) приходят security-guardrail-пакетом (`applies_to.boundary`); kernel задаёт лишь возрастание строгости. Контрпримеры «что сломается / чего не хватает». Обе стороны контракта: вход + ответ-успех + ответ-отказ, порождённый изменением. Обязательный OE-антипрогон по каждой external-I/O функции: (1) `общая популяция × частота` и backlog; (2) реальные клиентские преобразования/локали/цитаты; (3) пользовательское доказательство успеха; (4) автоматизированные/неизвестные источники и default policy; (5) аутентификация/валидация/шифрование/подпись на всей границе доверия; (6) повтор, частичный успех, рестарт, идемпотентность и backoff вне зависимости от доставки; (7) внешний accept ≠ конечная доставка, grey/silent drop и reconciliation; (8) управляющий ввод с пробелами/регистром/обрамлением/локалью и неоднозначностью.

## Дешёвое ядро (внедрять первым)
структурная полнота → OE set-diff и отсутствие `OPEN`/немого `N/A` → однозначность терминов по глоссарию → матрица состояний → трасса «событие → данные» → сценарий given-when-then с репрезентативностью.

## O_T — каталог отказов (scope=spec)

- **O_T1 полнота точек:** для каждой точки `scn → api` есть `unavailable`, `tech-error` и каждый
  business-reject из контракта либо предметно обоснованный `infeasible`.
- **O_T2 непротиворечивость:** одинаковая комбинация точки/модуса/состояния не даёт разных исходов.
- **O_T3 честность:** coverage не ниже criticality (`low→single-fault`, `medium→dependent-pairs`,
  `high→pairwise`); infeasible/deferred имеют причину.
- **O_T4 резолвинг:** fault-id уникален, шаг/scn/api и ссылки `tc-flt/FLT-*` существуют.
- **O_T5 свежесть:** `derived_from_*` соответствует текущим scn/api; расхождение блокирует.
- **O_T6 связь с tc:** каждая feasible-строка связана с `tc-int`; `TC-TODO`/deferred блокируют
  `codegen_ready`. Отказный tc имеет отдельный лог-oracle и ссылку на `OE-EVIDENCE`.

## Детерминированный pre-gate покрытия (до LLM-осей)
Перед запуском осей — дешёвая ДЕТЕРМИНИРОВАННАЯ сверка множеств (скриптом или ручным set-diff). OE-сверка обязательна при наличии `fn-` и выполняется `python3 <kernel-skill-dir>/scripts/check-operational-envelope.py <factory-root> --scope req|spec` (после реализации для codemap-аудита — `--scope code`; скрипт расположен `../scripts/` относительно этого reference; при недоступном Python — тем же ручным set-diff); остальные сверки опциональны. `expected` берётся из дерева/индекса как read-only вход — НЕ заводится новый реестр (иначе риск O0):
- **прямое** покрытие: каждое требование/сущность-источник имеет порождённый артефакт (нет недокрытого);
- **обратное** покрытие (анти-«призрак»): каждый документированный артефакт/элемент имеет основание в источнике (нет задокументированного без основания — то, что O2 «все упомянутые покрыты» не ловит в обратную сторону).
- **OE req:** expected = восемь ID на каждый external-I/O `fn-` либо одна internal-only строка;
  applicable = OE со статусом `APPLICABLE`;
  actual = ссылки из AC/`tc-acc-`;
- **OE spec:** expected = applicable OE функций; actual = union OE-строк только `scn-`; `tc-int-`
  покрывает OE транзитивно ссылкой на `scn-`; отдельные пары `OE-LOAD ↔ lp-` и применимый
  `OE-DELIVERY ↔ context-/external api-`.
Расхождение в любую сторону → FAIL ДО LLM-осей: дёшево отсекает то, что иначе ищут дорогие оси или пропускает самопроверка. Провайдер- и язык-нейтрально: где скрипта под стек нет — та же сверка вручную как чек-лист. Для не-OE сверок отсутствие инструмента не блокирует verify, лишь снимает дешёвый фильтр; OE-проверка не снимается, а деградирует в ручной set-diff.

## Для scope=spec — дополнительные оси O0, O6, O7
**O0 — single-source (hard-fail).** Ровно одна нормативная редакция каждого контракта (api) и схемы (data). Два значения одного поля в разных разделах до поэлементной вычитки = немедленный hard-fail. Машинная пара компаньон `.md` + sidecar `.yaml`/`.json` (`change-format.md §3.1`) — это ОДНА редакция (sidecar — машинная форма того же контракта, а не вторая нормативная), и нарушением O0 не считается; нарушение — два расходящихся определения одного поля.

**O6 — контракт интерфейса** (вопрос-критерий: можно ли написать клиента ТОЛЬКО по спецификации?). Чек-лист по каждой операции/событию api:
- задан идентификатор (endpoint / topic / method);
- ВХОД: все поля, их обязательность, значимые заголовки;
- ОТВЕТ-УСПЕХ: статус успеха + форма ответа;
- ОТВЕТ-ОТКАЗ: классы ошибок со статусами, включая порождённые ограничениями; различимые по смыслу исходы дают различимые ответы («не найдено» ≠ «истекло»);
- нефункциональные обязательства контракта (идемпотентность, SLA, безопасность ретраев, версионирование) заданы;
- **маппинги полей:** каждое поле, требующее наполнения (не вводимое клиентом напрямую), имеет указанный источник («поле ← `api-`/`data-`/контекст/константа»); поле без источника при необходимости наполнения = дыра контракта.

**O7 — физическая модель данных** (вопрос-критерий: готова ли к генерации хранилища?). Чек-лист по каждой data-сущности:
- у каждого атрибута задан тип (логический на спеке);
- ключи и уникальность определены;
- связи между сущностями с кратностью;
- ограничения (обязательность, диапазоны, TTL по смыслу) заданы;
- по каждой форме хранения выбор СДЕЛАН (не «TBD»); значимый выбор записан в `dr-`/`adr-`.

## Метрика (отчуждаемо от содержания)

Все метрики записываются по `templates/tpl-verify-report.md`: `last_verified`, возраст самой старой
сверки, wall-time/вызовы/токены/стоимость и доля машинных axis-run обязательны. Неизвестный счётчик
не оставляется пустым: `N/A — причина`. OE-покрытие выводится обычным и взвешенным по criticality.
holes по осям; codegen_ready (для scope=spec) или spec_ready (для scope=req); остаток по типу (вне-области / противоречие / недоведённое). Выносится без содержания — для приёмки и отчёта.


## Ось D-DUP: дублирование в change (machine)
Одинаковые текстовые фразы длиной >8 слов, встречающиеся в ≥3 артефактах одного change (diff-блоки ПОСЛЕ + new/), — дефект D-DUP: суть размазана вместо канона-в-одном-месте (`change-format.md §4 Лаконичность`). Проверка машинная (n-грамм сверка по нормализованному тексту); исключения — цитаты шаблонных заголовков и slug-ссылки.
