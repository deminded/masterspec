# Изоляция контекста оркестратора (lean-режим)

## Зачем
На модели с ограниченным контекстом (например 128k) последовательная работа оркестратора (`derive`/`evolve`) на фабрике среднего размера переполняет контекст: оркестратор по ходу читает мета-модель, шаблоны, research, вышестоящие артефакты, отчёты вычитки. Этот паттерн делает рост контекста оркестратора независимым от размера фабрики — через файловый контракт между субагентами.

## Принцип: оркестратор — диспетчер по файлам
В `context=lean` оркестратор держит в контексте ТОЛЬКО: план (`plan.md`), журнал (`route-run-<ts>.md`), пути и короткие сводки (≤ ~10 строк) от субагентов. Содержимое артефактов, мета-модель, шаблоны, research, отчёты вычитки — НЕ держит. Всё тяжёлое — изолированные субагенты, пишущие в файл и возвращающие сводку.

## Запрещено оркестратору (lean) — single-source
Собрано здесь как единый источник; ниже по тексту — операционные следствия. В `context=lean` оркестратор (любое из этого — нарушение режима, не оптимизация):
- НЕ читает СОДЕРЖИМОЕ: код, документы, артефакты фабрики, мета-модель, шаблоны, research, отчёты вычитки. Исключение — компактные УПРАВЛЯЮЩИЕ данные для маршрутизации, явно разрешённые ниже (шапка `change.md` + §2-таблицы в lean-apply шаг 1): это метаданные плана, не содержимое артефактов;
- НЕ пишет артефакты/diff/sidecar/целевые файлы сам — это делает субагент-исполнитель (`gen` при генерации, `apply-change`-субагент при применении change на строку §2);
- НЕ обходит граф зависимостей сам — scope-fence и набор узлов каскада берёт из файлов `planner`/`discovery`, не из собственного чтения;
- держит в контексте ТОЛЬКО план, журнал, пути и короткие сводки (≤ ~10 строк).

## Два уровня изоляции
- **explore** изолирует чтение КОДА → `.research/` (нужен только при наличии кода).
- **planner** изолирует чтение ДОКУМЕНТОВ (бизнес-запрос + research + мета-модель + вышестоящие артефакты) → план и фокус-наборы.
Оркестратор не читает ни код, ни документы.

## Режимы
`context=lean` (дефолт) — строгая делегация по этому контракту (ограниченный контекст). `context=full` — оркестратор может читать сам, меньше субагент-оверхеда (большой контекст).

## Рабочая зона и namespace
Всё ЭФЕМЕРНОЕ рабочее — в `masterspec/.work/<run-id>/` (run-id = `<layer|change-name>-<ts>`), чтобы параллельные/долгоживущие прогоны не делили `plan`/`.focus` и не сносили чужую зону. Внутри: `plan.md`, `.focus/<slug>.md`, `verify/` (частичные отчёты lean-verify: `<elem>.md`, `_negative.md`, `_spec-o0o6o7.md`, `_change-nodes.md`, `<node>.md`), `apply/` (lean-apply: `_dryrun.md`, `<slug>.md`, `_smoke.md`), `recover/` (lean-recover: `plan.md`, `_dangling.md`, `_gaps.md`).
РЕЗУЛЬТАТЫ-метрики (отчуждаемые ОТ содержания) хранятся ОТДЕЛЬНО и переживают чистку: `route-run-<ts>.md`, `verify-report.md`, `apply-report.md` — в корне фабрики (генерация) или `changes/<name>/` (изменение). В `.work/` их НЕ держим (иначе чистка их снесёт). Частичные `*/*.md` (verify/apply/recover) — эфемерны (в `.work/`), отчёты-результаты — вынесены.
По завершении прогона `masterspec/.work/<run-id>/` ОБЯЗАТЕЛЬНО удаляется: фокус-наборы содержат ВПИСАННЫЕ срезы артефактов (дубли содержания) — оставлять нельзя (рассинхрон, мусор). Итог: `route-run` сохранить, `masterspec/.work/<run-id>/` снести.

## Формат плана (`plan.md`)
План — НЕ список уникальных артефактов, а УПОРЯДОЧЕННАЯ ПОСЛЕДОВАТЕЛЬНОСТЬ ШАГОВ и LOOP-блоков (один slug может встречаться в нескольких шагах). Шаг:
```
- step: <N>
  slug: <artefact-slug>
  type: <fn|cmp|api|data|...>
  node: execution | decision        # узел-исполнение или узел-решение
  focus: masterspec/.work/<run-id>/.focus/<slug>.md
  target_path: <путь относительно masterspec/ | (default)>
  target_mode: file | diff | sidecar | (default)
  upstream: [<slug>, ...]           # на что опирается (для трассировки)
```
LOOP-блок (для канонического цикла `cdm ↔ fn`):
```
- loop: cdm-fn
  steps: [<cdm-draft>, <fn-*>, <cdm-states>]
  exit: нет новых событий/состояний в cdm ИЛИ достигнут предел проходов (по умолчанию 2)
```
planner раскладывает шаги в порядке состава слоя (`derive §2`); цикл cdm↔fn выражается loop-блоком, а не молчаливым повтором.

## Формат фокус-набора (`.focus/<slug>.md`)
Самодостаточен для `gen` (gen читает ТОЛЬКО его, в фабрику не лазит): бизнес-описание элемента + ВПИСАННЫЕ срезы вышестоящих (см. границу ниже) + `target_path`/`target_mode` (или пометка «default») + указание `tpl-<type>`.

## Адресация вывода (target + дефолты)
`target_path`/`target_mode` есть в контракте, но planner проставляет их ЯВНО только для нестандартного. Если не указано — выводятся по правилам:
- `target_path` по умолчанию — из `artifact-routing.md` по `type` (api — по `scope` в internal/external; fn — по `block`);
- `target_mode` по умолчанию: машинный `api`/`data` (`.yaml`/`.json`) → `sidecar` (+ компаньон `.md`); НОВЫЙ артефакт → `file`; изменение СУЩЕСТВУЮЩЕГО `.md` → `diff` (diff-блок в change.md §4).
`gen` принимает `from=<focus>` + `target_path`/`target_mode` (явные или default) и пишет ровно туда и так — не «по догадке».

## Write-scope субагентов (deny-by-default)
Каждый субагент пишет ТОЛЬКО в назначенный ему выход, никуда больше:
- `gen` — в свой `target_path` (file/diff/sidecar), как в «Адресация вывода»;
- `verify`/`recover`-субагенты — в свой нормированный `<file>.md` внутри `masterspec/.work/<run-id>/`;
- `apply-change`-субагент — в свой `apply/<slug>.md` (отчёт о применении, в `.work/`) И в целевой файл строки §2 (само применение change: diff-блок §4 / копия `new/` / удаление) — это его назначенный выход;
- `explore` — в `.research/`; `planner` — в назначенный план/фокус-output (`plan.md`/`.focus/*` для derive/evolve, `recover/plan.md` для recover).
Запись вне назначенного пути — нарушение (рассинхрон, гонки на чужой зоне): оркестратор не принимает такую сводку и перезапускает субагента с явным target. Это обобщение правила `gen` «пишет ровно туда, не по догадке» на все lean-субагенты — deny-by-default.

## Граница фокус-набора (что planner кладёт)
planner ВПИСЫВАЕТ вырезанный релевантный СРЕЗ (не путь, не весь файл), по таблице:
| Тип элемента | В фокус кладётся (срез вышестоящих) |
|---|---|
| `fn` | сущности `cdm`, на которые fn ссылается (только их разделы); относящиеся `rules`; термины глоссария из описания fn |
| `cdm` (states) | сам черновой cdm + события из `fn`, порождающие переходы |
| `cmp` | `fn`, которые компонент реализует (их AC) |
| `scn` | задействованные `cmp` (их `cap-*`) |
| `api` | `cmp`-владелец + связанные `data` |
| `data` | `cdm`-сущности, которые схема материализует |
| `as` | бизнес-запрос: назначение, границы, потребители (вышестоящих нет — корневой артефакт) |
| `rules` | домен правила из запроса + имена сущностей `cdm`, которых правило касается |
| `nfr` | `fn`, к которым метрика относится (их id), + профиль нагрузки из запроса |
| `dict` | сущность `cdm`, значения которой справочник перечисляет |
| `tc-acc` | AC конкретной `fn` (только её критерии приёмки) |
| `alg` | `scn`/`cmp`, чью пошаговую логику алгоритм детализирует |
| `cd` / `lp` | `cmp`/`scn`, которые диаграмма / профиль нагрузки покрывает |
| `ui-view` | `fn`/`scn`, которые экран обслуживает, + `data`, что он отображает |
| `nav` | `ui-view`, между которыми описывается навигация |
| `tc-int` | `scn`, который интеграционный тест проверяет (его шаги) |
Правило резки: брать РАЗДЕЛЫ, на которые элемент реально сослался (по `-> slug` и по бизнес-описанию), плюс обязательные ограничения этих разделов, даже если не названы буквально. НЕ брать: целые файлы «на всякий случай», несвязанные разделы. Эвристика «трогает + один уровень связей» — это потолок, не повод тащить всё.

## Planner: два режима вывода
- **derive** → `plan.md` (шаги/loop как выше) + `.focus/*`. target_mode по умолчанию `file` (генерация в дерево).
- **evolve** → `plan.md`, где у каждого шага явный режим: существующий артефакт → `target_mode: diff` (gen выдаёт diff-блок для `change.md §4`), новый → `target_mode: file` (gen пишет `changes/<name>/new/<slug>.md`), машинный → `sidecar`. planner также готовит якоря `scope-fence` (список соседей с пометкой «затронут/не затронут») в `plan.md`, чтобы оркестратор и приёмщик проверяли полноту каскада, не читая граф сами.

## Дозапрос (единый контракт для lean)
gen не хватило фокус-набора → один маршрут:
1. если нужен факт из КОДА → запрос в `explore` (добирает в `.research/`);
2. если нужен срез ДОКУМЕНТА/артефакта → запрос в `planner` (дописывает `.focus/<slug>.md`);
3. если нет ни в коде, ни в документах → открытый вопрос человеку.
Оркестратор только маршрутизирует запрос и перезапускает `gen`; сам не читает. Лимит — 2 раунда, потом п.3.

Симметрично на ВЫХОДЕ (`SPLIT_REQUIRED`): субагент, не способный выдать ПОЛНЫЙ результат (не хватило фокуса, предел длины ответа), возвращает явный `SPLIT_REQUIRED` с указанием, чего недостаёт, — а НЕ молча усечённый результат. Тихое усечение — худший дефект сабагентного лупа: выглядит как успех, проходит дальше непойманным. Оркестратор на `SPLIT_REQUIRED` либо дозапрашивает (п.1–2), либо дробит шаг на под-шаги — под ТЕМ ЖЕ лимитом 2 раунда, что и входной дозапрос; исчерпан лимит → эскалация (п.3). Частичное за полное не принимает. Лимит общий, бесконечного retry нет.

## Делегация шагов в lean
1. `explore` → `.research/` + сводка (если есть код).
2. `planner` → `masterspec/.work/<run-id>/plan.md` + `.focus/*` + сводка (путь + N шагов + открытые вопросы).
3. по шагам плана: `gen` (`from`, `target_path`, `target_mode`) → пишет артефакт/diff/sidecar → сводка (путь + статус + дозапросы).
4. `verify` (субагент) → `verify-report.md` + метрики/блокеры.
5. приёмщик (субагент) сводит редакции → сводка.
Между шагами у оркестратора только пути и сводки.

## Lean в других скиллах (verify, recover, apply-change)
Тот же принцип «диспетчер по файлам» применим к любому скиллу, который иначе сам читает много в свой контекст:
- **verify** (standalone-аудит большого слоя). Единица декомпозиции зависит от scope:
  - `scope=req|spec`: на каждый ЭЛЕМЕНТ — субагент дешёвого ядра осей (O1–O4), пишет `masterspec/.work/<run-id>/verify/<elem>.md` (оси этого элемента — РАЗДЕЛАМИ внутри файла, не отдельными файлами — коллизии путей нет). Негатив-ось O5 — отдельный субагент-антагонист на слой/критичные элементы → `verify/_negative.md`. Для `scope=spec` оси O0/O6/O7 — отдельный субагент по контрактам/данным → `verify/_spec-o0o6o7.md`. Спец-проверки лежат в именованных файлах `_*.md` (не `<elem>`), поэтому с поэлементными не сталкиваются.
  - `scope=change` (единица — не ось, а УЗЕЛ каскада), в три шага:
    1. **discovery** — отдельный субагент читает `change.md` (§2 якоря, затронутые slug'и) и считает набор узлов каскада ЗАНОВО (затронутые + соседи по обратным рёбрам, корни проверки), пишет `verify/_change-nodes.md` (список узлов + корни). Так независимость от генератора сохранена: оркестратор берёт узлы из файла, граф сам не обходит.
    2. **per-node** — субагент на каждый узел из списка → `verify/<node>.md` НОРМИРОВАННОГО формата: `node`, `role: затронут|сосед`, `cascade-AC: ok|missing` (порождён ли приёмочный критерий для изменения), `contract-both-sides: ok|n/a|missing`, `scope-fence: <вердикт «затронут/не затронут, потому что…»>`, `silent-verdict: yes|no` (узел на обратном ребре без правки И без вердикта), `silent-raise: yes|no` (подъём без материализации каскада вверх), `blockers: [...]`. Один файл на узел (slug уникален → дублей нет; два файла на узел = ошибка discovery).
    3. **aggregate** — оркестратор сводит `verify/<node>.md` в `verify-report.md`: немые вердикты/подъёмы = узлы с `silent-*: yes`; полнота каскада = нет `cascade-AC: missing`; scope-fence-покрытие = у каждого соседа непустой вердикт.
  Сводка: verify-оркестратор читает ТОЛЬКО частичные `verify/*.md` и агрегирует их в `verify-report.md` (holes по осям, by_severity, spec_ready/codegen_ready, остаток) — сам слой не читает. namespace `masterspec/.work/<run-id>/` и чистка — общие; verify-report выносится отдельно (он метрики, не содержание).
- **recover** (восстановление большой фабрики), по проходам:
  1. **план** — субагент-планировщик пишет `recover/plan.md` НОРМИРОВАННОГО формата: `pass1: [{slug, type, source: code|docs}]` (скелет сверху), `pass2: [{slug, type: cmap|trace|dmap, target_artefact}]` (код-слой).
  2. **проход 1 (скелет сверху)** — `explore` изолирует код → `.research/`; субагенты-`gen` по research/docs пишут СКЕЛЕТ спецификаций и требований (slug'и, `status: draft`, поле `provenance`) → артефакты в дерево. Так появляются целевые slug для ссылок снизу.
  3. **проход 2 — код-слой + висячие ссылки (source=code)** — субагенты собирают codemap (`cmap`/`trace`/`dmap`, ссылки ВВЕРХ на slug прохода 1); каждую НЕразрешённую ссылку дописывают в `recover/_dangling.md` НОРМИРОВАННОГО формата: `from` (codemap-файл, где ссылка), `ref` (на какой slug ведёт), `evidence` (`file:line` из кода), `resolution: pending | resolved | created`, `handled_by`.
  4. **перелинковка** — субагент проходит `recover/_dangling.md`: каждую строку с `resolution: pending` либо разрешает в существующий slug (`resolved`), либо ДОзаводит недостающий spec через `gen` (`created`, под общий `provenance`). На выходе — ни одной `pending`. Решение принимается по файлу, не «магией».
  5. **gaps** — субагент собирает «Белые пятна» (что не восстановилось из источника) в `recover/_gaps.md`, оттуда они переносятся в `00-masterspec-index.md §7`. recover-оркестратор держит `recover/plan.md` и сводки (`_dangling`/`_gaps`-счётчики), содержимое артефактов не читает; индекс перестраивает субагент.
  `recover/*` эфемерны (`.work/`); §7 «Белые пятна» в `00-masterspec-index.md` — в фабрике (результат, переживает чистку).
- **apply-change** (крупный change), пошагово:
  1. **dry-run.** Оркестратор читает ТОЛЬКО шапку `change.md` + §2-таблицы (MODIFIED/ADDED/REMOVED: slug + путь + тип) — компактно; формирует `apply/_dryrun.md`, человек подтверждает.
  2. **per-block apply+verify.** На каждую строку §2 — субагент: применяет (diff-блок §4 / копирование `new/` / удаление) И делает verification СВОЕЙ строки (`merge-workflow.md §9.2`), пишет `apply/<slug>.md` НОРМИРОВАННОГО формата: `op: modify|add|remove`, `target`, `result: applied | conflict | skipped_by_user`, `verification: confirmed | unconfirmed`, `note`. Решение пропустить на конфликте принимает ЧЕЛОВЕК (субагент возвращает `conflict` → оркестратор спрашивает AskUserQuestion → при «пропустить» строка получает `result: skipped_by_user`). Так §9 и `skipped_by_user` живут в файловом контракте, а не в памяти.
  3. **reindex.** Субагент полностью перегенерирует `00-masterspec-index.md` по `index-canonical.md` — ДО smoke-check (smoke проверяет «каждый ADDED в индексе», поэтому индекс к этому моменту обязан быть обновлён).
  4. **smoke-check (gate).** Отдельный субагент проверяет ГЛОБАЛЬНЫЕ инварианты фабрики (`merge-workflow.md §9.1`: число файлов, каждый ADDED в УЖЕ перегенерированном индексе, каждый REMOVED отсутствует, нет обратных ссылок) → `apply/_smoke.md`. Провал smoke → `git checkout` откат, к §9.3 (aggregate) НЕ переходим (как в full). Это структурный gate ПЕРЕД per-block агрегатом.
  5. **aggregate (§9.3).** Оркестратор сводит `apply/<slug>.md` + `apply/_smoke.md` в вердикт `confirmed / skipped_by_user / unconfirmed` — НЕ читая целевые файлы; статус change.md по вердикту (`Реализовано` / `leave` / `rollback` — `merge-workflow.md §9.3`).
  `apply/*.md` эфемерны (`.work/`); итоговый `apply-report.md` (§9.3-вердикт) — ВНЕ `.work/` (как verify-report), переживает чистку.
- **gen** и **explore** параметр `context` НЕ несут: gen атомарен (один артефакт в фокус-наборе, пухнуть нечем), explore lean по природе (изолированные разведчики → `.research/`, лид не читает сырьё).
`masterspec/.work/<run-id>/` и обязательная чистка — те же, что для derive/evolve.
