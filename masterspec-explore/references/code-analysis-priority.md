# Приоритет инструментов анализа кода

Когда скилл требует понять существующий код (пакеты, символы, зависимости, вызовы), всегда выбирай **наиболее семантический** из доступных инструментов. Грубые текстовые средства — последняя линия.

## Порядок приоритета

0. **Graphify-карта через adapter (discovery, опционально)** — если рядом с кодом построен граф (`<root>/graphify-out/graph.json` + `graph.meta.json`), первый шаг discovery — `python3 <kernel-skill-dir>/scripts/graph-adapter.py neighbors --of <path>#<symbol> --graph <путь к graph.json>` (соседи/вызовы готовыми локаторами) и `slice --root <r>` (срез по репе). Дёшево и без LLM. СТРОГО опционально: exit 2 (fallback: нет графа / stale / dirty / битый) → просто иди дальше по списку, причина уже напечатана. Граф — discovery/cache, НЕ доказательство: каждый локатор, попадающий в артефакт, подтверждается наведением/чтением (Serena/LSP/Read), не ребром графа.
   **Multi-repo (несколько реп кода в roots):** у КАЖДОЙ репы своя независимая пара граф+манифест в её `graphify-out/` — объединённого графа НЕ существует и не строится (tree-sitter не видит межрепных связей; мерж дал бы иллюзию полноты). Адаптеру передавай все пары повторяемым флагом: `--graph <repoA>/graphify-out/graph.json --graph <repoB>/graphify-out/graph.json` — он держит их изолированными и не создаёт ложных cross-repo рёбер. Межрепные связи ищи НЕ в графе, а на слое спеки (`api-`/`scn-`/`context-`) и fallback-механикой (Serena/grep по-репно). Построение (если графа нет и пользователь просит): `<kernel-skill-dir>/scripts/graph-build.sh <root>...` — ТОЛЬКО по репам кода, НИКОГДА по репозиторию аналитики (корень с `masterspec/00-masterspec-index.md` — отказ: graphify извлёк бы «код» из fenced-блоков md и отравил discovery мусором).
1. **Serena (MCP)** — `find_symbol`, `get_symbols_overview`, `find_referencing_symbols`. Семантический анализ, точная навигация по символам, поиск всех мест использования. Наведение и верификация — всегда здесь (или LSP/чтением), даже когда discovery дал граф.
2. **LSP-инструменты** — go-to-definition, find-references, workspace symbols. Работают, если в среде поднят языковой сервер (gopls, pyright, jdtls и т. д.).
3. **Code index / embeddings** — семантический поиск по кодовой базе (например, через MCP-сервер индексации).
4. **Другие MCP-серверы** — любые специализированные инструменты анализа, навигации, индексации, которые доступны в сессии.
5. **Grep / Glob** — текстовый поиск. Используется только когда ни один из семантических инструментов не подходит или не доступен.

## Правила применения

- **Не ограничивайся чтением отдельных файлов.** Строй полную картину: зависимости, вызывающий код, связанные сущности.
- **Не деградируй преждевременно.** Если Serena доступна — используй её; не падай сразу в grep только потому, что «так быстрее».
- **Комбинируй.** Семантический инструмент + точечный grep по литералам (строки, регулярки, магические константы) — рабочая связка.
- **Фиксируй контекст.** Прежде чем принимать решение по change.md / design.md, убедись, что изучены все точки интеграции, вызывающий код и конфигурация.
