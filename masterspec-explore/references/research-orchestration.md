# Оркестрация исследования (masterspec)

Инструкция для лид-агента: как провести structured research кодовой базы под фабрику (картирование новой или зоны изменения), чтобы собрать достаточно данных для заполнения артефактов трёх слоёв (требования / спецификации / кодовая база).

Правила написаны так, чтобы их могла применять слабая модель (окно 200к): числа заданы явно, промпты — в [research-roles.md](research-roles.md) как готовые шаблоны, рекурсия субагентов запрещена.

---

## 1. Вход

Лид-агент передаёт:
- **source**: `code` (default) — исследуем кодовую базу; `docs` — исследуем сырьё (`00-source-data/` и внешние документы). Оси декомпозиции у них РАЗНЫЕ (§ 3), не путать.
- **target**: `factory-spec` (картируем всю фабрику под её описание) или `factory-change` (картируем зону изменения)
- **factory**: slug фабрики (kebab-case) — определяет корень `masterspec/...`
- **roots**: корневые пути исходного кода (для монорепо — путь модуля; для одного корня — `.`)
- **anchor** (только для `factory-change`): пути/символы/slug'и из change, попадающие в скоуп
- **thoroughness** (опционально): `quick` (default) / `medium`

Без `target` и `factory` — не запускаемся. Для `source=code` обязателен `roots`; для `source=docs` — `docs` (пути к материалам; дефолт `masterspec/00-source-data/`).

## 2. Классификация размера

Быстрые команды:

```bash
find . -type f \( -name "*.java" -o -name "*.kt" -o -name "*.py" -o -name "*.go" -o -name "*.ts" -o -name "*.js" -o -name "*.rs" -o -name "*.rb" -o -name "*.cs" \) \
  | grep -v -E "(node_modules|build|dist|target|\.git)" | wc -l
```

| Уровень | Критерий | Кол-во субагентов |
|---------|----------|-------------------|
| **Малый** | <50 файлов кода ИЛИ 1 модуль/пакет | **2** параллельных субагента |
| **Средний** | 50–500 файлов ИЛИ 2–10 модулей | **2–3** параллельных субагента |
| **Крупный** | >500 файлов ИЛИ multi-module / multi-repo | **3–5** параллельных субагентов |

> **Субагенты обязательны всегда, когда tool запуска субагента доступен.** Лид не должен читать исходный код в своё окно — ломается дедупликация и рассинхронизируется YAML-контракт. Вариант «0 агентов» допустим только как fallback (§ 8).

Границы нестрогие: 200 файлов без внутренних границ = Средний (2 агента); 80 файлов с 5 bounded-context = Средний.

### 2.1. Размер массива документов (`source=docs`)

Файлы кода считать бессмысленно — у документа вес не в файлах, а в СЛОВАХ: одна выгрузка Confluence может весить как весь остальной массив.

```bash
find <docs_roots> -type f \( -name "*.md" -o -name "*.html" -o -name "*.txt" -o -name "*.yaml" -o -name "*.json" -o -name "*.csv" \) | wc -l
find <docs_roots> -type f \( -name "*.md" -o -name "*.html" -o -name "*.txt" \) -exec wc -w {} + | tail -1
```

| Уровень | Файлов | Слов (или эквивалент) |
|---------|--------|------|
| **Малый** | <10 | <20k |
| **Средний** | 10–50 | 20k–150k |
| **Крупный** | >50 | >150k |

**Класс = МАКСИМУМ по обеим метрикам, не «или».** 100 мелких файлов на 10k слов — крупный по файлам, малый по словам → берём крупный. Иначе классификация двусмысленна и агент выберет удобную.

Фаза B: Малый — 2 субагента (роли парами), Средний — 3, Крупный — 4 (роль = агент) + дробление батчей.

**Не текстовые форматы считаются своей мерой** (слова в них `wc -w` не берёт): таблицы (xlsx/csv) — строки × колонки; pdf/docx — страницы; yaml/json/xml — узлы верхнего уровня. Пересчёт в эквивалент слов для класса: 1 страница ≈ 500 слов, 1 строка таблицы ≈ 20 слов, 1 узел ≈ 10 слов. Грубо — и достаточно: цель не бухгалтерия, а не проглядеть массив, который не влезет.

**Потолок батча — ~40k слов на одного субагента.** Батч тяжелее — режь по материалам (не по слоям: слои режутся ролями). Один материал тяжелее потолка (`oversized: true` в описи) — режь его по разделам и раздай разделы, а не файл целиком.

> Сайзинг делает не лид на глаз, а фаза A: `docs-inventory` возвращает `totals` и готовое предложение `batches[]` с `est_words`. Лид проверяет предложение против потолка и корректирует. Это и есть «разбирать с учётом размера массива» — молчаливая деградация большого документа в переполненном окне недопустима: лучше честный `gap`.

## 3. Оси декомпозиции

### 3.1. `target=factory-spec` — ось «по слоям мета-модели»

Каталог ролей для описания новой фабрики (или обновления картирования существующей). Полный список — [research-roles.md § Роли для factory-spec](research-roles.md).

| Роль | Слой | Целевые артефакты |
|------|------|-------------------|
| `requirements-scope` | Требования | `fn-*`, `nfr-*`, `rules-*`, `cdm-*`, `dict-*` |
| `components-scope` | Спецификации | `cmp-*` (с разделами `cap-*`) |
| `scenarios-scope` | Спецификации | `scn-*`, `alg-*` |
| `data-scope` | Спецификации | `data-*`, `api-*` (internal/external) |
| `codemap-scope` | Кодовая база | `repo-map`, `cmap-*`, `trace-*`, `dmap-*` |

**Группировка по размеру проекта**:

| Размер | Группы (каждая = один субагент) |
|--------|---------------------------------|
| Малый (2 агента) | [requirements-scope + components-scope], [scenarios-scope + data-scope + codemap-scope] |
| Средний (3 агента) | [requirements-scope + components-scope], [scenarios-scope + data-scope], [codemap-scope] |
| Крупный (5 агентов) | [requirements-scope], [components-scope], [scenarios-scope], [data-scope], [codemap-scope] |

### 3.2. `target=factory-change` — ось «по зоне изменения»

Каталог ролей для картирования зоны change.md. Работаем в пределах `anchor_paths` и смежных файлов.

| Роль | Что картируем |
|------|---------------|
| `requirements-scope` | Затронутые `fn-*`/`nfr-*`/`rules-*` — текущие AC, инварианты, правила в коде |
| `components-scope` | Затронутые `cmp-*`/`cap-*` — ответственности, границы модулей в zone |
| `scenarios-scope` | Затронутые `scn-*`/`alg-*` — текущие цепочки вызовов |
| `data-scope` | Затронутые `data-*`/`api-*` — схемы/API в zone |
| `codemap-scope` | Только если change затрагивает кодовый слой фабрики — обновление существующих `cmap-*`/`trace-*`/`dmap-*` |

**Группировка по размеру**:

| Размер | Группы |
|--------|--------|
| Малый (2 агента) | [requirements-scope + components-scope], [scenarios-scope + data-scope] |
| Средний (3 агента) | [requirements-scope], [components-scope + scenarios-scope], [data-scope] |
| Крупный (4 агента) | [requirements-scope], [components-scope], [scenarios-scope], [data-scope (+ codemap-scope при необходимости)] |

Для простых change (правка одного AC, редактирование строки таблицы) исследование кода вообще можно пропустить — пользовательский чек `Нужно ли картировать код?` при создании change (`evolve`, шаг 0).

### 3.3. `source=docs` — ось «по материалу» (две фазы)

Код режется по путям, документ — нет: одна страница Confluence несёт и требования, и контракт, и тест-кейсы вперемешку. Поэтому сырьё режется по МАТЕРИАЛУ, а слои извлекаются из каждого материала.

**Фаза A — опись и разметка.** Один субагент `docs-inventory` (промпт — [research-roles.md](research-roles.md)) обходит `docs_roots`, классифицирует каждый файл (тип материала, объём, слои-мишени, machine-ready, дубли) и возвращает `docs-inventory.yaml` + предложение `batches[]`. Читает СТРУКТУРУ, не тело — иначе не удержит массив.

Фаза A не параллелится: опись должна быть цельной, иначе дубли и границы батчей не сойдутся.

**Фаза B — извлечение по слоям.** Роли `requirements-scope`, `components-scope`, `scenarios-scope`, `data-scope` (docs-вариант — см. research-roles.md) получают границами не пути кода, а списки материалов из `batches[]`. `codemap-scope` не применяется — кода нет.

| Размер массива | Группы (каждая = один субагент) |
|--------|---------------------------------|
| Малый (2 агента) | [requirements-scope + components-scope], [scenarios-scope + data-scope] |
| Средний (3 агента) | [requirements-scope], [components-scope + scenarios-scope], [data-scope] |
| Крупный (4 агента) | по одной роли на агента; батч > 40k слов — дробится по материалам, роль запускается несколько раз |

**`testcases-scope` — опциональная пятая роль фазы B.** Запускается ТОЛЬКО если опись нашла `material_type: test-cases`. Состав фазы B определяет опись, а не лид на глаз. Готовые ТК не забираются as-is: каждый обязан стыковаться с элементом спеки (`traces_to` с `confidence`), нестыкованный уходит в открытые вопросы — висячий `tc-` не проверяем на полноту и молча разъедется со спекой при первом изменении.

**Machine-ready материал** (готовый OpenAPI/AsyncAPI/JSON Schema/BPMN) через роли фазы B НЕ пересказывается: он попадает в агрегат записью `machine_artifacts[]` (путь + формат + перечень операций), а тело остаётся в файле — `recover` подключит его сайдкаром. Пересказывать прозой то, что уже машинно выражено, — терять точность.

## 4. Запуск

1. **Выбери роли** по `target` и размеру (таблицы выше).
2. **Для каждой группы** — один субагент. Промпт берётся из [research-roles.md](research-roles.md). Если группа — несколько ролей, склей их YAML-контракты в один промпт с пометкой `## Роли`.
3. **Запуск в одном батче** tool-вызовов — несколько вызовов tool'а запуска субагента в одном ответном сообщении. Имя tool-а и `subagent_type` — по алгоритму [invocation-contract.md § 3.1](invocation-contract.md).
4. **subagent_type**:
   - Если доступен read-only субагент (`Explore` / `explore`) — используй его.
   - Иначе — `general-purpose` / `general` с явным запретом записи в промпте (см. invocation-contract.md § 4).
5. **Thoroughness**: `quick` по умолчанию. `medium` — только если пользователь явно сказал «подробнее» и проект крупный.

## 5. Инвариант scope

- Каждому субагенту — **непересекающийся набор путей**. Если зоны перекрываются — укрупни задачи, уменьши число агентов.
- **Исключение для `source=docs`: пересекается ЧТЕНИЕ, не пересекается ВЛАДЕНИЕ.** Один документ физически содержит материал разных слоёв (страница Confluence: требования + контракт + ТК), поэтому запрет на общий read-set здесь неисполним — его нельзя выполнить, не потеряв данные. Инвариант переформулируется: несколько ролей МОГУТ читать один материал, но каждая коллекция агрегата (`functions[]`, `apis[]`, `test_cases[]`, …) имеет РОВНО ОДНУ роль-владельца, и роль пишет только в свои коллекции. Дедупликация тогда идёт внутри коллекции, а не между агентами.
- Зоны задаются явно в промпте (`Границы: читай только <dir/**>, <dir2/**>`). Общие пути (`src/**`) допустимы только когда модулей мало и разделение бессмысленно.
- При `target=factory-change` — зоны ограничены `anchor_paths` и их прямыми импортёрами/вызывателями.

## 5.1. Шардирование (роль запускается несколько раз)

Когда батч роли не влезает в потолок, роль запускается по шардам. Тогда:

- выход шарда — `<role>-<batch-id>.yaml`, НЕ `<role>.yaml`: иначе шарды перезаписывают друг друга и агрегация тихо соберёт последний;
- лид ведёт **манифест ожидаемых шардов** (`.research/_shards.yaml`: role → [batch-id, статус]) и агрегирует ТОЛЬКО когда все ожидаемые получены. Недополученный шард — не «меньше данных», а неизвестная дыра: фиксируется в `gaps[]` явно.

Та же дисциплина для `source=code`, если крупная роль дробится по модулям.

## 6. Агрегация

После возврата всех YAML — один проход лид-агента:

1. **Склей поля по ролям** в единые коллекции (ключи дедупа — в [invocation-contract.md § 7](invocation-contract.md)):
   - `functions[]`, `nfrs[]`, `rules[]`, `entities[]`, `dictionaries[]` — из `requirements-scope`
   - `components[]`, `capabilities[]` — из `components-scope`
   - `scenarios[]`, `algorithms[]` — из `scenarios-scope`
   - `apis[]`, `data_schemas[]`, `openapi_specs[]`, `proto_files[]` — из `data-scope`
   - `repo_entries[]`, `component_maps[]`, `scenario_traces[]`, `data_maps[]` — из `codemap-scope`
   - **только `source=docs`**: `machine_artifacts[]` — из `data-scope` (готовые контракты: path, machine_format, target_artifact, operations); `test_cases[]` — из `testcases-scope` (с `traces_to`); каждая находка docs-ветки несёт `declared: true` и `source: <path>#<якорь>`
2. **Сними дубли** по ключам из [invocation-contract.md § 7](invocation-contract.md). Ключи docs-коллекций: `machine_artifacts` — по `path`; `test_cases` — по (`name`, `source`). Находку с `declared: true` НЕ схлопывать с находкой из кода: это разные факты (заявленное ≠ фактическое), их сводит сверка в `recover source=both`, а не дедуп.
3. **Сводные gaps** — объединить все `gaps[]`. Это вопросы к Этапу 2 интервью в вызывающем скилле.
4. **Проверка полноты — set-diff, а не «есть хотя бы одна запись».** «В слое что-то есть» не отличает 1000 разобранных материалов от 1 разобранного и 999 потерянных. Поэтому:
   - **`source=code`**: для каждого слоя — есть ли записи (§ 6.1); пусто при `target=factory-spec` → gap.
   - **`source=docs`**: детерминированный set-diff. `expected` = список файлов, полученный `find` на шаге сайзинга (лид считает его САМ, до субагентов); `inventoried` = пути из описи; `processed` = пути/срезы, покрытые батчами фазы B. Итог агрегата несёт `coverage: {expected: N, inventoried: N, processed: N, status: complete|partial}` и поимённый список непокрытых. **`partial` — законный исход, тихая потеря — нет.**

Результат возвращается в вызывающий скил в виде пути к `.research-notes.md` + краткая сводка. Сырой YAML в контекст лида не копируется.

## 7. Лимиты для слабой модели

- **Числа — явные**: «2 субагента», «≤15 файлов», «≤200 слов summary».
- **Промпт — копипаст** из [research-roles.md](research-roles.md), без импровизации.
- **Запуск — строго параллельно**: все вызовы tool'а запуска субагента — одним сообщением лида. См. [invocation-contract.md § 3](invocation-contract.md).
- **Рекурсия субагентов запрещена.**
- **Сырые тексты из субагентов в контекст лида не копируются.** Храни только YAML-сводки.
- **Персистентность обязательна для Среднего и Крупного**, рекомендуется для Малого. Каждый возвратный YAML сразу идёт в `.research/<role>.yaml`, итоговый агрегат — в `.research/_aggregate.yaml` и `.research-notes.md`. См. [invocation-contract.md § 8](invocation-contract.md).
- **Близко к 200к** — после записи YAML в файлы выгружай сырые выводы из контекста лида.
- **`.research/` ПЕРСИСТЕНТЕН** — `.research/` и `.research-notes.md` сохраняются для аудита (аудит-след контекста под артефакт), автоматически не удаляются. Если фабрика версионируется, держать их под git нормально.

## 8. Fallback: нет tool запуска субагента

Последовательный sweep по каталогу ролей: для каждой роли — прочитай её YAML-контракт из [research-roles.md](research-roles.md), пройди по указанным путям сам, заполни YAML в рабочих заметках. В конце — та же фаза агрегации.

**Понижение класса НЕ уменьшает массив.** «Крупный → стратегия Среднего» меняет только число агентов; когда агентов ноль, читать всё равно приходится столько же — просто в одно окно. Поэтому в fallback работает жёсткий лимит:

- Лид читает материалы ПОРЦИЯМИ с лимитом батча и после каждой сбрасывает YAML на диск (`.research/<role>-<batch-id>.yaml`), не удерживая содержимое в контексте.
- Массив, который не проходит даже так (крупный класс: >500 файлов кода / >150k слов), — **STOP**: честно сообщи, что без механизма субагентов этот объём не разбирается, и предложи либо сузить `roots`/`docs` до части, либо запустить в harness'е с субагентами. Молча «прочитать сколько влезет» и выдать это за research — недопустимо: неполнота не видна, а решения на ней строятся.
