# Guardrails для masterspec-propose

Запреты, уровни обязательности и чек-лист ревью для `change.md`.

---

## 1. Что такое change.md (роль документа)

Это предложение системного аналитика. Описывает **что** и **зачем** меняется в артефактах фабрики. **Не описывает как в коде** — это задача `masterspec-impl-plan` (dev-design) и `masterspec-implement`.

---

## 2. Запреты в change.md

### 2.1. Общие запреты

- **Не копировать код приложения.** Код — в `design.md`/`tasks.md`, не в change.md.
- **Не перечислять unit-тесты** поимённо. Покрытие тестами — в §8 как AC уровня change, не список тестов.
- **Не вставлять целиком конфиг-файлы.** Если меняется конфиг — описать какие параметры изменятся (логически), без дампа yaml целиком.
- **Не класть в change.md детальный план миграции с SQL-командами.** План миграции — в `design.md`. В change.md только: что мигрируется и принципы (zero-downtime / downtime / rollback window).

### 2.2. Запреты по дисциплине слоёв

Применяются к **каждому diff-блоку** и **каждому файлу в `new/`** согласно `masterspec/references/layer-discipline.md`:

- В правках/создании артефактов слоя требований (`01-requirements/`): **НЕТ** имён компонентов, таблиц, технологий, файлов, URL.
- В правках/создании артефактов слоя спецификаций (`02-specifications/`): **НЕТ** кода, путей файлов, имён классов, конкретных таблиц БД, технологий.
- В правках артефактов кодового слоя (`03-codemap/`): `file:line` и реальные имена — единственное допустимое.

При нарушении — блок, заставить переформулировать.

### 2.3. Запреты по формату

- **Не сохраняй HTML-комментарии** из шаблона. Они — инструкции, не содержимое.
- **Не оставляй пустые секции без явной маркировки.** Если секция не затрагивается — поставь `Нет изменений.`
- **Не вставляй примеры из `references/example-change.md`** в итоговый файл — они ориентир формата, не копипаст.

---

## 3. Уровни обязательности (ДОЛЖЕН / СЛЕДУЕТ / МОЖЕТ)

Используй эти маркеры в формулировках AC и критериев приёмки:

| Уровень | Значение |
|---------|----------|
| **ДОЛЖЕН** | Обязательное требование. Невыполнение = change не принят. |
| **СЛЕДУЕТ** | Рекомендуемое. Отсутствие выполнения требует обоснования. |
| **МОЖЕТ** | Допустимое. Необязательно. |

---

## 4. Чек-лист ревью change.md

Перед тем как сообщить пользователю «change готов к PR» — пройди по чек-листу:

### 4.1. Структурная полнота

- [ ] Шапка заполнена (все поля, статус = `На согласовании`)
- [ ] §1 заполнен (цель, инициатор, приоритет, затронутые слои)
- [ ] §2 — хотя бы одна непустая таблица (MODIFIED / ADDED / REMOVED)
- [ ] §3 заполнен (да/нет + план миграции, если нет)
- [ ] §4 — блок для каждой строки §2.1 (или `Нет изменений.`)
- [ ] §5 — список файлов `new/` для каждой строки §2.2 (или `Нет изменений.`)
- [ ] §6 — обоснование для каждой строки §2.3 (или `Нет изменений.`)
- [ ] §7 — анализ ссылок (или `Нет изменений.`)
- [ ] §8 — критерии приёмки change

### 4.2. Содержательная точность

- [ ] Каждый MODIFIED-slug — существующий файл в `masterspec/`
- [ ] Каждый ADDED — файл `new/<slug>.md` создан, YAML-фронтматтер валиден (`type`, `slug`, `factory`, `status: draft`, `updated`)
- [ ] Каждый REMOVED — файл в `masterspec/` существует на момент создания change
- [ ] Slug'и уникальны: для каждого slug'а из §2.2 выполнен `Glob("masterspec/**/<slug>.md")` (исключая `masterspec/changes/`) и ничего не найдено. Пройти эту проверку механически, а не на глаз — особенно при ≥3 ADDED.

### 4.3. Дисциплина слоёв

- [ ] В каждом diff-блоке и файле `new/` соблюдены запреты по слоям (`masterspec/references/layer-discipline.md`)
- [ ] В §7 все новые ссылки идут снизу вверх (нет обратных)

### 4.4. Реализуемость

- [ ] Каждый diff-блок имеет указание типа правки (`modify-bullet` / `replace-section` / `add-subsection`)
- [ ] Блок `ДО:` для `modify-bullet` — точная копия строки из целевого файла (можно проверить через Read + Grep)
- [ ] Для `replace-section` / `add-subsection` — указан существующий в целевом файле заголовок (## / ###)

### 4.5. Консистентность

- [ ] §2.1 ↔ §4: число строк MODIFIED совпадает с числом diff-блоков
- [ ] §2.2 ↔ §5 ↔ `new/`: число строк ADDED, строк в §5 и файлов в `new/` совпадает
- [ ] §2.3 ↔ §6: число строк REMOVED совпадает с числом строк в §6

## 5. Ревью в субагенте (опциональный шаг)

Если доступен tool запуска субагента — запусти отдельного ревьюера с чек-листом из §4 выше:

```
Ты — read-only ревьюер change.md для мета-модели masterspec. Прочитай:
1. <путь к change.md>
2. <путь к masterspec/references/layer-discipline.md>
3. <путь к этому guardrails.md>

Проверь change.md по чек-листу §4 guardrails.md. Для каждой строки — статус
(OK / FAIL) и короткое объяснение при FAIL.

Границы: Не модифицировать ни одного файла. Не запускать других субагентов.

Формат вывода:
- checklist_results: [{section, item, status, reason_if_failed}]
- blocking_issues: [<проблемы, из-за которых change НЕЛЬЗЯ отправить в PR>]
- soft_issues: [<замечания для улучшения, не блокирующие>]
```

Лид скилла применяет blocking_issues в качестве правок перед финальной сдачей.

---

## 6. Что делать с найденными проблемами

- **blocking_issue** (нарушение дисциплины слоёв, отсутствие обязательного раздела, дубликат slug'а, отсутствующий файл в `new/`) → исправить до сдачи. Не сообщай «change готов», пока есть blocking.
- **soft_issue** (стиль формулировок, неполнота объяснения, отсутствие AC уровня change) → зафиксируй в «Открытых вопросах» change.md и сообщи пользователю, чтобы он решил до merge PR.
