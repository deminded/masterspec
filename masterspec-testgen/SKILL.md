# masterspec-testgen — исполняемые автотесты со слепой верификацией

Брат `masterspec-implement`. implement переводит спеку в код; я перевожу спеку + её `tc-`
(тест-кейсы) в ИСПОЛНЯЕМЫЕ автотесты и кладу их в ОТДЕЛЬНЫЙ репозиторий автотестов. Метод —
element-workflow (`../masterspec/references/patterns/element-workflow.md`): изолированный контекст по
фокус-набору, чужое — через ссылку, ничего не домысливаю сверх источника.

**Границы роли:** `tc-` артефакты (спец-слой, `08-test-cases`) я НЕ генерирую — их порождает
`derive layer=spec` / `evolve`+`apply-change`. На вход беру ГОТОВЫЕ tc- как постановку. Одна стадия:
`tc- (+спека) → runnable`.

## Вход и выход — два репозитория
- **`spec=`** — репо/фабрика со СПЕКОЙ и уже сгенерированными `tc-` (tc-acc/tc-int/tc-flt в
  step-contract-формате) + связанные scn/api/rules/nfr. ВХОД, постановка. **Read-only.**
- **`tests=`** — отдельный репо автотестов, ВЫХОД. Пишу СЮДА **только runnable**, плоско («валом»:
  строгую раскладку не навязываю — соглашение репо доопределится). Папку `masterspec/` НЕ создаю.
  - (опционально, выключено по умолчанию) сводная таблица кейсов CSV — включается флагом, когда формат
    колонок согласован с тестировщиками.
- **`framework=`** — целевой стек, **дефолт `junit-kotlin`**; переопределяется аргументом (pytest / …).
  Не зашит: маппинг шага в assert — `references/executable-mapping.md`.
- **Чек-лист** — `references/test-rules.md` (обобщённый rules-профиль; провайдер-нейтральный).

## Что делаю
Для каждого `tc-` из `spec` собираю исполняемый тест по `references/executable-mapping.md`: каждый шаг
(Действие/Тестовые данные/Ожидаемый результат) → arrange/act/assert; каждая строка ожидаемого →
отдельный assert; презентационные требования → контентные assert; лог-чек ошибок → assert журнала;
негатив → assert специфицированной ошибки. Полнота покрытия: каждый `tc-` спеки → тест в `tests`.
Каждый тест несёт **back-ref на свой `tc-`** (тег/аннотация, напр. `testCaseKey`) — лёгкая связка для
трассы `tc → тест → (codemap) → код` без владения codemap здесь.

## Обязательный контур генерации (ядро скилла)
Любой сгенерированный тест проходит петлю из ДВУХ ролей с изоляцией контекста:

1. **Агент-кодер** (инструменты доступны) — пишет runnable-тест из tc/постановки + чек-листа.
2. **СЛЕПОЙ агент-верификатор** (отдельный субагент, изолированный) — получает РОВНО: tc/постановку +
   чек-лист + СГЕНЕРИРОВАННЫЙ код теста. **НЕ видит рассуждений кодера** — только вход и результат.
   Проверяет по каждому пункту чек-листа: реально ли код ассертит то, что в tc (не заглушка, не
   always-true, не закомментировано), покрыта ли каждая строка ожидаемого, контентны ли презентационные
   проверки, есть ли лог-чек у ошибок. Возвращает `pass` | `fail + перечень пробелов` (пункт → чего нет).
3. **Петля**: `fail` → кодер правит ТОЛЬКО по названным пробелам → переверификация свежим слепым
   верификатором. До `pass` или до `rounds` (дефолт 3). Не сошлось → `BLOCKER: тест не сошёлся с tc по
   <пункты>` (честно держит артефакт неготовым).

> **Зачем слепота.** Верификатор, видящий рассуждения кодера, поддаётся его рационализациям и
> пропускает пробел. Слепой судит тест по постановке, а не по истории написания — тот же принцип, что
> состязательная проверка «default to refuted» и отдельный проверяющий у оси O5. Генератор и судья не
> должны быть одним контекстом.

## Замыкание детерминированным гейтом
Поверх LLM-контура (соответствие постановке) — машинное: сгенерированные тесты СОБИРАЮТСЯ/компилируются
целевым стеком (`framework`) без ошибок; каждый `tc-` спеки имеет соответствующий тест (set-diff
tc↔тест, нет непокрытых). Валидность самих `tc-` (step-contract, лог-чек, O_T) обеспечивается ДО меня
детектором формы на спеке (`derive`/`verify`) — я беру готовые.

## Границы (что НЕ делаю)
- Не генерирую `tc-` (это `derive`/`evolve`), не правлю спеку — при расхождении tc↔реальность правится
  источник и tc-, тест перегенерируется.
- Не тестирую раздел **Требования безопасности** — для него отдельные проверки.
- Не трогаю `COOKBOOK.md` и канон-ядро — я узел-исполнение, брат implement.

## Композиция
`derive layer=spec` / `evolve`+`apply-change` порождают `tc-`; `verify` вычитывает спеку; я (как
`implement`) — отдельный такт «спека+tc → исполняемая реализация», но в репозиторий автотестов.

**Связь с кодовым покрытием — рядом, не внутри меня** (симметрия: как `tc-` не мой, а derive/evolve,
так codemap не мой, а `recover`/`codemap`). Я лишь ТЕГИРУЮ каждый тест back-ref'ом на `tc-`; трасса
`tc → scn → код` живёт в `03-codemap`. Отдельный **codemap-шаг В КОНЦЕ процесса** подхватывает эти теги
и сводит forward-покрытие «код ← тест» — не в testgen.

См. `references/test-rules.md`, `references/executable-mapping.md`.
