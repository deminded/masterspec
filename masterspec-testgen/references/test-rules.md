# Чек-лист автотестов testgen (fidelity runnable к tc/спеке)

Обобщённый, провайдер-нейтральный. `tc-` — ВХОД (их сделал derive/evolve); testgen генерит RUNNABLE.
Каждый пункт — ПРОВЕРЯЕМЫЙ инвариант СООТВЕТСТВИЯ runnable своей постановке (tc + спека): агент-кодер
соблюдает, СЛЕПОЙ верификатор проверяет по каждому пункту (пункт → чего в тесте не хватает). Правила
про содержание (R1–R3, R13–R14) и полноту (R4–R8) читаются как «runnable верно и полно реализует то,
что уже зафиксировано в tc-», а не как «сгенерируй tc-».

## Источник и домысел
- **R1 Только из постановки.** Тест-кейс создаётся ТОЛЬКО на основе поданной аналитики. Не
  придумывать сценарии, данные, коды ошибок, которых нет в источнике.
- **R2 Негатив — из специфицированных ошибок.** Негативный сценарий генерируется только по описанной
  в спеке ошибке ИЛИ описанному негативному пользовательскому сценарию. Нет «правдоподобного» негатива.
- **R3 Данные из спеки.** Тестовые данные конкретны и взяты из постановки. Если тело запроса/ответа
  намеренно опущено — явно объяснить почему (не «забыл»).

## Полнота и минимальность
- **R4 Фокус покрытия.** Обязательные классы: (а) позитив; (б) негатив с ожидаемой ошибкой;
  (в) граничные значения; (г) критические интеграционные точки. Прочее — по необходимости.
- **R5 Дедуп.** Минимизировать число кейсов: объединять схожие сценарии, устранять дубли. Один
  кейс — один связный сценарий; один шаг — одна проверка.

## Секция-триггеры (ОБЯЗАТЕЛЬНЫЕ тесты при наличии раздела)
- **R6 Аудит.** Есть раздел «Аудит» → обязательны кейсы проверки функциональности аудита.
- **R7 Мониторинг.** Есть «Мониторинг»/«Точки мониторинга»/«Сбор метрик» → обязательны кейсы
  работоспособности мониторинга (метрика поднимается, событие фиксируется).
- **R8 Нагрузка/масштаб.** Есть load-profile / параметры масштаба (объём, частота, backlog) →
  обязателен нагрузочный/масштабный тест (не только happy-path на N=1). Прямая связь с
  load-profile-нервом (`../ROADMAP.md`): наивная генерация покрывает счастливый путь, не масштаб.

## Форма шага (step-contract) — совпадает с каноном tc masterspec
- **R9 Структура.** Каждый шаг = Действие / Тестовые данные / Ожидаемый результат. Без вложенных
  ```json/```curl-огородок внутри полей; данные — в поле «Тестовые данные».
- **R10 Лог-чек.** Для каждой ошибки/технического сбоя — отдельный шаг проверки лога/журнала (запись
  есть, со специфицированной причиной). Совпадает с O_T-требованием form-detector.
- **R11 Авторизация (Шаг 0).** Для UI/сессионных сценариев — первый шаг «Авторизоваться … » →
  «Авторизация выполнена». Для не-UI не требуется.
- **R12 Вызов внешней проверки** («call to test») — отдельным шагом-ссылкой без ожидаемого результата,
  если стек так устроен.

## Презентация и реализм (из догфуда, `../ROADMAP.md`)
- **R13 Презентационное требование → контентный assert.** Требование «показать X» (автора, дату,
  deep-link, формат) проверяется СОДЕРЖАНИЕМ результата (поле = ожидаемое значение), не только путём
  (доставлено/маршрут). Path-тест без content-assert для презентационного требования — пробел.
- **R14 Реализм фикстур.** Фикстуры воспроизводят ПРОДАКШН-форму артефактов (как их строит реальный
  код), а не упрощённую — иначе тест слеп к значениям, которые триггерит прод.

## Границы
- **R15 Безопасность не тестируется здесь.** Раздел «Требования безопасности» — вне scope этого
  скилла (для него отдельные проверки). Не генерировать по нему кейсы.
- **R16 При редактировании** существующих тестов — актуализировать/исправлять, НЕ плодить новые.

## Использование верификатором
Слепой верификатор для каждого пункта R1–R14 (R15/R16 — как ограничения) отвечает: соблюдён ли в
данном тесте относительно ЭТОЙ постановки; при `fail` — какой пункт и чего конкретно не хватает.
