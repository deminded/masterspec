# Маппинг tc → исполняемый автотест (стадия 2)

Как шаг tc (Действие / Тестовые данные / Ожидаемый результат) превращается в исполняемый тест.
Абстрактно; конкретный синтаксис — по `framework`. Скелеты внизу — иллюстрация, не догма.

## Единица
- Один tc-кейс → один тест-метод (имя = суть кейса, без нумерации).
- **Back-ref (R17):** тест несёт ссылку на свой `tc-` — тег/аннотация стека (напр. `@Tag("<tc-key>")`
  в junit-kotlin, маркер в pytest). Для трассы `tc → тест → codemap → код`.
- Предусловия tc → **setup / fixtures**. Фикстуры воспроизводят ПРОДАКШН-форму данных (R14): как их
  строит реальный код, не упрощённо — иначе тест слеп к боевым значениям.
- Каждый шаг → блок **arrange (данные) → act (действие) → assert (ожидаемый)**.

## Поля шага
- **Действие** → вызов: API-запрос / UI-драйвер / публикация события / вызов метода — по природе
  контракта из постановки.
- **Тестовые данные** → входные параметры/фикстуры шага. Пусто в tc → без входа. Данные берутся из
  постановки (R3), не выдумываются.
- **Ожидаемый результат** → assert'ы: **по каждой строке ожидаемого — отдельный assert**.
  - Презентационное требование (R13) → **контентный assert**: проверяется ЗНАЧЕНИЕ поля (автор/дата/
    формат/deep-link = ожидаемое), а не только статус/путь/факт доставки.
  - Различимые исходы дают различимые проверки: «не найдено» ≠ «истекло» (assert на конкретный код/тип).

## Специальные шаги
- **Лог-чек** (R10) → assert на запись журнала: ровно одна запись ожидаемого события со
  специфицированной причиной (для ошибок/техсбоев).
- **Негатив** (R2) → assert конкретной ошибки/кода ИЗ СПЕКИ; тело ошибки — если задано в постановке.
- **Нагрузка/масштаб** (R8) → отдельный тест с N>1 / backlog / частотой из load-profile — не
  дублировать happy-path на масштаб, а проверить инвариант под объёмом.

## Провайдер-нейтральность
`framework` — параметр (pytest / junit-kotlin / … ). Маппинг «шаг → arrange/act/assert» одинаков;
меняется синтаксис и раннер. Не зашивать стек в скилл: целевой стек приходит аргументом.

Скелет (pytest):
```
def test_<суть>():
    ctx = setup_production_like_fixture(...)   # R14
    resp = act_do_something(ctx, <данные>)     # Действие + Тестовые данные
    assert resp.status == <ожидаемый статус>
    assert resp.body["<поле>"] == <ожидаемое значение>   # R13 контентный assert
    assert find_log_entry(ctx, "<событие>", cause="<причина>")  # R10 если ошибочный
```
Скелет (junit-kotlin) — та же тройка arrange/act/assert, синтаксис стека.

## Слепая верификация стадии 2
СЛЕПОЙ верификатор получает tc + чек-лист + СГЕНЕРИРОВАННЫЙ код теста (без рассуждений кодера) и
проверяет: код реально ассертит то, что в Ожидаемом (не заглушка, не always-true, не закомментировано);
каждая строка ожидаемого имеет assert; презентационные — контентные; лог-чек присутствует у ошибок.
`fail` → перечень: какой шаг/строка ожидаемого не проверяется кодом.
