# pack-template — скелеты и разобранный пример

Схема и семантика — `../../masterspec/references/patterns/guardrails.md`. Здесь — рыба
для заполнения и один сквозной пример трансформации «сырой абзац → карточка».

## manifest.yaml (скелет)
```yaml
id: <kebab-id-пакета>
version: 0.1
source: universal | corporate | domain | factory
owner: <владелец>
description: >
  <одна-две строки: что за нормы и для каких фабрик>
rules:
  - id: <rule-id>
    applies_to: {operation: [<...>], layer: [<...>], element_type: [<...>], input_type: [<...>], tags: [<...>]}
  # ...по строке на правило; указывай ТОЛЬКО нужные ключи селектора
```

## rules/<rule-id>.md (скелет карточки)
```markdown
# <rule-id>: <короткое имя нормы>
**Statement.** <одна проверяемая норма — что должно быть истинно у артефакта/входящего>
**Conformance checklist**
(blocking — нет → стоп):
- [ ] <пункт> [machine: <как проверяется механически>]
- [ ] <пункт> [llm]
(non-blocking — дозапросить, но не стоп):
- [ ] <пункт> [llm]
**Rationale.** <коротко ЗАЧЕМ: какой отказ/дефект норма предотвращает>
**Anti-patterns.** <как норму обходят/имитируют>
**Criticality.** critical | major | minor
```

## Разбор: как из абзаца рождается карточка
**Сырой абзац (например, из корпоративного API-стандарта):**
> «Все публичные REST-эндпоинты обязаны быть версионированы через префикс пути (/v1/…).
> Ломающие изменения выпускаются только поднятием мажорной версии; старая версия живёт
> не менее двух релизных циклов. Ответы об ошибках — по RFC 7807 (problem+json).»

Здесь ТРИ нормы, не одна — разбиваем:
1. `api-path-versioning` — версия в префиксе пути.
2. `api-breaking-change-policy` — ломающее только через мажор + окно жизни старой версии.
3. `api-error-format` — problem+json (RFC 7807).

**Карточка `api-path-versioning.md`:**
```markdown
# api-path-versioning: версионирование публичных API префиксом пути
**Statement.** Каждый публичный контракт `api-` несёт версию в префиксе пути (/vN/…).
**Conformance checklist**
(blocking):
- [ ] В контракте задан путь с префиксом версии [machine: regex ^/v\d+/]
- [ ] Версия согласована со статусом ломающих изменений артефакта [llm]
(non-blocking):
- [ ] Политика депрекации предыдущей версии упомянута или сослана [llm]
**Rationale.** Версия в пути делает несовместимость явной и маршрутизируемой; без неё
ломающее изменение бьёт молча по действующим потребителям.
**Anti-patterns.** Версия только в заголовке «на будущее»; /v1 как украшение при
фактически меняющемся контракте без смены N.
**Criticality.** major
```
Строка в manifest:
```yaml
  - id: api-path-versioning
    applies_to: {operation: [derive, gen, evolve], layer: [spec], element_type: [api]}
```

Мораль примера: нарратив («живёт не менее двух циклов») сам по себе не норма артефакта —
он становится проверяемым пунктом ЛИБО отдельной карточкой (`api-breaking-change-policy`),
ЛИБО уходит в PACK-REPORT как политика, если проверить на артефакте её нельзя.
