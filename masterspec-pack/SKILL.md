# masterspec-pack — собрать guardrail-пакет из документа

Беру сырой корпоративный документ и превращаю его в guardrail-пакет, пригодный для
подключения к фабрике. Схема пакета и механика применения — в kernel
(`../masterspec/references/patterns/guardrails.md`); эталон готового пакета —
любой существующий `<pack>/manifest.yaml`. Карточку и манифест-скелет держу в
`references/pack-template.md`.

**Граница.** Пакет — корпоративный/проектный материал, он лежит ВНЕ masterspec и
НЕ коммитится в kernel. Скилл только рождает пакет и (по запросу) вписывает ссылку
на него в конфиг фабрики. Отсечку (severity) пакет не знает — она у фабрики.

## Вход
- `from=` — путь к сырому документу (обязателен). Один прогон = один источник → один пакет.
- `source=` — уровень: `universal` (общеинженерные) | `corporate` (компания целиком) |
  `domain` (предметная область, напр. банковский) | `factory` (только этот проект).
  При конфликте правил на применении выигрывает более специфичный (factory > domain >
  corporate > universal) — но это решается НЕ здесь, а при применении.
- `out=` — каталог пакета (дефолт: рядом с источником, `<имя>-guardrails/`).
- `factory=` — корень фабрики; если задан, в её `masterspec-config.yaml` дописывается `- path: <pack>`.
- `owner=` — владелец пакета (дефолт: спросить или из git config).

## Выход (контракт — что считается «готово»)
1. `<pack>/manifest.yaml` — `id, version, source, owner, description, rules[]`; каждый
   элемент `rules[]` = `id` + `applies_to` (селекторы).
2. `<pack>/rules/<rule-id>.md` — по карточке на правило: **Statement** (одна проверяемая
   норма), **Conformance checklist** (каждый пункт помечен `[machine]` или `[llm]` и
   отнесён к blocking / non-blocking), **Rationale** (коротко — ЗАЧЕМ, не пересказ),
   **Anti-patterns**, **Criticality** (`critical|major|minor`).
3. `<pack>/PACK-REPORT.md` — покрытие источника: сколько правил извлечено, что осознанно
   отброшено (не норма, а нарратив), и открытые вопросы владельцу (нормы, которые нельзя
   сделать проверяемыми без уточнения).
4. Если задан `factory=` — строка `- path: <pack>` в его `masterspec-config.yaml` (файл
   создаётся, если его нет).

## Дисциплина прогона (обязательна)
Полный вход, узкий деливерабл, запись по ходу:
- **Документ читаю целиком** — контекст не режу; норма может зависеть от соседнего абзаца.
- **Пишу карточку сразу, как только правило вычленено** (checkpoint per rule), а не в конце
  оптом. Обрыв прогона стоит одной карточки, не всего пакета.
- **manifest.yaml собираю последним** — из уже написанных карточек, не вперёд них.
- Мандат узкий: цель — не «понять документ», а произвести карточки по схеме. Всё, что не
  ложится в карточку, — в PACK-REPORT, не в тело правил.

## Алгоритм дистилляции
1. Прочитай `from=` целиком. Отдели НОРМЫ (проверяемые ограничения) от нарратива,
   мотивации, примеров. Норма — то, что можно предъявить артефакту как «выполнено/нет».
2. Разбей на атомарные правила: **одна норма = одна карточка**. Не склеивай разнородное,
   не дроби единое. Дай `id` в kebab-case по смыслу (напр. `api-versioning`, `pii-masking`).
3. Для каждого правила определи `applies_to` — ГДЕ в конвейере фабрики норма кусается
   (см. «Селекторы» ниже). Это главное суждение: правило про генерацию API ≠ правило про
   приём тикета.
4. Заполни карточку. Каждый пункт чек-листа помечай:
   - `[machine]` — проверяется механически (наличие поля/секции, regex, именование, счётчик);
   - `[llm]` — нужно суждение (смысловая полнота, наблюдаемость критерия).
   И `blocking` (нет → стоп) vs `non-blocking` (нет → дозапрос, не стоп). Проставь `criticality`.
5. Запиши карточку в `rules/<id>.md` НЕМЕДЛЕННО (checkpoint).
6. Когда карточки готовы — собери `manifest.yaml`. Провалидируй: каждое правило манифеста
   имеет карточку и наоборот; селекторы используют известные значения словаря; `source`
   выставлен по аргументу.
7. Напиши PACK-REPORT: покрытие, отброшенное, открытые вопросы.
8. Если задан `factory=` — допиши `- path: <pack>` в `masterspec-config.yaml` (идемпотентно:
   не дублируй уже присутствующий путь).

## Селекторы applies_to (словарь)
И-логика внутри правила (все ключи должны совпасть). Значения — из мета-модели kernel:
- `operation`: `intake | derive | gen | evolve | verify`
- `layer`: `req | spec | code`
- `element_type`: `fn | nfr | cdm` (req) · `cmp | scn | api | data | alg` (spec) ·
  `cmap | trace | dmap` (code) — актуальный список в `../masterspec/meta_model.md`
- `input_type`: `ticket | business-requirement | arch-sketch | …` (для `intake`)
- `boundary`: `intra-factory | inter-factory | perimeter` (для `element_type: api` — уровень границы/security, `../masterspec/references/boundary-registry.md`)
- `tags`: свободные метки проекта
Правило БЕЗ ограничения по ключу = применимо ко всем его значениям. Указывай минимально
достаточный набор — лишний селектор сужает применимость молча.

## Честность (не выдумывать)
- Норму, которую нельзя сделать проверяемой без уточнения, НЕ превращай в карточку с
  придуманным критерием — вынеси в PACK-REPORT открытым вопросом владельцу.
- Не добавляй ограничений, которых нет в источнике. Пакет — дистиллят документа, не твоих
  представлений о хорошем.
- Держи карточки лаконичными: statement + чек-лист + короткий rationale + анти-паттерны +
  criticality. Без воды (норма лаконичности — как в change-format).

## Границы
- Пакет не коммитить в kernel masterspec. Конфликты источников решаются на ПРИМЕНЕНИИ по
  специфичности, не здесь. Поставка глоссария через MCP — возможное расширение интерфейса
  пакета, не задача этого скилла.

> **Без субагентов (`Task`):** скилл атомарен по природе (один документ → один пакет),
> субагенты не требуются; при большом документе разнеси проход по разделам последовательно.
