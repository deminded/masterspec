# Merge workflow для фабрики (мультиартефактный мерж)

Детальная механика вливания change.md + `new/` в артефакты фабрики. Целевой артефакт — не один `spec.md`, а дерево артефактов в `masterspec/01-*/02-*/03-*/04-*`. Применяются (а) diff-блоки из §4 change.md в существующие файлы, (б) копирование файлов из `new/` в целевые директории по `type:` фронтматтера.

---

## 1. Git как единая точка отката

Работай над всей директорией `masterspec/` (исключая `changes/`) — она и есть «целевой артефакт». Точка отката — `HEAD`.

### 1.1. Предпроверка

```bash
git status --porcelain masterspec/ | grep -v "^.. masterspec/changes/"
```

Должно быть пусто. Если есть незакоммиченные изменения в `masterspec/01-*/02-*/03-*/04-*` или `00-masterspec-index.md` — блок, попроси пользователя сначала закоммитить или откатить (через AskUserQuestion).

### 1.2. Команда отката после мержа

Единая (с явным исключением `changes/`, чтобы директория change'а не пострадала):

```bash
git checkout HEAD -- masterspec/ ':(exclude)masterspec/changes/'
```

Откатит все diff-блоки, все скопированные файлы из `new/`, перегенерацию `00-masterspec-index.md`. Pathspec-исключение гарантирует, что `masterspec/changes/<name>/` (под отдельным жизненным циклом change'а) не затрагивается. Если change ещё не закоммичен — его файлы untracked, и `checkout` их и так не тронет; исключение страхует случай уже закоммиченного change.

**`.bak`-файлы создавать запрещено.**

## 2. Парсинг change.md

Прочитай `masterspec/changes/<name>/change.md` целиком и извлеки:

- Шапку: статус, фабрика.
- §2.1 MODIFIED — таблица (slug, путь, описание правки).
- §2.2 ADDED — таблица (slug, путь-куда, шаблон).
- §2.3 REMOVED — таблица (slug, путь, причина).
- §4 — diff-блоки (для каждой строки §2.1).
- §5 — список файлов в `new/` (дубль §2.2).
- §6 — обоснования REMOVED.
- §7 — влияние на ссылки (для финальной валидации).

Строки §2.1 MODIFIED бывают двух видов — различай по пути (`.md` против объявленного `sidecar:`):
- **путь `.md`** (обычный артефакт или компаньон) → правится diff-блоком в §4 (ниже);
- **путь = объявленный `sidecar:`** (машинный сайдкар любого формата) → diff-блока в §4 НЕТ; есть файл-замена `new/<sidecar>` (полная замена целиком, обработка — §4.6). Отсутствие diff-блока для такой строки — норма, не ошибка парсинга.

Для каждого diff-блока в §4 (т.е. для каждой `.md`-строки §2.1) извлеки:
- `**Файл**:` — путь относительно masterspec/
- `**Раздел**:` — цепочка заголовков (`## ...` / `### ...`)
- `**Тип правки**:` — `modify-bullet` | `replace-section` | `add-subsection`
- Блок `ДО:` (в code fence)
- Блок `ПОСЛЕ:` (в code fence)

Если блок не распарсен — конфликт, см. §6.

## 3. Dry-run — сформируй список операций

Перед применением соберй план и покажи пользователю через AskUserQuestion для подтверждения.

Для REMOVED-компаньона с `sidecar:` план фиксирует ОБА пути пары (компаньон + сайдкар — прочитав фронтматтер ДО удаления) и валидирует, что `sidecar:` — локальный basename; невалидный/отсутствующий сайдкар = конфликт (§6) ещё на dry-run, до мутаций. Для ADDED-пары план показывает оба целевых пути (коллизия проверяется по обоим, §5.3).

Пример отчёта:

```
## Dry-run apply-change: add-retry-for-failed-delivery

### MODIFIED (diff-блоки)
- fn-send-notification.md → Критерии приёмки / AC-03 : modify-bullet
- cmp-delivery-router.md → Возможности : add-subsection (cap-retry)

### ADDED (из new/)
- scn-retry-flow.md → 02-specifications/02-scenarios/
- adr-backoff-policy.md → 04-decisions/

### REMOVED
(нет)

### Обновление 00-masterspec-index.md
- будет перегенерирован целиком процедурой полной перегенерации индекса (§8).
  §1 «Паспорт» и §7 «Белые пятна» сохранятся дословно, §3–§6 перестроятся по файлам фабрики.

Применить? [Да / Нет]
```

Только после явного «Да» — переходи к применению.

## 4. Применение diff-блоков (MODIFIED)

Для каждого блока из §2 (по порядку в §4 change.md):

### 4.1. Прочитай целевой файл

`Read(<masterspec/<путь>)`. Если файл отсутствует — конфликт (§6).

### 4.2. Найди раздел по заголовку

Ищем раздел **по тексту заголовка, не по номеру**. Регистр важен.

Парсинг:
1. Найди в файле строку, начинающуюся с `## <заголовок уровня 2>` (или `### ...` для подраздела).
2. Границы раздела — от найденного заголовка до следующего заголовка того же или выше уровня (или конца файла).

Если указан вложенный маршрут (`## A → ### B`) — сначала найди `## A`, затем внутри него найди `### B`.

Если раздел не найден — конфликт (§6). **Не пытайся создать раздел самостоятельно** — сигнал о рассинхроне change.md с текущим состоянием файла.

### 4.3. Применение по типу

#### modify-bullet

1. Извлеки текст из блока `ДО:` (между code fence'ами).
2. Нормализуй leading whitespace (trim leading spaces для поиска, но сохраняй в новом тексте).
3. Ищи этот текст **целиком, построчно** внутри найденного раздела.
4. Если найден уникально — замени на текст из `ПОСЛЕ:`.
5. Если не найден или найден несколько раз — конфликт (§6).

#### replace-section

1. Удали всё содержимое найденного раздела (между его заголовком и следующим заголовком того же уровня).
2. Вставь текст из `ПОСЛЕ:` (без заголовка самого раздела — только тело).
3. Сохрани заголовок раздела.

Блок `ДО:` для `replace-section` может быть пустым или содержать старое содержимое для истории ревью — его содержимое при мерже НЕ проверяется.

#### add-subsection

1. Найди раздел (обычно `## ...`).
2. Определи **границу раздела** — первую из следующих строк после заголовка раздела:
   - следующий заголовок того же уровня (`## ...` для раздела `## ...`) — это жёсткая граница, вставка идёт перед ним;
   - следующий заголовок более высокого уровня (`## ...` для раздела `### ...`, `# ...` для раздела `## ...`);
   - строка-разделитель `---` на отдельной строке — мягкая граница, вставка идёт перед ней (сохраняем разделитель между секциями);
   - конец файла — вставка в самый конец.
3. Внутри раздела найди позицию вставки:
   - если есть хоть одна подсекция уровнем ниже (`### ...` внутри `## ...`) — вставь `ПОСЛЕ:` сразу после последней подсекции и перед границей раздела (одна пустая строка до и после новой подсекции);
   - если подсекций нет — вставь `ПОСЛЕ:` непосредственно перед границей раздела (одна пустая строка до).
4. Если `ПОСЛЕ:` начинается с заголовка уровня ниже текущего раздела (например, `### cap-retry` внутри `## Возможности`) — это корректная подсекция. Если уровень некорректный — конфликт: diff-блок помечен `add-subsection`, но содержимое не является подсекцией целевого раздела.

**Примеры границ:**

```markdown
## Возможности          <- целевой раздел для add-subsection

### cap-foo
<тело>

### cap-bar
<тело>
                         <- вставка идёт СЮДА (перед следующей `## ...` или `---`)
---

## Следующий раздел
```

```markdown
## Возможности
<тело без подсекций>
                         <- вставка идёт СЮДА (перед `## ...` / `---` / EOF)

## Следующий раздел
```

### 4.4. Обнови YAML-фронтматтер

После применения diff-блока в файле — обнови `updated:` на сегодняшнюю дату:

```yaml
updated: 2026-04-23
```

Если поля нет — добавь.

### 4.5. Запиши файл

Единой записью через Write.

### 4.6. MODIFIED машинного сайдкара (api/data/scn/alg — любой формат по `sidecar:`)

Если строка §2.1 MODIFIED указывает на машинный сайдкар (путь = значению `sidecar:` компаньона, а не `.md`) — это НЕ diff-блок, а полная замена файла (`change-format.md §3.1`):
1. Целевой сайдкар — по ПОЛНОМУ пути из строки §2.1 (`masterspec/<путь-из-строки>`, не достраивать имя повторно) — ДОЛЖЕН существовать (иначе это ADDED — §5, не MODIFIED); его basename обязан совпадать с полем `sidecar:` компаньона того же slug. Нет файла → конфликт (§6).
2. Проверь, что новая версия `new/<sidecar>` парсится по своему `sidecar_format`: yaml/json — `yaml.safe_load`/`json.loads`; xml (bpmn/dmn/wsdl) — well-formed XML; текстовые нотации (mermaid/plantuml) — непустота без markdown-огородок ```. Битая → конфликт (§6).
3. Перезапиши целевой файл содержимым из `new/`. `updated:` обновляется в компаньоне `.md` (сайдкар метаданных masterspec не несёт).

## 5. Копирование `new/` в целевые директории (ADDED)

Для каждой строки §2.2 ADDED. Файл обычно `.md`. Машинный `api`/`data`/`scn`/`alg` — это ПАРА (`change-format.md §3.1`): компаньон `<slug>.md` (несёт masterspec-фронтматтер) + сайдкар, объявленный полем `sidecar:` (машинная спека любого формата); копируются ОБА, метаданные всегда из компаньона.

### 5.1. Прочитай компаньон `new/<slug>.md`

Извлеки YAML-фронтматтер. Машинный сайдкар (по полю `sidecar:`, если есть) фронтматтера masterspec не содержит — `type`/`slug`/`status`/`scope`, а также `sidecar_format`/`sidecar` берутся из компаньона.

### 5.2. Определи целевую директорию по `type:`

Используй таблицу маршрутизации `<Skill dir>/../masterspec/references/artifact-routing.md § 1`.

Специальные случаи:
- `type: api` — смотри поле `scope:` фронтматтера: `internal` → `02-specifications/04-apis/internal/`, `external` → `02-specifications/04-apis/external/`.
- `type: function` — смотри поле `block:` фронтматтера: если задано — поддиректория `01-requirements/02-functions/<block>/`; иначе — `01-requirements/02-functions/`.
- `type: decision-record` (`dr-*`) — путь НЕ вычисляется по `type:`: бери его из колонки «Путь (куда)» в §2.2 change.md (dr ложится на слой артефакта-владельца, см. `meta_model.md §6.1.2`). Поле `about:` — для индекса-среза, не для маршрутизации.

### 5.3. Проверь отсутствие конфликта имён

Если `<target-dir>/<slug>.md` ИЛИ объявленный компаньоном сайдкар `<target-dir>/<sidecar>` уже существует — конфликт (§6): ADDED не должен молча перезаписать чужой машинный файл. Значение `sidecar:` обязано быть локальным basename (без `/`, `..`, абсолютного пути), иначе — конфликт формы (`change-format.md §3.1`).

### 5.4. Скопируй файл(ы)

```bash
mkdir -p masterspec/<target-dir>
# имя сайдкара — из поля `sidecar:` компаньона через YAML-парсер (не sed: устойчиво к кавычкам):
sidecar=$(python3 -c "import sys,yaml; print((yaml.safe_load(open(sys.argv[1]).read().split('---',2)[1]) or {}).get('sidecar','') or '')" \
  masterspec/changes/<name>/new/<slug>.md)
# оба целевых пути свободны (проверено §5.3) — копируем пару как одну операцию:
cp masterspec/changes/<name>/new/<slug>.md masterspec/<target-dir>/<slug>.md
[ -n "$sidecar" ] && cp "masterspec/changes/<name>/new/$sidecar" "masterspec/<target-dir>/$sidecar"
```

### 5.5. Нормализуй YAML-фронтматтер

В скопированном файле:
- `status`: для артефактов слоёв → `actual` (change согласован мержем PR — applied-артефакт сразу актуален); для `adr-`/`dr-` → сохрани как было в `new/` (`proposed`/`accepted`), НЕ переписывай в `actual` (статусная модель решений — `meta_model.md §6.1.2`).
  - **Исключение — migration-класс** (шапка change.md `> **Класс**: migration`): СОХРАНИ объявленный `status:` из `new/<slug>.md`, не форси `actual`. Миграция меняет форму артефакта под новую версию мета-модели, а не его зрелость: `draft`-фабрика после миграции остаётся `draft` (сертификат версии — отдельная ось, ставится §11, см. `../masterspec/references/migration.md`).
- `updated: YYYY-MM-DD` (сегодня)
- Все остальные поля — сохрани как были в `new/<slug>.md`.

### 5.6. Удали HTML-комментарии

Если файл содержит инструкции шаблона `<!-- ... -->` — удали их (на этапе создания change должны были быть удалены, но подстрахуемся).

## 6. Конфликты

При любом из перечисленных сценариев — **остановись**, покажи пользователю через AskUserQuestion два варианта: (a) отменить apply целиком (`git checkout HEAD -- masterspec/ ':(exclude)masterspec/changes/'`), (b) пропустить операцию и продолжить.

### 6.1. MODIFIED-файл отсутствует

Слуг из §2.1 не найден в `masterspec/`:
```
Change ссылается на masterspec/01-requirements/02-functions/fn-X.md, но файл не существует.
Это рассинхрон change.md с текущим состоянием фабрики.
```

### 6.2. Раздел в diff-блоке не найден

Заголовок из `**Раздел**:` не найден в целевом файле:
```
В masterspec/.../fn-send-notification.md раздел "## Критерии приёмки" не найден.
Доступные разделы уровня ##: "Назначение", "Use-case", "Приёмочные критерии".
Возможно, раздел переименован. Отменить apply целиком или пропустить этот блок?
```

**Не пытайся угадать по синониму** — это решение человека.

### 6.3. Блок `ДО:` не найден или не уникален

Текст из `ДО:` не встречается в целевом разделе или встречается несколько раз:
```
В fn-send-notification.md раздел "Критерии приёмки" не содержит строку:
  "- AC-03. Система возвращает статус доставки в течение 30 секунд."
Отменить или пропустить блок?
```

### 6.4. `new/<slug>.md` уже существует в целевой директории

```
Файл masterspec/02-specifications/02-scenarios/scn-retry-flow.md уже существует.
Change пытается создать его заново. Отменить или пропустить?
```

### 6.5. REMOVED-файл отсутствует

```
Change помечает masterspec/.../fn-X.md как REMOVED, но файл уже отсутствует.
Пропустить (считать noop) и продолжить?
```

По умолчанию — предложи «пропустить», warning не блокирующий.

### 6.6. Грязное git-состояние

См. § 1.1 — блок, не дожидаясь запуска мержа.

## 7. Удаление REMOVED

Для каждой строки §2.3 REMOVED:

```bash
# REMOVED компаньона с сайдкаром удаляет ПАРУ: висячий сайдкар индекс и smoke-check (ходят по .md) не заметят.
target="masterspec/<путь>"
if [ "${target##*.}" = "md" ] && [ -f "$target" ]; then
  sidecar=$(python3 -c "import sys,yaml; print((yaml.safe_load(open(sys.argv[1]).read().split('---',2)[1]) or {}).get('sidecar','') or '')" "$target")
  case "$sidecar" in
    "") : ;;  # сайдкара нет — удаляем только компаньон
    */*|*'\'*|*..*) echo "CONFLICT: невалидный sidecar '$sidecar' у $target — не удаляю"; exit 1 ;;  # до мутаций
    *) rm -- "$(dirname "$target")/$sidecar" || { echo "CONFLICT: не удалить сайдкар"; exit 1; } ;;  # сначала сайдкар
  esac
fi
rm -- "$target"
```

dry-run и финальная verification (§9) показывают/проверяют ОБА пути пары; висячий сайдкар без компаньона иначе ловит form-detector (F2 orphan-проверка, `masterspec/references/patterns/verification-axes.md §O1`).

Потом — убедись, что никакой другой артефакт не ссылается на удалённый slug (Grep по `masterspec/` за исключением `changes/`). Если есть ссылки — ВТОРОЙ конфликт (rotated §6.7): change должен был зафиксировать правку этих ссылок в §2.1 как MODIFIED. Если не зафиксировал — предупреди пользователя, но удаление выполни (пользователь потом поправит через отдельный change).

### 7.1. Очистка опустевших директорий

После удаления всех файлов из REMOVED — проверь родительские директории удалённых файлов. Если после удаления директория опустела (и в ней нет поддиректорий):

```bash
rmdir masterspec/<dir-path> 2>/dev/null
```

`rmdir` без `-r` — сработает только если директория реально пуста. Если внутри остались другие файлы/поддиректории (или служебные точки) — команда молча завершится и директория сохранится.

Предупреждение: не удаляй канонические слой-директории из `masterspec/references/artifact-routing.md § 1` (например, `01-requirements/01-system/`), даже если они пусты — они остаются как «вешалки» для будущих артефактов. Удаляй только «нестандартные» директории, появившиеся вне мета-модели (как `01-requirements/07-ui/` до введения типа `ui-view`).

## 8. Обновление `00-masterspec-index.md` — полная перегенерация индекса

Этот файл — единственный shared «spine» фабрики. После всех ADDED/REMOVED он перегенерируется **полностью** процедурой перегенерации индекса (по `../masterspec/references/index-canonical.md`), а не точечными правками.

### 8.1. Почему полная перегенерация, а не diff-блоки

- Алгоритм размещения строк/нумерации разделов — в одном месте (`<Skill dir>/../masterspec/references/index-canonical.md`).
- Change.md НЕ содержит diff-блоков по `00-masterspec-index.md`. Попытка их добавить должна отклоняться на ревью (см. `<Skill dir>/../masterspec/references/change-format.md`).
- Сортировка, нумерация, каноничные заголовки секций — одинаковы между фабриками и между прогонами apply-change. Нет merge-конфликтов по индексу.

### 8.2. Алгоритм (что делает apply-change в §8)

1. Прочитай `<Skill dir>/../masterspec/references/index-canonical.md` (spec + алгоритм reindex).
2. Выполни полную перегенерацию индекса по `<Skill dir>/../masterspec/references/index-canonical.md` на директории `masterspec/`:
   - прочитай текущий `masterspec/00-masterspec-index.md`, сохрани §1 «Паспорт» и §7 «Белые пятна и открытые вопросы»;
   - сохрани словарь `path → comment_text` из строк артефактов (для сохранения комментариев);
   - просканируй `00-glossary.md`, `01-requirements/**/*.md`, `02-specifications/**/*.md`, `03-codemap/**/*.md`, `04-decisions/**/*.md`;
   - распредели артефакты по каноническим секциям (§1.1–§1.4 из `index-canonical.md`);
   - сгенерируй итоговый markdown по шаблону §4 из `index-canonical.md`;
   - перезапиши `masterspec/00-masterspec-index.md` (атомарно), проставь `updated:` = сегодня.
3. Собери отчёт: файлы без фронтматтера, дубли `slug:`, неизвестные типы. Отчёт включи в вывод apply-change (не блокирует успех, но пользователь должен увидеть).

Ручных правок в индексе на этом шаге нет. Если reindex упал (например, не найден `index-canonical.md`) — блок с сообщением, `git checkout HEAD -- masterspec/ ':(exclude)masterspec/changes/'`, апдейт статуса НЕ делается.

## 9. Финальная валидация

Двухэтапная: сначала **smoke-check** (структурные инварианты фабрики, §9.1), затем **verification** (каждая строка §2.1/§2.2/§2.3 change.md дошла до файлов, §9.2). Итоговый **вердикт** (§9.3) определяет, можно ли ставить статус «Реализовано» в §10.

### 9.1. Smoke-check

Структурные инварианты фабрики — быстрые проверки:

1. `find masterspec/ -name "*.md" -type f | wc -l` — число файлов соответствует ожидаемому (добавлены N, удалены M относительно до-apply состояния).
2. Каждый ADDED-файл присутствует в `00-masterspec-index.md`.
3. Каждый REMOVED-файл отсутствует и в `00-masterspec-index.md`, и в дереве; парный сайдкар удалённого компаньона (по плану dry-run) — тоже.
4. Grep по `masterspec/` на предмет обратных ссылок — нет ссылок из требований на `cmp-`/`scn-`/`api-`/`data-`/`cmap-`/`trace-`/`dmap-`.

Провалилась хотя бы одна проверка — `git checkout HEAD -- masterspec/ ':(exclude)masterspec/changes/'`, сообщи пользователю, расследуйте причину вместе. В §9.2 не переходим.

### 9.2. Verification — применённость каждой строки §2

Smoke-check покажет, что дерево файлов в порядке, но не покажет, что конкретно внутри MODIFIED-файла оказалась та правка, которую change описывал. Пример негативного сценария: `modify-bullet` не нашёл `ДО:`, в §6 пользователь выбрал «пропустить», файл не изменён — smoke-check это не заметит, статус уйдёт в «Реализовано».

Для каждой строки §2.1/§2.2/§2.3 на выходе имеется один из трёх статусов:

- **`confirmed`** — факт применения подтверждён (см. алгоритм ниже).
- **`skipped_by_user`** — в §6 на конфликте пользователь выбрал «пропустить». Скилл хранит список таких строк **в памяти сессии** (не в файлах change.md) и исключает их из проверки. В отчёте — перечислены отдельной группой. (В `context=lean` это `result: skipped_by_user` в `apply/<slug>.md` — файловый контракт, не память; см. `../masterspec/references/patterns/context-isolation.md`.)
- **`unconfirmed`** — применение не подтверждено и не было пропущено. Блокирует §10.

#### 9.2.1. MODIFIED — `modify-bullet`

1. Взять `ПОСЛЕ:` из diff-блока, отфильтровать полностью пустые строки.
2. Обозначить `first` — первую непустую строку, `last` — последнюю (если строк 0 — особый случай 9.2.5; если строк 1 — `first == last`, проверка вырождается в одну).
3. Нормализовать (trim leading/trailing whitespace) и Read целевого файла.
4. Проверить: `first` и `last` встречаются в файле хотя бы по одному разу.
5. Обе нашлись → `confirmed`. Иначе → `unconfirmed`.

Выбран компромиссный порог «первая + последняя» (а не все строки или любая одна): устойчив к нормализации whitespace посередине длинного блока и одновременно ловит ситуацию, когда правка вообще не дошла до файла.

#### 9.2.2. MODIFIED — `replace-section`

1. Найти в целевом файле строку заголовка раздела из поля `**Раздел**:` (по точному тексту).
2. Определить границы раздела — от найденного заголовка до следующего заголовка того же уровня или до конца файла (см. §4.2).
3. Взять первую непустую строку `ПОСЛЕ:` — она должна встречаться **внутри границ раздела**.
4. Встретилась → `confirmed`. Иначе → `unconfirmed`.

#### 9.2.3. MODIFIED — `add-subsection`

1. Найти раздел из `**Раздел**:`, определить границы (см. §4.3).
2. Первая непустая строка `ПОСЛЕ:` (обычно `### <slug>` или `#### <slug>`) должна встречаться в границах раздела.
3. Встретилась → `confirmed`. Иначе → `unconfirmed`.

#### 9.2.4. Особый случай — `ПОСЛЕ:` пуст

Пустой `ПОСЛЕ:` для `modify-bullet` означает «удалить строку из ДО». Проверка: текст `ДО:` больше **не встречается** в целевом файле → `confirmed`; встречается → `unconfirmed`.

Для `replace-section` с пустым `ПОСЛЕ:` (очистка раздела) — раздел найден, между его заголовком и следующим заголовком того же уровня нет непустых строк → `confirmed`.

Для `add-subsection` с пустым `ПОСЛЕ:` — невалидный блок (нечего вставлять), на этапе валидации change.md (`evolve`, §5 change-format) это должно было отловиться; в verification — `unconfirmed` с пометкой «невалидный блок».

#### 9.2.5. ADDED

Для каждой строки §2.2:

1. Построить целевой путь `masterspec/<target-dir>/<slug>.md` по `type:` фронтматтера `new/<slug>.md` (см. `artifact-routing.md`).
2. Проверить: файл существует.
3. Read первых ~15 строк — проверить YAML-фронтматтер:
   - `slug:` = ожидаемый;
   - `type:` = ожидаемый;
   - `status:` присутствует (артефакты слоёв — `actual`, уже применены; `adr-` — `proposed`/`accepted`, `dr-` — `accepted`);
   - `updated:` = сегодня.
4. Grep по всему файлу на `<!--` — комментарии шаблона должны быть удалены (§5.6).
5. Если артефакт машинный (api/data/scn/alg с sidecar): объявленный `<target-dir>/<sidecar>` существует И парсится по своему `sidecar_format` (yaml/json — `yaml.safe_load`/`json.loads`; xml для bpmn/dmn — well-formed; текстовые нотации — непустота без ```). Битый/отсутствующий сайдкар → `unconfirmed`. Детерминированно это же проверяет детектор формы `<Skill dir>/../masterspec/scripts/check-operational-envelope.py --layer spec` (F2).

Все пункты выполнены → `confirmed`. Иначе → `unconfirmed` с указанием первого непрошедшего пункта.

#### 9.2.6. REMOVED

Для каждой строки §2.3: файла по `masterspec/<путь>` **не** существует → `confirmed`; существует → `unconfirmed`. Если удалённый компаньон (по плану dry-run, §3) объявлял `sidecar:` — парный сайдкар тоже обязан отсутствовать, иначе `unconfirmed` (висячий сайдкар индекс не отражает — ходит по `.md`).

### 9.3. Вердикт и реакция

Собрать агрегат по §9.2:

- `C` = число `confirmed`.
- `S` = число `skipped_by_user`.
- `U` = число `unconfirmed`.

Сформировать отчёт в виде таблицы по трём группам (MODIFIED / ADDED / REMOVED) и блока «Пропущено пользователем». Пример:

```
## Verification

### MODIFIED
| Slug | Тип | Результат |
|------|-----|-----------|
| fn-X | modify-bullet | ✓ confirmed |
| cmp-Y | add-subsection | ✓ confirmed |
| scn-Z | replace-section | ⚠ unconfirmed — первая строка ПОСЛЕ не встречается в разделе «## Основной поток» |

### ADDED
| Slug | Результат |
|------|-----------|
| fn-new | ✓ confirmed |

### REMOVED
(нет)

### Пропущено пользователем
| Slug | Причина (из §6) |
|------|-----------------|
| fn-A | ДО: не найден в разделе |

### Итог
confirmed: 2 · skipped_by_user: 1 · unconfirmed: 1
```

Реакция:

- **`U == 0`** → §10, ставим статус `Реализовано`.
- **`U > 0`** → блок перехода в §10. Через AskUserQuestion предложить три опции:

  | Опция | Действие |
  |-------|----------|
  | **rollback** | `git checkout HEAD -- masterspec/ ':(exclude)masterspec/changes/'` — откатить все правки. Статус change.md не меняется (остаётся `Согласовано` / `В реализации`). Скилл завершается с ненулевым кодом. |
  | **override** | Пользователь просмотрел unconfirmed-строки глазами и подтверждает, что правки на самом деле применены (ложноотрицательный verdict verification). Переходим в §10 с пометкой в выводе «verification overridden manually». |
  | **leave** | Не откатывать. Статус change.md → `В реализации` (или остаётся, если уже такой). Change **не** уходит в архив. Пользователь разбирается позже. |

Опция `override` фиксируется в выводе скилла, чтобы был audit-след. В сам change.md дополнительные пометки не пишутся.

## 10. Обновление статуса change.md

Только после §9 с вердиктом `U == 0` ИЛИ явного выбора пользователем `override` в §9.3:

```
> **Статус**: Реализовано
```

Замени строку в шапке `masterspec/changes/<name>/change.md`. Меняй **только** эту строку. YAML-фронтматтер change.md (если есть) — синхронизируй поле `status:` с текстовой шапкой.

Если §9.3 вернул `rollback` — статус change.md не трогаем, скилл выходит.
Если §9.3 вернул `leave` — статус в change.md меняем на `В реализации` (не `Реализовано`), скилл выходит без перехода в архив.

## 11. Сертификация версии мета-модели

После §9 с вердиктом `confirmed`/`override` (правки применены) — единственный шаг, который ставит/обновляет штамп `meta_model_version` фабрики. Версия = ВЫХОД-сертификат (`../masterspec/references/meta-model-version.md`).

### 11.1. Когда запускается
- Всегда после успешного apply (регресс-гейт: любой change к 3.0-фабрике пере-подтверждает, что она осталась 3.0).
- Отдельно, без применения change, — когда apply-change вызван в режиме сертификации (миграция `draft`-фабрики, дозаполнение которой уже в дереве; `../masterspec/references/migration.md` шаг 5). Тогда §§5–10 пропущены, выполняется только §11.

### 11.2. Прогон детекторов
Прогони детекторы инвариантов текущей версии на применимых scope фабрики:
```bash
python3 <Skill dir>/../masterspec/scripts/check-operational-envelope.py masterspec --layer req
# + --layer spec, если есть 02-specifications/ ; + --layer code, если есть 03-codemap/
python3 <Skill dir>/../masterspec/scripts/check-verify-report.py <verify-report>   # если verify-report присутствует
```
Собери BLOCKER-строки со всех прогонов = остаточная дельта до контракта.

### 11.3. Штамп или отчёт
- **Все применимые scope зелёные** (каждый детектор exit 0, ноль BLOCKER) → полное сквозное соответствие. Впиши/обнови во фронтматтере Паспорта `01-requirements/01-system/as-<slug>.md`: `meta_model_version: <текущая версия ядра>` (сейчас `3.0`, без квалификатора) и `updated: <сегодня>`. Идемпотентно: штамп уже равен → no-op.
- **Часть scope зелёная, часть — нет** → scope-квалифицированный штамп: `meta_model_version: <версия>-<scope>` (например `3.0-req`, если req чист, а spec/code — нет). Честно фиксирует, докуда дотянута версия (`../masterspec/references/meta-model-version.md`). В отчёте — какие scope прошли, какие в дельте.
- **Все применимые scope красные** → штамп НЕ ставить и существующий НЕ понижать. В отчёте: «фабрика остаётся до-<версия>, дельта: N BLOCKER» + список. Не откат apply (правки применены) — фабрика ещё не дотягивает.
- Правится ТОЛЬКО поле `meta_model_version` (+ `updated`) Паспорта; остальное не трогай.

### 11.4. В отчёт apply-change
```
### Сертификация версии
meta_model_version: 3.0 (проставлен) | до-3.0 (дельта: N BLOCKER — <первые строки>)
детекторы: check-oe req=OK[ spec=OK][ code=OK][ ; check-verify-report=OK]
```

## 12. Что скилл НЕ делает

- Не модифицирует файлы в `masterspec/changes/<name>/` (кроме строки статуса в change.md). Все исходники change остаются для аудита.
- Не коммитит в git — коммит пользователь делает отдельно, видя diff.
- Не вызывает `masterspec-archive-change` — архивация отдельный шаг.
- Не создаёт ветки, не пушит.
