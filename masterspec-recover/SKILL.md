# masterspec-recover — восстановить описание

Восстанавливаю описание из имеющегося материала. Не домысливаю: что не подтверждено источником — помечаю как открытый вопрос. Мета-модель и дисциплина слоёв — в kernel (`../masterspec/`). Раскладка и инициализация фабрики — как в `derive` (`../masterspec-derive/SKILL.md §0`, мета-модель §3). **Каждый артефакт кладётся строго в свой канонический подкаталог по `../masterspec/references/artifact-routing.md` (`type:` → путь; `api` — по `scope`, `fn` — по `block`), НЕ валом в корень слоя (`02-specifications/cmp-…` вместо `02-specifications/01-components/cmp-…` — дефект).** Это инвариант ПЛАНИРОВАНИЯ, не постобработки: в lean оркестратор передаёт каждому субагенту ПОЛНЫЙ `target_path` из routing (слой + подкаталог), а не папку-слой; субагент пишет ровно туда. Плоскую раскладку в корень слоя detected-и-блокирует детерминированный контроль перед перегенерацией индекса (`../masterspec/scripts/check-layout.py`, см. `verify`).

> **Lean для docs/both.** `recover/plan.md` несёт типизированные фазы, и оркестратор не читает содержимое сам ни в одной из них: `docs-research` (explore source=docs) → `docs-materialize` (gen-субагенты пишут артефакты из агрегата) → `machine-import` (субагент копирует контракты, собирает тонкие компаньоны) → `test-link` (стыковка ТК, несостыкованные → `recover/_gaps.md`) → `reconcile` (только для `both`: сверка агрегатов → `recover/_reconciliation.yaml`). Оркестратор держит план и счётчики, содержимое — субагенты.
>
> **`context=full` / `context=lean` (дефолт)** — при восстановлении большой фабрики в `lean` (файловый контракт — `../masterspec/references/patterns/context-isolation.md §Lean в других скиллах`): субагент-планировщик пишет `recover/plan.md`; проход 1 — `explore` по коду (`.research/`) + `gen`-субагенты пишут скелет спецификаций/требований; проход 2 — codemap-субагенты, каждую висячую ссылку → `recover/_dangling.md` (поля `from/ref/evidence/resolution/handled_by`); перелинковка-субагент гонит ссылки `pending → resolved|created` по этому файлу; «Белые пятна» субагент собирает в `recover/_gaps.md` → §7 индекса. recover держит только `recover/plan.md` и сводки (`_dangling`/`_gaps`-счётчики), содержимое не читает; индекс — субагент. Для ограниченного контекста.

## Provenance (обязательно для всех восстановленных артефактов)
Каждый артефакт несёт во фронтматтере поле `provenance:` с источником: `provenance: docs:<файл>#<якорь>` или `provenance: code:<path>[:line]`. **При `source=both` источников может быть несколько — тогда `provenance:` СПИСОК** (`provenance: [docs:…, code:…]`), а не повторяющийся ключ: повторный YAML-ключ тихо теряет одно из значений. Артефакт, восстановленный из обоих источников, обязан показывать оба. В разделе «Открытые вопросы» — «восстановлено из <X>, требует подтверждения владельцем». Это делает восстановление повторяемым и отличимым от знания «из головы». ОТСУТСТВИЕ источника тоже фиксируется — типизированным белым пятном (шаг 5a: `negative-provenance` «искал, не нашёл» ≠ «не смотрел»), а не молчанием.

## source=docs — из документов и сырья

Разбор сырья — НЕ чтение файлов в своё окно. Массив (выгрузка Confluence, экспорт тикетов, таблицы ТК) переполняет контекст и деградирует молча. Поэтому разбирает `explore source=docs`, а recover работает по его агрегату.

1. **Research.** Запусти `explore source=docs docs=<пути, дефолт 00-source-data/>` (`../masterspec-explore/SKILL.md § Шаги — source=docs`). Он даёт: `docs-inventory.yaml` (опись — что за материал, объём, куда относится, что machine-ready, что дубль) и `_aggregate.yaml` (кандидаты по слоям, каждый с провенансом `<path>#<якорь>`).

2. **Создавай артефакты сверху вниз** (требования → спецификации) из агрегата — каждый со `status: draft` и `provenance: docs:<файл>#<якорь>`. Якорь обязателен: имя файла не делает находку проверяемой.

3. **Machine-import: готовый контракт берётся КАК ЕСТЬ.** Материал из `machine_artifacts[]` (OpenAPI/AsyncAPI/JSON Schema/BPMN) — это уже сайдкар, а не сырьё для пересказа. Пересказывать машинный контракт прозой запрещено: потеря точности + вторая нормативная редакция (O0).

   Контракт импорта:
   - файл **копируется** в каталог артефакта под каноническим именем `<slug>.<ext>` (имя объявляется во фронтматтере компаньона полем `sidecar:` — lifecycle читает поле, а не гадает);
   - многофайловый контракт (OpenAPI с внешними `$ref`) — либо тянется ЦЕЛИКОМ (все файлы замыкания зависимостей, ссылки остаются рабочими), либо детерминированно раздереференсивается в один файл. Половина контракта = битые ссылки и осиротевшие файлы, которые verify сочтёт мусором;
   - компаньон `<slug>.md` — **тонкий, но не пустой**: фронтматтер (`type`, `slug`, `status: draft`, `sidecar_format`, `sidecar`, `contract_origin: imported`, `contract_source: <откуда>`, `provenance: docs:<…>`), назначение, связи вверх, и обязательный семантический профиль по типу:
     - `api` — маппинги полей («поле ← источник»; в OpenAPI/AsyncAPI не выражаются, это O0-собственность компаньона), ограничения живой эксплуатации (SLA, идемпотентность, ретраи);
     - `data` — состояния и переходы, если сущность их имеет (JSON Schema их не выражает);
     - `bpmn` — маппинг lane/pool/task → `cmp`/`cap`.
     Пусто в этих разделах — не «нечего писать», а `MIGRATE-TODO`/открытый вопрос: без них артефакт завалит O1/O6/O7.
   - `contract_origin: imported` значит: контрактом владеет ВНЕШНЯЯ сторона. `gen`/`evolve`/`migrate` его не правят; изменение — через `evolve entry=ext` (`patterns/sidecar-formats.md §Происхождение контракта`).

4. **Готовые тест-кейсы — только со стыковкой.** Кандидат из `test_cases[]` заводится как `tc-acc` (слой требований, под критерий приёмки функции) или `tc-int` (слой спецификаций, под сценарий) ТОЛЬКО если `traces_to.confidence` = `exact`/`probable` И целевой артефакт существует. `probable` — завести и вынести в «Открытые вопросы» на подтверждение владельцем. `confidence: none` → НЕ заводить: висячий `tc-` не проверяем на полноту и молча разъедется со спекой; такие — в §7 поимённо.

   **`tc-flt` из сырья НЕ заводится.** Каталог отказов — один на сценарий, детерминированная матрица «точка `scn → api` × модус → результат»; он выводится из спеки, а не из чужих тестов. `fault_observations[]` из research — подсказка при его построении, не артефакт.

   `criticality` обязательна для всех `tc-*`. Не выводится из материала — открытый вопрос владельцу, не догадка.

5. **Всё неоднозначное — в «Открытые вопросы».** Не домысливай. Собери глоссарий.

6. Перестрой `00-masterspec-index.md` полной перегенерацией (`../masterspec/references/index-canonical.md`). Заполни §7 «Белые пятна».

## source=code — из кода (ДВА ПРОХОДА)
Код-слой (`03-codemap/`) ссылается ВВЕРХ на `cmp/scn/data`. Поэтому codemap строится не первым, а после того, как есть целевые slug'и спецификаций — иначе ссылки снизу некуда вести.

**Проход 1 — скелет сверху (даёт целевые slug'и):**
1. Анализ структуры через `explore` (target=factory-spec, `roots=`): языки, фреймворки, модули.
2. По сигнатурам, контроллерам и трассам набросай СКЕЛЕТ спецификаций (`cmp-` с возможностями, `scn-`, `api-`, `data-`) и требований (`fn-`, `cdm-`). Все — `status: draft`, `provenance: code:<path>`. Содержание чернове; цель прохода — устойчивые slug'и. **Каждый артефакт — сразу в свой канонический подкаталог по `../masterspec/references/artifact-routing.md` (`cmp-`→`02-specifications/01-components/`, `scn-`→`…/02-scenarios/`, `api-`→`…/04-apis/{internal,external}/`, `fn-`→`01-requirements/02-functions/`, …), НЕ валом в корень слоя. В lean целевой путь субагенту даёт оркестратор ПОЛНЫМ (слой+подкаталог).**
2a. **Машинные сайдкары (api/data — обязательно; scn/alg — только когда нотация/форма несёт машинный файл: `bpmn` обязателен, `sequence`/`workflow` при потребности, `yaml-graph`/`pull-rules`/`procedural` — нет, см. `scenario-notation-registry.md`).** Для каждого набросанного `api-`/`data-` создай рядом машинный сайдкар по `../masterspec/references/patterns/sidecar-formats.md`: природу контракта видно из кода (REST-роуты → OpenAPI; поллинг/вебхуки/брокер → AsyncAPI; сущность/конфиг → JSON Schema). Компаньон `.md` несёт во фронтматтере `sidecar_format:` и `sidecar:`; машинный файл валидно парсится, без markdown-ограждений, провенанс наследует от компаньона. Формат не ложится в дефолт → выбери корректный стандарт (WSDL/protobuf/GraphQL SDL/…) или пометь `negative-provenance`-пятном — нотацию НЕ выдумывай.

**Проход 2 — код-слой с перелинковкой:**
3. Собери codemap, `generated: true`, каждый артефакт ссылается ВВЕРХ на slug из прохода 1: `cmap-*` (компонент → код `file:line`) → `cmp-`; `trace-*` (цепочка вызовов сценария) → `scn-`; `dmap-*` (сущности → таблицы/ORM) → `data-`; плюс `repo-map`.
3.5. **Pre-flight ссылок (детерминированно).** Перед перелинковкой — сверка множеств: собрать все `-> slug` из codemap и сверить со множеством slug прохода 1 (скриптом или ручным set-diff). Висячие ссылки выявляются списком разом. Язык-нейтрально. **full:** рекомендуется — pre-flight даёт список дешевле, чем поэлементный обход; если его не делали, висячие всё равно выявит обход на шаге 4 (дороже). **lean:** список висячих ОБЯЗАТЕЛЬНО материализуется в `recover/_dangling.md` (сверка инструментальная или ручная — но файл нужен; шаг 4 идёт по нему; см. `../masterspec/references/patterns/context-isolation.md §Lean → recover`).
4. **Перелинковка (обязательный шаг):** пройди по всем codemap-ссылкам (в lean — по списку из шага 3.5) — каждая должна разрешаться в существующий slug прохода 1. Если ссылка ведёт в пустоту — доздай недостающий spec-артефакт (черновой) и сошлись на него. Ни одной висячей ссылки.
5. Что НЕ видно из кода (SLA, приоритеты, бизнес-правила, мотивация) — не выдумывай, заводи вопросы бизнес-владельцу.
5a. **Типизированные белые пятна (§7) — помечай категорией, не сваливай в кучу:**
   - `missing-business` — не выводится из кода (SLA, мотивация, правило) → вопрос владельцу;
   - `negative-provenance` — ИСКАЛ в источнике и не нашёл (отличать от «не смотрел»/«вне области»): запись «искал в `<X>`, не найдено» — это факт восстановления, а не пробел внимания; делает «пусто» проверяемым;
   - `contract-gap` — реализация ЕСТЬ (endpoint/обработчик в коде), а контракта `api-` в спеке НЕТ → кандидат на доздание контракта.
   Эти категории живут в recover (записи в §7 «Белые пятна» / `recover/_gaps.md`), НЕ протекают в derive/spec как общие поля артефактов (single-source: provenance и его разновидности — recover-специфика).
6. Перестрой индекс полной перегенерацией. Заполни §7 «Белые пятна».

## source=both — документы И код (со сверкой)

Документы говорят, как ЗАДУМАНО. Код говорит, как СДЕЛАНО. Восстанавливать по обоим и не сверить их — значит выбросить самое ценное, что даёт двойной источник.

1. **Два research-прогона:** `explore source=docs` и `explore source=code`. Агрегаты кладутся раздельно (`.research/docs/`, `.research/code/`) — смешивать нельзя: провенанс и статус находок разной природы (`declared: true` против `code:<path>`).

2. **Скелет строится по КОДУ** (`source=code`, оба прохода): код даёт устойчивые slug'и и трассы, документы — нет.

3. **Обогащение из документов:** то, чего в коде не видно (SLA, мотивация, бизнес-правила, приоритеты), берётся из docs-агрегата и ложится на существующие slug'и с `provenance: docs:<…>`. Артефакт с двумя источниками несёт оба.

4. **СВЕРКА (обязательный шаг).** Сопоставляются коллекции docs-агрегата и code-агрегата. Без явных ключей сверка нарожает ложных расхождений (в Confluence функция названа иначе, чем в коде), поэтому:

   **Ключи сопоставления по типам** (identity):
   - функции — нормализованное имя + триггер + актор; API — метод+путь (REST) / канал+операция (event); правила — нормализованный текст утверждения; сущности — нормализованное имя; NFR — категория+метрика.
   - **Нормализация** (обе стороны, до сравнения): kebab/camel/snake → единый регистр и разделитель, стоп-слова сервис/service/api/handler отбрасываются, транслитерация ru↔en для имён сущностей.
   - **Alias-map** — человек может задать соответствия, которые машинно не выводятся (`recover/_aliases.yaml`: `docs-имя → code-имя`). Один раз заданный alias переиспользуется.

   **Каждой паре — `match_confidence`** (`exact` — ключи совпали; `probable` — совпали после нормализации/по alias; `none`). Неоднозначные сопоставления (одно docs-имя → несколько code-кандидатов и наоборот) идут в `ambiguous_matches[]` и **не классифицируются как расхождение до разрешения человеком**: «не смог сопоставить» ≠ «не реализовано». Это ключевое: иначе переименование модуля выглядит как дыра в реализации.

   **Классы расхождений** (в §7 «Белые пятна», каждый своей категорией):
   - `declared-not-implemented` — заявлено в документах, в коде не найдено. Вопрос владельцу: не сделано, выпилено, или живёт вне исследованных `roots`? Молча выбрасывать заявленное нельзя.
   - `implemented-not-declared` — реализовано в коде, ни в одном документе не описано.
   - `spec-drift` — есть в обоих, но расходятся по существу: другой порог/таймаут, другой набор статусов, другая ветка отказа. recover НЕ выбирает, кто прав: фиксирует ОБА значения с провенансами и отдаёт владельцу. Правда о желаемом — у владельца, не у скилла.

   (Не путать с `contract-gap` из §5a: тот — про отсутствие `api-`артефакта в спеке при наличии реализации, внутри code-ветки.)

   **Результат сверки материализуется**, а не остаётся прозой: `recover/_reconciliation.yaml` — `matched[]`, `ambiguous_matches[]`, три класса расхождений, счётчики. Свободный текст §7 оси O1–O7 не читают, и `codegen_ready` может позеленеть при незакрытом `spec-drift`. Неразрешённые расхождения — **блокер** для `accept` слоя и для `codegen_ready`.

   **Импортированный контракт против кода.** Если docs дал machine-ready контракт на тот же api, что набросал code-проход, — НЕ создавать две пары. Сначала структурный diff (операции/схемы): совпало — оставить импортированный (`contract_origin: imported`, он нормативен), кодовый набросок выбросить; разошлось — это `spec-drift` на уровне контракта: импортированный остаётся нормативным, расхождение с реализацией фиксируется владельцу. Две нормативные редакции одного контракта = hard-fail O0.

5. Дальше — как в `source=code`: перелинковка, типизированные белые пятна, перегенерация индекса.

> Сверка — не отчёт «на посмотреть». Расхождение заявленного и фактического — главный улов восстановления по двум источникам: именно в нём живёт то, что команда считает работающим, а оно не работает.

## Выход
- черновики артефактов (`status: draft`, `provenance: …`);
- `00-masterspec-index.md` (перегенерирован) с §7 «Белые пятна»;
- список открытых вопросов для уточнения.

> **Без субагентов (`Task`):** если механизма нет — выполняй проходы и артефакты последовательно; результат тот же, дольше.

После восстановления — вычитка `verify` и доведение через `derive`/`evolve`.
