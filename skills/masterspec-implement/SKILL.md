# masterspec-implement — реализация change по tasks.md

Реализовать задачи из `tasks.md` строго по плану. Если есть `design.md` — опирайся на него; для простого CR `design.md` может отсутствовать.

**Input**: Имя change (например `add-retry-for-failed-delivery`). Навык автоматически добавит префикс `masterspec/changes/<name>/`.

## Bundle-пути

`references/X` — относительно директории этого скилла. Если harness даёт `Skill directory: <abs>` — используй. Иначе `Glob("**/masterspec-skills/skills/masterspec-implement/references/<file>")`. Пусто → спроси пользователя.

## Справочники (читай по необходимости)

- `references/runbook.md` — статусы change.md, парсинг tasks.md, цикл выполнения, форматы вывода (During / Completion / Pause).
- `references/verification.md` — команды сборки/тестов/линтера по стекам, интерпретация результатов.
- `references/edge-cases.md` — отсутствующий design.md, конфликт tasks.md ↔ change.md, частично выполненные задачи, прерывание.

**AskUserQuestion fallback**: если инструмент недоступен — задай вопрос обычным текстом и дождись ответа.

## Шаги

### 1. Выбери change

Если имя не указано:
```bash
ls masterspec/changes/ 2>/dev/null | grep -v "^archive$"
```
Автовыбор при одном активном; AskUserQuestion если несколько. Объяви: `Реализую change: <name>`.

### 2. Проверь файлы

Путь `masterspec/changes/<name>/`. `tasks.md` — **обязателен** (без него выход, см. `edge-cases.md`). `design.md` — **опционален**.

### 3. Проверь статус change.md

Прочитай `> **Статус**:` в шапке и действуй по матрице из `runbook.md`:
- `Согласовано` → авто-переход в `В реализации` (меняй ТОЛЬКО строку статуса).
- `В реализации` → продолжай без правок.
- `Реализовано` / `Архивировано` / `На согласовании` → выход с подсказкой.

### 4. Покажи прогресс

Посчитай выполненные и оставшиеся задачи. Формат: `Прогресс: N/M задач выполнено`.

### 5. Прочитай design.md (если есть)

Если существует — изучи для контекста. Если нет — используй `change.md` и пропусти шаг. Детали — `edge-cases.md`.

### 6. Найди первую невыполненную задачу

Область поиска — **только** H2-секции, чей заголовок матчится regex `^##\s+\d+\.\s` (двойная решётка, пробел, число, точка, пробел).

Пронумерованные (ищи `- [ ]` внутри):
- `## 1. Подготовка`
- `## 2. Реализация`
- `## N. Финализация`

Без номера (ПРОПУСТИ — чекбоксы тут не трогай):
- `## Чеклист верификации`
- `## Статус выполнения`
- `## Лог изменений`
- `## Замечания ревью`

Полные правила парсинга — в `runbook.md`.

### 7. Выполни задачу (цикл)

Для каждой невыполненной: покажи `Выполняю X.Y: ...`, реализуй строго по описанию, замени `- [ ]` → `- [x]` сразу после, обнови прогресс, переходи к следующей. Пауза на непонятной задаче / ошибке / архитектурном вопросе без design.md / прерывании. Формат пауз и деталей цикла — в `runbook.md`.

### 8. Верификация перед завершением (ОБЯЗАТЕЛЬНО)

Запускается **после** того, как все `- [ ]` в рабочих секциях отмечены `- [x]`, и **до** любого сообщения о завершении.

- Определи команды по признакам проекта — полная таблица в `verification.md`. Если признаков нет — спроси через AskUserQuestion.
- Прогони **минимум**: сборку (если применима), затронутые тесты, линтер (если настроен).
- Все зелёные → шаг 9. Красное → `Output On Pause`. Верификация технически невозможна → явно зафиксируй «Автоматическая верификация не выполнена — <причина>». Не маскируй под «всё ок».

### 9. Завершение

Покажи итоговый прогресс. Если все задачи выполнены **И** верификация прошла — поздравь, процитируй итог (команды + результат), подскажи next step: `masterspec-apply-change` для вливания в артефакты фабрики. Иначе — объясни причину и спроси что делать. Формат — `runbook.md`.

## Guardrails

- ПРОДОЛЖАЙ выполнять задачи, пока не завершены или не заблокированы.
- ВСЕГДА отмечай задачу сразу после выполнения.
- В **tasks.md** — трогай ТОЛЬКО чекбоксы `- [ ]` / `- [x]` в пронумерованных секциях. НЕ трогай чеклист верификации, статус tasks.md, лог изменений.
- В **change.md** — меняй ТОЛЬКО строку `> **Статус**:` (шаг 3). Остальное не трогай.
- **Артефакты фабрики (`masterspec/01-*/02-*/03-*/04-*`) НЕ ТРОГАЙ.** Реализация происходит в коде проекта. Вливание в артефакты фабрики — это отдельный шаг `masterspec-apply-change`.
- ВСЕГДА запускай верификацию (шаг 8) перед объявлением завершения. НЕ пропускай сборку/тесты, даже если все чекбоксы зелёные.
- НИКОГДА не докладывай «реализация завершена», пока верификация не прошла или не зафиксирована как технически невозможная — явно, в отчёте.
- ЕСЛИ задача непонятна — спроси перед выполнением.
- ЕСЛИ реализация выявила ошибку в `design.md` **и файл существует** — предложи исправить; если `design.md` отсутствует — предложи запустить `masterspec-impl-plan`.
- ПАУЗА на ошибках и блокерах — не угадывай.
- КРЕАТИВ ЗАПРЕЩЁН — работай строго по плану задачи.

## Fluid Workflow

- Можно вызывать, когда есть частично выполненные задачи — продолжит с места остановки.
- Необязательно все артефакты должны быть готовы (`design.md` опционален).
