# Верификация: сборка / тесты / линтер

Расширение шага 8 основного SKILL.md. Запускается **после** того, как все `- [ ]` в рабочих секциях отмечены `- [x]`, и **до** любого сообщения пользователю о завершении.

## 1. Определение команд по признакам

Поиск сверху вниз, первый сработавший признак выигрывает:

| Признак в корне | Команды |
|---|---|
| `Makefile` | `make build` / `make test` (или цели, указанные в Makefile) |
| `pnpm-lock.yaml` | `pnpm install --frozen-lockfile && pnpm build && pnpm test && pnpm lint` |
| `yarn.lock` | `yarn install --frozen-lockfile && yarn build && yarn test && yarn lint` |
| `package-lock.json` | `npm ci && npm run build && npm test && npm run lint` |
| `package.json` без lock-файла | `npm install && npm run build && npm test && npm run lint` (если скрипты определены) |
| `pom.xml` | `mvn -q verify` |
| `build.gradle` / `build.gradle.kts` | `./gradlew build` |
| `go.mod` | `go build ./...`, `go test ./...`, `go vet ./...` |
| `pyproject.toml` / `setup.py` / `requirements*.txt` | `pytest`, `ruff`/`flake8`, `mypy` (если настроены) |
| `Cargo.toml` | `cargo build`, `cargo test`, `cargo clippy` |
| `CMakeLists.txt` / `Makefile.am` | Соответствующая цель сборки |

CI-файлы (`.github/workflows/*`, `.gitlab-ci.yml`, `Jenkinsfile`) — **подсказка**, какие команды считать «обязательными» в этом проекте.

Если в `package.json` скрипт `build` / `test` / `lint` отсутствует — просто пропусти его, не падай.

**Если признаков нет или они противоречивы** — спроси пользователя через **AskUserQuestion tool**: какие команды запустить для верификации. Не угадывай.

## 2. Минимальный набор для прогона

- **Сборка / компиляция** (если применима к стеку).
- **Автотесты**, затронутые изменением (или полный прогон, если выборочный запуск не настроен).
- **Линтер / статический анализ** (если настроены в проекте).

## 3. Интерпретация результатов

| Исход | Действие |
|---|---|
| Все команды зелёные | Переходи к шагу 9 (`Output On Completion`). Процитируй какие команды запускались и результат. |
| Есть ошибки | **Не объявляй change завершённым.** Перейди в `Output On Pause`: опиши, какая команда упала, покажи ключевой фрагмент вывода, спроси что делать (исправить сейчас, отложить, передать пользователю). |
| Верификация невозможна технически | Нет окружения, недоступны внешние зависимости и т. п. — зафиксируй **явно в отчёте**: «Автоматическая верификация не выполнена — причина <...>». Не маскируй под «всё ок». |

## 4. Rollback при падении

- Если сборка/тесты упали из-за реализации одной конкретной задачи — откати именно этот диффф (или укажи пользователю, какие файлы править). Не откатывай весь change.
- Если задача в tasks.md уже отмечена `[x]`, но её реализация оказалась неработоспособной — **верни `[ ]`** и объясни в паузе: задача вернулась в pending из-за падения верификации.
- Если падение — следствие внешнего окружения (недоступен сервис, сломан CI-раннер) — не трогай код и tasks.md, зафиксируй «верификация невозможна».

## 5. Антипаттерны

- «Все чекбоксы зелёные — значит готово» — **нет**, без прогона сборки/тестов не готово.
- «Тесты не запускал, но вроде всё должно работать» — **нет**, либо прогнал, либо явно написал «не выполнено, причина такая-то».
- Молча пропустить линтер, потому что «он шумный» — **нет**, линтер часть обязательного набора, если он настроен в проекте.
