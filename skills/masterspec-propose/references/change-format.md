# Формат change — diff-блоки vs файлы в new/

Как `masterspec-propose` выбирает формат для каждой правки. Таблица-решение + примеры. Файл читают `propose` при заполнении change.md и `apply-change` при мерже.

---

## 1. Правило выбора

| Характеристика правки | Формат |
|-----------------------|--------|
| Одна строка/bullet (AC, правило, строка таблицы, описание в компоненте) | diff-блок §4, тип `modify-bullet` |
| Добавление одного bullet в уже существующий список | diff-блок §4, тип `modify-bullet` (блок ДО пустой или ставим якорный bullet) |
| Замена раздела (`## ...` / `### ...`) целиком, содержимого < 50 строк | diff-блок §4, тип `replace-section` |
| Новый подраздел (`### cap-<имя>`) внутри существующего раздела | diff-блок §4, тип `add-subsection` |
| Добавление содержимого в начало файла (YAML-фронтматтер к файлу без него; вводный блок перед первым заголовком) | diff-блок §4, тип `prepend-to-file` |
| Замена раздела ≥ 50 строк ИЛИ требующая перестройки структуры | файл `new/<slug>-<section>.patch.md` + пометка в change.md §5 |
| Новый артефакт целиком (fn-, cmp-, scn-, alg-, nfr-, rules-, cdm-, dict-, adr-, data-, api-, tc-acc-, tc-int-, cmap-, trace-, dmap-, ui-view-, nav-) | файл `new/<slug>.md` + строка в §2.2 |
| Удаление артефакта целиком | только строка в §2.3 + обоснование в §6; файл не трогаем до apply |
| Любая правка `00-masterspec-index.md` | **запрещено вручную** — `apply-change` запускает `masterspec reindex`, он перегенерирует §3–§6 индекса целиком по файлам фабрики. Diff-блоки по индексу должны отклоняться на ревью. Исключение — точечные правки §1 «Паспорт» или §7 «Белые пятна» через diff-блок `modify-bullet`/`replace-section`. |

---

## 2. Структура diff-блока

Каждый блок в `change.md §4` имеет один и тот же формат:

```markdown
### 4.N. <slug> — <что меняем одной строкой>

**Файл**: <путь относительно masterspec/>
**Раздел**: <заголовок: `## ...` → `### ...` → pointer внутри>
**Тип правки**: modify-bullet | replace-section | add-subsection

ДО:
\`\`\`
<точный текст из целевого файла, который будет заменён>
\`\`\`

ПОСЛЕ:
\`\`\`
<новый текст>
\`\`\`
```

### 2.1. modify-bullet

Замена точной строки (обычно bullet списка или одна строка в тексте).

- `apply-change` ищет текст из `ДО:` в указанном разделе целевого файла строгим совпадением (после нормализации leading whitespace).
- Если не найден — конфликт, см. `merge-workflow.md`.

Пример:

```
**Файл**: 01-requirements/02-functions/fn-send-notification.md
**Раздел**: ## Критерии приёмки
**Тип правки**: modify-bullet

ДО:
- AC-03. Система возвращает статус доставки в течение 30 секунд.

ПОСЛЕ:
- AC-03. Система возвращает статус доставки в течение 30 секунд.
  При таймауте внешнего канала — ретрай до 3 раз с backoff 1s/5s/15s.
```

### 2.2. replace-section

Замена содержимого раздела целиком.

- `apply-change` находит раздел по заголовку, удаляет всё между ним и следующим заголовком того же уровня, вставляет `ПОСЛЕ:`.
- `ДО:` — можно оставить пустым, можно продублировать старое содержимое (для истории ревью).

Пример:

```
**Файл**: 01-requirements/03-nfr/nfr-delivery-latency.md
**Раздел**: ## Целевые показатели
**Тип правки**: replace-section

ДО:
(старое содержимое — опционально)

ПОСЛЕ:
- p50: 500 ms
- p95: 2000 ms
- p99: 5000 ms
- timeout: 10000 ms
```

### 2.3. add-subsection

Добавление нового подраздела в существующий раздел.

- `apply-change` находит указанный раздел, вставляет `ПОСЛЕ:` в конец раздела (после последней подсекции того же уровня).
- Используется в первую очередь для новых `cap-*` внутри существующего `cmp-`.

Пример:

```
**Файл**: 02-specifications/01-components/cmp-delivery-router.md
**Раздел**: ## Возможности
**Тип правки**: add-subsection

ПОСЛЕ:
### cap-retry — ретрай неудачных отправок

**Вход**: неуспешный исход отправки (-> scn-retry-flow).

**Поведение**: повторить отправку через тот же канал с backoff.

**Ограничения**: максимум 3 попытки; backoff 1s/5s/15s.

**Ошибки**: `retry-budget-exhausted` при исчерпании попыток.
```

### 2.4. prepend-to-file

Вставка содержимого **в самое начало файла**, до первой его строки. Применяется в первую очередь для дописывания YAML-фронтматтера к существующим файлам, у которых его нет (обычно — старые ADR, документы, импортированные извне).

- `apply-change` проверяет, что первые строки целевого файла **не** начинаются с блока `---` (иначе у файла уже есть фронтматтер — конфликт).
- Вставляет `ПОСЛЕ:` в начало файла; существующее содержимое сдвигается вниз без изменений.
- Поле `**Раздел**:` не требуется — ставь `(начало файла)` или опусти поле.
- Блок `ДО:` не требуется — ставь `(пустой)` или опусти.

Пример:

```
**Файл**: 04-decisions/adr-001-architecture-style.md
**Раздел**: (начало файла)
**Тип правки**: prepend-to-file

ПОСЛЕ:
---
type: adr
slug: adr-001-architecture-style
factory: sprint-management
status: actual
updated: 2026-04-23
---

```

**Важно**: в блоке `ПОСЛЕ:` для `prepend-to-file` ставь **завершающую пустую строку** перед закрывающим code fence — она сохранит визуальный отступ между фронтматтером и последующим `# Заголовком`.

## 3. Файлы в `new/`

### 3.1. Новый артефакт целиком

- Имя файла = slug (с префиксом: `fn-`, `cmp-`, `scn-` и т. д.)
- YAML-фронтматтер обязателен:
  ```yaml
  ---
  type: <из artifact-routing.md>
  slug: <префикс>-<kebab-case>
  factory: <slug-фабрики>
  status: draft
  updated: YYYY-MM-DD
  ---
  ```
- Содержимое — по шаблону `masterspec/templates/tpl-*.md`. HTML-комментарии из шаблона удалить.
- В change.md §2.2 — строка с этим slug'ом; в §5 — продублирован список для удобства ревью.

### 3.2. Патч раздела целиком (редко)

Для крупной замены раздела (≥ 50 строк), которую неудобно класть в diff-блок:

- Имя файла: `new/<slug>-<section-id>.patch.md`
- Содержимое — новое содержимое раздела без заголовка самого раздела.
- В change.md §4 — diff-блок типа `replace-section` с указанием: `ПОСЛЕ: (см. new/<slug>-<section>.patch.md)`.

Этот паттерн применяется редко — обычно проще разбить change на несколько мелких или сделать replace-section inline.

## 4. Запреты

- **Не мешай в одном change.md изменения из разных слоёв фабрики в таком объёме, чтобы change потерял фокус.** Если правка затрагивает больше 5 артефактов в разных слоях — подумай о разбиении на несколько change'ей. Исключение: согласованное архитектурное решение (ADR + несколько `cmp-`/`scn-`).
- **Не используй diff-блок для добавления нового артефакта.** Любой новый артефакт с уникальным slug'ом идёт через `new/<slug>.md`.
- **Не пиши в теле diff-блока что-то, что нарушает дисциплину слоёв.** Пример: в diff-блоке `fn-` нельзя упоминать имя компонента из ПОСЛЕ. См. `masterspec/references/layer-discipline.md`.
- **Не правь YAML-фронтматтер целевого файла через diff-блок** (кроме случаев миграции типа/slug'а, которые оформляются через remove + new). Поле `updated:` `apply-change` обновит автоматически.

## 5. Валидация change.md перед созданием

`propose` перед записью файла проверяет:

1. Шапка заполнена (все поля, статус = `На согласовании`).
2. §1 заполнен (цель + приоритет + слои).
3. §2 — хотя бы одна непустая таблица (MODIFIED / ADDED / REMOVED).
4. Для каждой строки §2.1 — есть соответствующий блок в §4.
5. Для каждой строки §2.2 — соответствующий файл `new/<slug>.md` создан и имеет валидный YAML-фронтматтер.
6. §7 заполнен или содержит `Нет изменений.`
7. §8 критерии приёмки на месте.
8. Дисциплина слоёв (`masterspec/references/layer-discipline.md`) проверена для каждого MODIFIED и ADDED.

При нарушении — не сохранять, показать проблемы пользователю.
