# masterspec-gen — сгенерировать один артефакт

Узел-исполнение (`../masterspec/references/patterns/element-workflow.md`). Я генерирую РОВНО ОДИН артефакт указанного типа в изолированном контексте — не вижу всю фабрику, только поданный фокус-набор. Чужое — только через ссылку.

## Вход (изолированный фокус-набор)
Готовит оркестратор/explore: бизнес-описание ровно этого элемента + релевантный срез вышестоящих артефактов (для fn — cdm+rules+глоссарий; для cmp — fn; для api — cmp+data) + шаблон. НЕ подаётся остальная фабрика.

**Формат фокус-набора:** связный markdown — бизнес-описание элемента, процитированные (не пересказанные) срезы вышестоящих артефактов и ссылки `-> slug`, плюс какой `tpl-<type>` заполнять. Это НЕ сырой YAML-агрегат explore. Оркестратор передаёт его gen-субагенту как содержимое задачи (Task-промпт) или файлом, путь к которому даёт `from=<путь>`.

**Куда писать (`target_path`/`target_mode`):** в lean-режиме приходят из плана (`patterns/context-isolation.md §Адресация`). Если не заданы — выводи по умолчанию: путь — из `artifact-routing.md` по `type` (api — по `scope`, fn — по `block`); режим — машинный api/data (`.yaml`/`.json`) → `sidecar` (+ компаньон `.md`), новый артефакт → `file`, изменение существующего `.md` → `diff` (выдай diff-блок для `change.md §4`, не пиши файл напрямую). Пиши ровно туда и так — не «по догадке».

## Метод
1. Прочитай шаблон `../masterspec/templates/tpl-<type>.md`, скопируй структуру.
2. Заполни по мета-модели. Дисциплина слоёв — `../masterspec/references/layer-discipline.md` (запреты для слоя элемента).
3. Тип-специфика.
   - **Требования:** `as` — назначение, границы, потребители, что умеет и не умеет; `fn` — обязательны альтернативные и исключительные потоки, каждый критерий приёмки как проверяемый оракул given-when-then; `nfr` — числовая метрика с квалификаторами (профиль нагрузки, окно измерения, условие); `rules` — формулировка абстрактная (без компонентов/таблиц), проставлен `immutable`; `cdm` — раздел «Состояния и переходы» (матрица состояние×событие без пустых ячеек, у перехода триггер и момент); `dict` — значения и источник структуры (ссылка на сущность `cdm`); `tc-acc` — под конкретные критерии приёмки функции.
   - **Спецификации:** `cmp` — раздел возможностей (`cap-*`); `scn` — порядок вызова возможностей; `api` — обе стороны контракта (вход / ответ-успех / ответ-отказ) и типы; `data` — состояния и логические ограничения; `alg` — пошаговая логика, ветвления, инварианты; `ui-view` (для систем с UI) — экран/представление: состав, источники данных, переходы между состояниями; `nav` — карта навигации между `ui-view` (входы/выходы экранов).
4. Ссылки `-> slug` только вверх/в слое.

## Дозапрос (≤2 раунда)
Не хватает фокус-набора (упомянута сущность вне поданного среза, неясное правило) — НЕ выдумывай. Запрос «не хватает X» идёт оркестратору, который маршрутизирует по единому контракту (`../masterspec/references/patterns/context-isolation.md §Дозапрос`): нужен факт из КОДА → `explore`; нужен срез ДОКУМЕНТА/артефакта (в lean) → `planner` дописывает `.focus`; нет ни там, ни там → открытый вопрос человеку. Лимит 2, потом «Открытый вопрос».

## Самопроверка перед отдачей
Прогон своего артефакта по оси 1 (`patterns/verification-axes.md`): пустой обязательный пункт → доделать или пометить «Открытый вопрос». Это НЕ замена verify слоя — лишь элементная гигиена.

## Выход
Зависит от `target_mode`:
- `file` (дефолт для нового) → `<type>-<slug>.md` (`status: draft`);
- `diff` (изменение существующего `.md`) → diff-блок для `change.md §4` (файл напрямую НЕ пишется);
- `sidecar` (машинный api/data) → пара: компаньон `<slug>.md` + машинный `<slug>.yaml`/`.json`.
Плюс список дозапросов и открытых вопросов → оркестратору в route-run.

## Развилка внутри генерации
Если по ходу встретился реальный ВЫБОР (несколько жизнеспособных вариантов модели/контракта/алгоритма) — не решай молча. Эскалируй оркестратору как развилку: он развернёт узел-решение (`../masterspec/references/patterns/decision-node.md`) и зафиксирует decision record. Молчаливый выбор = немое решение, потом его переоткроют.
