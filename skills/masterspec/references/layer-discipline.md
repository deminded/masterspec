# Дисциплина слоёв

Обязательна к применению перед каждым Write/Edit в артефакт фабрики. Полная версия — `meta_model.md`, раздел 2; здесь — операционная выжимка для скиллов.

---

## 1. Три слоя и их границы

| Слой | Вопрос | Директория | Что описывает |
|---|---|---|---|
| **Требования** | ЧТО даёт система и как себя ведёт | `01-requirements/` | Полезность, внешнее поведение, ограничения |
| **Спецификации** | КАК устроено взаимодействие частей | `02-specifications/` | Компоненты, их кооперация, контракты |
| **Кодовая база** | ГДЕ в коде это реализовано | `03-codemap/` | Файлы, классы, строки, таблицы |

**Трассировка и ссылки идут снизу вверх:**
- слой требований НЕ ссылается на спецификации и код;
- слой спецификаций ссылается вверх на требования (и в пределах своего слоя);
- слой кода ссылается вверх на спецификации (и на требования напрямую при необходимости).

Цель: читатель видит цепочку «вот это мы хотим → вот так это происходит → вот так это реализовано», без дублирования между слоями.

## 2. Жёсткие запреты по слоям

### 2.1. В слое требований ЗАПРЕЩЕНО

- имена компонентов, сервисов, модулей (`cmp-*`, «оркестратор», «адаптер»);
- имена таблиц, коллекций, индексов, полей, колонок;
- языки, фреймворки, библиотеки, СУБД, брокеры, протоколы (кроме случаев, когда это явное внешнее ограничение бизнеса);
- классы, методы, функции, файлы, пути, URL, эндпоинты;
- ссылки на `cmp-`, `scn-`, `alg-`, `api-`, `data-`, `cd-`, `lp-`, `tc-int-`, `cmap-`, `trace-`, `dmap-`.

### 2.2. В слое спецификаций ЗАПРЕЩЕНО

- код, псевдокод, имена классов/методов/функций;
- имена файлов, пути, строки, пакеты, namespace;
- конкретные таблицы, колонки, типы СУБД, миграции, ORM-классы;
- конкретные технологии (PostgreSQL, Redis, Kafka, Spring) в телах контрактов и данных — если только это не внешнее ограничение, зафиксированное в ADR;
- ссылки на `cmap-`, `trace-`, `dmap-`.

### 2.3. В слое кодовой базы

Всё конкретное (`file:line`, классы, таблицы) — ТОЛЬКО здесь. Каждый code unit ссылается вверх на slug из спецификации (при отсутствии — на требования).

## 3. Что где живёт (во избежание путаницы)

- **Use-case** (черновой сценарий в терминах «актор — система») — в функции АС/ФП (`fn-`), не в сценарии.
- **Порядок вызова компонентов** — только в сценарии (`scn-`), не в функции.
- **Функциональность компонента** — в разделе «Возможности» внутри `cmp-` (как `cap-*`). Отдельных файлов на функциональность нет.
- **Алгоритм принятия решений** внутри компонента — в алгоритме (`alg-`), не в функции.
- **Контракт взаимодействия** (вход/выход, ошибки, SLA) — в API (`api-`), не в компоненте.
- **Ключевые справочники/классификаторы** предметной области — в `dict-` (только если малые, стабильные, доменные). Структура полей — в сущности `cdm-`.
- **Приёмочные тесты** (чёрный ящик, проверяют AC функции) — в `tc-acc-`, слой требований.
- **Интеграционные тесты** (проверяют сценарий) — в `tc-int-`, слой спецификаций.

Если сомневаешься — см. `meta_model.md`, раздел 2.

## 4. Направление ссылок

```
КОД          cmap  trace  dmap                              ──► ссылаются на cmp, scn, cdm, data
             ▲     ▲      ▲
СПЕЦЫ        cmp(cap)  scn  alg  data  api  lp  cd  tc-int  ──► ссылаются на fn, as, cdm, nfr, rules, dict
             ▲         ▲                                 ▲
ТРЕБОВАНИЯ   as  fn  nfr  rules  context  fd  cdm  dict  tc-acc
```

- Стрелка `▲` = «ссылается на вышестоящий слой».
- Обратных стрелок быть не должно. Если в артефакте требований встретилась ссылка на `cmp-` / `scn-` / `api-` / `data-` / `lp-` / `tc-int-` / `cmap-` / `trace-` / `dmap-` — это ошибка, убрать.
- `cap-*` не выделяется в отдельный файл — это раздел внутри `cmp-`. Ссылка из сценария: `-> cmp-<slug>/cap-<name>`.

## 5. Проверка перед сохранением

Чек-лист для workflow-скиллов (propose, apply-change):

1. **Слой определён.** Куда кладу — `01-requirements/`, `02-specifications/`, `03-codemap/` или `04-decisions/`?
2. **Содержимое соответствует слою.** Запреты из §2 применены.
3. **Ссылки идут снизу вверх.** Из этого файла не указаны slug'и нижележащих слоёв.
4. **Нет запрещённой лексики.** В требованиях — нет имён компонентов/технологий; в спецификациях — нет кода/путей/таблиц.

Если хоть один пункт не выполнен — блок, показать пользователю, не сохранять.
