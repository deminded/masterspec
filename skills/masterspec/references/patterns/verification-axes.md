# Паттерн: 5 осей верификации (предмет скилла `verify`)
_Вычитка артефактов как ДАННЫЕ. Слой задаёт конфиг (verify scope=req|spec|change), но оси общие. Доказано эмпирически: оси 4-5 ловят блокеры и security, недостижимые статическими 1-3._

## Ось 1 — элемент изолированно [уровень: элемент]
Чек-лист по типу артефакта: все обязательные разделы заполнены; AC проверяемы (число/факт, не «удобно/корректно»); nfr с числовой метрикой + квалификаторами (peak/sustained, окно, условие); cdm/data с разделом «состояния и переходы».

## Ось 2 — внутрислоевая консистентность [слой]
`-> slug` ведёт на существующий артефакт; нет дублей (single-source, повтор→ссылка); сущности покрыты; нет конфликтующих правил; при конфликте NFR↔AC — требовать развязку, не висячее противоречие.

## Ось 3 — согласованность с эталоном / межслой [слой]
термины в глоссарии; ссылки спека→требования корректны и не фиктивны; на каждое rule — alg либо явное «тривиально»; каждый AC покрыт приёмочным тестом.

## Ось 4 — динамика-прогон [оба]
**4a содержание:** все альтернативные/исключительные ветки покрыты; полная матрица состояние×событие (пустая ячейка = блокер); реестр триггеров переходов (триггер+исполнитель+момент); трасса «событие→данные» с side-effects на каждой ветке; реализация не стирает различие контракта (404≠410).
**4b годность для следующего слоя:** каждый AC = оракул given-when-then; нет неопределённого «корректно»; единый источник для вычисляемых величин (now).

## Ось 5 — состязательный прогон (негатив) [оба] — отдельный антагонист
обязательный security/приватность-чеклист (SSRF, нормализация, коллизии идентификаторов, enumeration→rate-limit, PII-retention, BVA границы); контрпримеры «что сломается/чего не хватает»; обе стороны контракта (вход + ответ-успех + ответ-отказ от изменения).

## Дешёвое ядро (внедрять первым)
структполнота → глоссарий → матрица состояний → трасса событие→данные → оракул given-when-then.

## O0 (для scope=spec) — single-source контракта, hard-fail
РОВНО одна нормативная редакция каждого api/data. Два значения одного поля до поэлементного ревью = немедленный hard-fail. O6 контракт (можно ли написать клиента только по спеке?) и O7 физмодель (типы/ключи/индексы; выбор сделан) — здесь же.

## Метрика (R-METRICS, отчуждаемо от содержания)
holes по осям; codegen_ready (scope=spec) / spec_ready (scope=req); остаток по типу {вне-scope / противоречие / недоведённый-негатив}. Выносится без содержания.
