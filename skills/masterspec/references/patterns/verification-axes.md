# 5 осей верификации (предмет скилла verify)

Вычитка артефактов как набор проверок-данных. Слой задаёт конфигурацию (verify scope=req|spec|change), оси общие. Оси 4-5 (динамика и негатив) находят дефекты, которых не дают статические оси 1-3.

## Ось 1 — элемент изолированно
Чек-лист по типу артефакта: все обязательные разделы заполнены; критерии приёмки проверяемы (число или факт, не «удобно/корректно»); НФТ с числовой метрикой и квалификаторами (пиковая/средняя нагрузка, окно измерения, условие); cdm/data с разделом «состояния и переходы».

## Ось 2 — внутрислоевая консистентность
Каждая ссылка `-> slug` ведёт на существующий артефакт; нет дублирования (один факт в одном месте, повтор → ссылка); все упомянутые сущности покрыты; нет конфликтующих правил; при конфликте НФТ и критерия приёмки — требовать развязку, не оставлять висячее противоречие.

## Ось 3 — согласованность с эталоном и между слоями
Термины определены в глоссарии; ссылки из спецификаций на требования корректны и не фиктивны (срабатывают при прогоне); на каждое правило есть алгоритм либо явная пометка «тривиально»; каждый критерий приёмки покрыт приёмочным тестом.

## Ось 4 — динамика (прогон)
**Содержание:** все альтернативные и исключительные ветки покрыты; полная матрица «состояние × событие» (пустая ячейка = блокер); у каждого перехода состояния задан триггер, исполнитель и момент (при чтении / по расписанию / при записи); трасса «событие → данные» с побочными эффектами на каждой ветке, включая ошибочные; выбранная реализация не стирает различие, требуемое контрактом (например, «не найдено» ≠ «истекло»).
**Годность для следующего слоя:** каждый критерий приёмки выражен как проверяемый сценарий given-when-then; нет неопределённого «обрабатывает корректно»; для вычисляемых величин задан единый источник (например, единый источник текущего времени).

## Ось 5 — состязательный прогон (негатив), отдельным проверяющим
Обязательный чек-лист безопасности и приватности: подделка серверных запросов (SSRF), нормализация ввода, коллизии генерируемых идентификаторов, перебор идентификаторов и наличие ограничения частоты, политика хранения персональных данных, граничные значения. Контрпримеры «что сломается / чего не хватает». Обе стороны контракта: вход + ответ-успех + ответ-отказ, порождённый изменением.

## Дешёвое ядро (внедрять первым)
структурная полнота → однозначность терминов по глоссарию → матрица состояний → трасса «событие → данные» → сценарий given-when-then.

## Для scope=spec — ось O0 (single-source), hard-fail
Ровно одна нормативная редакция каждого контракта (api) и схемы (data). Два значения одного поля в разных разделах до поэлементной вычитки = немедленный hard-fail. Здесь же: O6 контракт интерфейса (можно ли написать клиента только по спецификации?) и O7 физическая модель данных (типы, ключи, индексы заданы; выбор сделан, не оставлен открытым).

## Метрика (отчуждаемо от содержания)
holes по осям; codegen_ready (для scope=spec) или spec_ready (для scope=req); остаток по типу (вне-области / противоречие / недоведённое). Выносится без содержания — для приёмки и отчёта.
