# 5 осей верификации (предмет скилла verify)

Вычитка артефактов как набор проверок-данных. Слой задаёт конфигурацию (verify scope=req|spec|change), оси общие. Метки: O1, O2, O3, O4 (с под-осями O4a и O4b), O5 — общие 5 осей; O0, O6, O7 — дополнительные для scope=spec. Критерий spec_ready (scope=req) = нет блокеров O1, O4b, O5; codegen_ready (scope=spec) = O0 пройден, O6/O7 полны, нет блокеров. Оси 4-5 (динамика и негатив) находят дефекты, которых не дают статические оси 1-3.

## O1 — элемент изолированно
Чек-лист по типу артефакта: все обязательные разделы заполнены; критерии приёмки проверяемы (число или факт, не «удобно/корректно»); НФТ с числовой метрикой и квалификаторами (пиковая/средняя нагрузка, окно измерения, условие); cdm/data с разделом «состояния и переходы».

## O2 — внутрислоевая консистентность
Каждая ссылка `-> slug` ведёт на существующий артефакт; нет дублирования (один факт в одном месте, повтор → ссылка); все упомянутые сущности покрыты; нет конфликтующих правил; при конфликте НФТ и критерия приёмки — требовать развязку, не оставлять висячее противоречие.

## O3 — согласованность с эталоном и между слоями
Термины определены в глоссарии; ссылки из спецификаций на требования корректны и не фиктивны (срабатывают при прогоне); на каждое правило есть алгоритм либо явная пометка «тривиально»; каждый критерий приёмки покрыт приёмочным тестом.

## O4 — динамика (прогон)
**O4a — прогон содержания:** все альтернативные и исключительные ветки покрыты; полная матрица «состояние × событие» (пустая ячейка = блокер); у каждого перехода состояния задан триггер, исполнитель и момент (при чтении / по расписанию / при записи); трасса «событие → данные» с побочными эффектами на каждой ветке, включая ошибочные; выбранная реализация не стирает различие, требуемое контрактом (например, «не найдено» ≠ «истекло»).
**O4b — годность для следующего слоя:** каждый критерий приёмки выражен как проверяемый сценарий given-when-then; нет неопределённого «обрабатывает корректно»; для вычисляемых величин задан единый источник (например, единый источник текущего времени).

## O5 — состязательный прогон (негатив), отдельным проверяющим
Обязательный чек-лист безопасности и приватности: подделка серверных запросов (SSRF), нормализация ввода, коллизии генерируемых идентификаторов, перебор идентификаторов и наличие ограничения частоты, политика хранения персональных данных, граничные значения. Контрпримеры «что сломается / чего не хватает». Обе стороны контракта: вход + ответ-успех + ответ-отказ, порождённый изменением.

## Дешёвое ядро (внедрять первым)
структурная полнота → однозначность терминов по глоссарию → матрица состояний → трасса «событие → данные» → сценарий given-when-then.

## Для scope=spec — дополнительные оси O0, O6, O7
**O0 — single-source (hard-fail).** Ровно одна нормативная редакция каждого контракта (api) и схемы (data). Два значения одного поля в разных разделах до поэлементной вычитки = немедленный hard-fail.

**O6 — контракт интерфейса** (вопрос-критерий: можно ли написать клиента ТОЛЬКО по спецификации?). Чек-лист по каждой операции/событию api:
- задан идентификатор (endpoint / topic / method);
- ВХОД: все поля, их обязательность, значимые заголовки;
- ОТВЕТ-УСПЕХ: статус успеха + форма ответа;
- ОТВЕТ-ОТКАЗ: классы ошибок со статусами, включая порождённые ограничениями; различимые по смыслу исходы дают различимые ответы («не найдено» ≠ «истекло»);
- нефункциональные обязательства контракта (идемпотентность, SLA, безопасность ретраев, версионирование) заданы.

**O7 — физическая модель данных** (вопрос-критерий: готова ли к генерации хранилища?). Чек-лист по каждой data-сущности:
- у каждого атрибута задан тип (логический на спеке);
- ключи и уникальность определены;
- связи между сущностями с кратностью;
- ограничения (обязательность, диапазоны, TTL по смыслу) заданы;
- по каждой форме хранения выбор СДЕЛАН (не «TBD»); значимый выбор записан в `dr-`/`adr-`.

## Метрика (отчуждаемо от содержания)
holes по осям; codegen_ready (для scope=spec) или spec_ready (для scope=req); остаток по типу (вне-области / противоречие / недоведённое). Выносится без содержания — для приёмки и отчёта.
