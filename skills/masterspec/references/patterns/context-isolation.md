# Изоляция контекста оркестратора (lean-режим)

## Зачем
На модели с ограниченным контекстом (например 128k) последовательная работа оркестратора (`derive`/`evolve`) на фабрике среднего размера переполняет контекст: оркестратор по ходу читает мета-модель, шаблоны, research, вышестоящие артефакты, отчёты вычитки — и пухнет. Этот паттерн делает рост контекста оркестратора независимым от размера фабрики.

## Принцип: оркестратор — диспетчер по файлам, не читатель
В `context=lean` оркестратор держит в контексте ТОЛЬКО:
- план слоя (`plan-<layer>.md`: список элементов, тип узла, путь к фокус-набору);
- журнал прогона (`route-run-<ts>.md`: статусы, метрики);
- пути и КОРОТКИЕ сводки от субагентов.

Оркестратор НЕ держит: содержимое артефактов, мета-модель, шаблоны, сырой research, отчёты вычитки. Всё тяжёлое чтение и обработку выполняют изолированные субагенты, которые пишут результат в файл и возвращают сводку (≤ ~10 строк: путь + метрики + блокеры/дозапросы).

## Два уровня изоляции
- **explore** изолирует чтение КОДА: read-only разведчики по репозиторию → `.research/`. Нужен только при наличии кода.
- **planner** изолирует чтение ДОКУМЕНТОВ: бизнес-запрос + `.research/` (если был) + мета-модель + поимённо нужные вышестоящие артефакты → план и фокус-наборы.

Оркестратор не читает ни код, ни документы — между шагами у него только план, сводки и пути.

## Режимы
Параметр `context=lean|full` у `derive`/`evolve`:
- `full` (дефолт) — текущее поведение: оркестратор может читать сам, меньше субагент-оверхеда. Для моделей с большим контекстом.
- `lean` — строгая делегация по этому паттерну. Для ограниченного контекста; рост контекста оркестратора не зависит от размера фабрики.

## Planner-субагент
Вход: бизнес-запрос; `.research/_aggregate` (если explore был); мета-модель (состав и дисциплина слоя); шаблоны типов слоя.
Делает: разворачивает слой на элементы в порядке состава (`derive §2`), на каждый элемент готовит фокус-набор и пишет `.work/.focus/<slug>.md`; пишет `.work/plan-<layer>.md` (элемент → тип узла → путь фокус-набора → вышестоящие slug'и).
Возвращает оркестратору: путь к плану + число элементов + открытые вопросы. Содержимое в контекст оркестратора не возвращается.

planner работает в порядке состава слоя (например `cdm` раньше `fn`): фокус-набор элемента раскладывается тогда, когда его вышестоящие уже созданы. Каждый проход planner одноразовый — его контекст не накапливается через слой.

## Граница фокус-набора (что planner кладёт)
Фокус-набор — самодостаточный для `gen`: gen читает ТОЛЬКО свой `.focus/<slug>.md`, в фабрику не лазит. Поэтому planner ВПИСЫВАЕТ в него вырезанный релевантный срез вышестоящих (не путь, а кусок), а не весь слой.
- кладётся: бизнес-описание ровно этого элемента + поимённый срез вышестоящих, на которые элемент сошлётся по своему описанию (для `fn` — конкретные сущности `cdm`, относящиеся `rules`, термины глоссария), + указание шаблона `tpl-<type>`;
- НЕ кладётся: вся фабрика, целые вышестоящие файлы «на всякий случай».
planner читает вышестоящие по одному и вырезает нужное, не держа всё разом. Эвристика границы: «то, что элемент трогает по бизнес-описанию + один уровень связей».

## Дозапрос в lean
gen не хватило фокус-набора → запрос идёт НЕ оркестратору (он ничего не читает), а planner'у: planner добирает нужный срез в `.focus/<slug>.md`, возвращает оркестратору «добрал X для <slug>», оркестратор перезапускает gen. Лимит — 2 раунда, потом открытый вопрос человеку.

## Делегация шагов в lean
1. `explore` → `.research/` + сводка (если есть код).
2. `planner` → `.work/plan-<layer>.md` + `.work/.focus/*` + сводка.
3. на каждый элемент плана: `gen` получает ПУТЬ к фокус-набору, пишет артефакт, возвращает путь + статус + дозапросы.
4. `verify` (субагент) читает слой → `.work/verify-report-<ts>.md` + метрики/блокеры.
5. приёмщик (субагент) сводит редакции → пишет на месте + сводка.

Между шагами оркестратор оперирует только путями и сводками.

## Рабочая зона и обязательная чистка
Эфемерные lean-файлы — в `masterspec/.work/` (`plan-*.md`, `.focus/*`, `verify-report-*`), вне боевого дерева.

`.work/` ОЧИЩАЕТСЯ после завершения прогона (слой согласован / изменение применено) — обязательный финальный шаг, не опция. Причина: фокус-наборы содержат ВПИСАННЫЕ срезы вышестоящих артефактов, то есть дубли содержания фабрики; оставлять их нельзя — при правке оригинала срез в `.work/` устаревает (рассинхрон) и копится мусор.

Аудит-след прогона — это `route-run-<ts>.md` (метрики, отчуждаемые ОТ содержания: что собрано, дозапросы, оси, кто принял). Он содержания не дублирует, поэтому хранится отдельно (в корне фабрики при генерации / в `changes/<name>/` при изменении) и переживает чистку. Итог: `route-run` сохранить, `.work/` удалить.
