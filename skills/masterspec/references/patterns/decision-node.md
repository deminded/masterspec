# Узел-решение

Тип шага, где есть реальная развилка — несколько жизнеспособных вариантов с разными компромиссами. Здесь не исполняют сразу, а вырабатывают решение.

## Когда узел-решение, а когда исполнение
Оцени перед запуском: есть ли реальный выбор (модель данных, форма контракта, алгоритм)? Да → узел-решение. Нет (вариант один или механический) → прямое исполнение (см. `element-workflow.md`). Не разворачивай дивергенцию на тривиальном — это лишние затраты.

## Процесс
1. Контекст и критерий: что выбираем, ограничения, что значит «лучше».
2. Дивергенция: 2-3 варианта с компромиссами, рекомендуемый первым.
3. Сравнение по критерию.
4. Выбор: один вариант или обоснованный гибрид.
5. Decision record: зафиксируй ПОЧЕМУ выбран вариант в `04-decisions/adr-<slug>.md`. Доступен при необходимости, не загромождает рабочий контекст.
6. Hard-gate: человек одобряет выбор до перехода к исполнению (см. `enforcement.md`).

## Рекурсия по уровням
Применим на любом уровне: общий замысел и архитектура (там развилок больше всего) и отдельный артефакт с реальным выбором — тот же процесс в миниатюре.

## Удержание и снятие
Дивергенция удерживает варианты как ресурс; на выходе фиксируется ОДИН выбор (особенно для контракта — одно значение). Не путай продуктивную развилку (выбор с decision record) с расщеплением — двумя несведёнными редакциями одного и того же, которое есть дефект (его сводит приёмщик, см. `element-workflow.md`).

## Дисциплина в скиллах (обязательность записи)
Узел-решение НЕ завершён без записи. «Решение принято, но не записано» = НЕМОЕ РЕШЕНИЕ = непройденный узел (по аналогии с немым вердиктом). Правила, встроенные в скиллы:
- **decision-node:** на выходе обязателен decision record — `adr-*` (если развилка архитектурная, сквозная) или `dr-*` (если касается одного артефакта). В записи фиксируются и ОТВЕРГНУТЫЕ варианты с причиной — иначе через полгода выбор переоткроют.
- **gen** (элемент-генератор): встретил развилку с реальным выбором — НЕ решает молча. Эскалирует оркестратору (→ узел-решение → запись). Молчаливый выбор внутри генерации = немое решение.
- **derive / evolve** (оркестраторы): каждый пройденный узел-решение обязан оставить запись; выбор без записи блокирует переход (hard-gate).
- **verify:** значимый выбор без decision record — флаг «немое решение» (недоведённое), как пустая ячейка состояний.
Критерий обязательности: запись нужна там, где был РЕАЛЬНЫЙ выбор (≥2 жизнеспособных варианта). На тривиальном (вариант один/механический) запись не плодится.
