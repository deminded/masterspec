# Формат change — diff-блоки vs файлы в new/

Канонический формат change.md и выбор формата для каждой правки. Читают: `evolve` (шаг 0 — заполнение change.md) и `apply-change` (мерж).

change.md состоит из шапки (статус, фабрика) и 8 секций: §1 цель, §2 таблицы (2.1 MODIFIED / 2.2 ADDED / 2.3 REMOVED), §4 diff-блоки, §5 список new/, §6 обоснования REMOVED, §7 влияние на ссылки, §8 критерии приёмки.

---

## 1. Правило выбора формата правки

| Характеристика правки | Формат |
|-----------------------|--------|
| Одна строка/bullet (AC, правило, строка таблицы, описание в компоненте) | diff-блок §4, тип `modify-bullet` |
| Добавление одного bullet в существующий список | diff-блок §4, тип `modify-bullet` (`ДО:` = якорный соседний bullet) |
| Замена раздела (`## ...` / `### ...`) целиком | diff-блок §4, тип `replace-section` |
| Новый подраздел (`### cap-<имя>`) внутри существующего раздела | diff-блок §4, тип `add-subsection` |
| Новый артефакт целиком (fn-, cmp-, scn-, alg-, nfr-, rules-, cdm-, dict-, adr-, dr-, data-, api-, tc-acc-, tc-int-, ui-view-, nav-) | файл `new/<slug>.md` + строка в §2.2 |
| Новый артефакт кодового слоя (repo-map, cmap-, trace-, dmap-) | тот же `new/<slug>.md` + строка §2.2, фронтматтер `generated: true`. Обычно codemap не заводят через change вручную — он порождается `recover source=code`/`derive`; через change — только если правится осознанно. |
| Удаление артефакта целиком | строка в §2.3 + обоснование §6; файл не трогаем до apply |
| Правка `00-masterspec-index.md` | **не вручную** — `apply-change` делает полную перегенерацию §3–§6 по файлам фабрики (`index-canonical.md`). Diff-блоки по §3–§6 индекса отклоняются на ревью. Исключение — §1 «Паспорт» / §7 «Белые пятна» через `modify-bullet`/`replace-section`. |

Три типа diff-блока (`modify-bullet`, `replace-section`, `add-subsection`) покрывают все правки существующих артефактов. Большая замена — это `replace-section`; если правок много — разбей change на несколько.

## 2. Структура diff-блока

Каждый блок в `change.md §4`:

```markdown
### 4.N. <slug> — <что меняем одной строкой>

**Файл**: <путь относительно masterspec/>
**Раздел**: <заголовок: `## ...` → `### ...`>
**Тип правки**: modify-bullet | replace-section | add-subsection

ДО:
```
<точный текст из целевого файла, который будет заменён>
```

ПОСЛЕ:
```
<новый текст>
```
```

### 2.1. modify-bullet
Замена точной строки (bullet списка или строка текста). `apply-change` ищет текст из `ДО:` в указанном разделе строгим совпадением (после нормализации leading whitespace); не найден — конфликт (`merge-workflow.md`).

```
**Файл**: 01-requirements/02-functions/fn-send-notification.md
**Раздел**: ## Критерии приёмки
**Тип правки**: modify-bullet

ДО:
- AC-03. Система возвращает статус доставки в течение 30 секунд.

ПОСЛЕ:
- AC-03. Система возвращает статус доставки в течение 30 секунд.
  При таймауте внешнего канала — ретрай до 3 раз с backoff 1s/5s/15s.
```

### 2.2. replace-section
Замена содержимого раздела целиком. `apply-change` находит раздел по заголовку, удаляет всё между ним и следующим заголовком того же уровня, вставляет `ПОСЛЕ:` (без самого заголовка). `ДО:` — можно оставить пустым или продублировать старое для истории ревью.

```
**Файл**: 01-requirements/03-nfr/nfr-delivery-latency.md
**Раздел**: ## Целевые показатели
**Тип правки**: replace-section

ПОСЛЕ:
- p50: 500 ms
- p95: 2000 ms
- p99: 5000 ms
- timeout: 10000 ms
```

### 2.3. add-subsection
Добавление нового подраздела в существующий раздел. `apply-change` находит раздел, вставляет `ПОСЛЕ:` в конец раздела (после последней подсекции, перед границей). Прежде всего — новые `cap-*` внутри существующего `cmp-`.

```
**Файл**: 02-specifications/01-components/cmp-delivery-router.md
**Раздел**: ## Возможности
**Тип правки**: add-subsection

ПОСЛЕ:
### cap-retry — ретрай неудачных отправок

**Вход**: неуспешный исход отправки (-> scn-retry-flow).
**Поведение**: повторить отправку через тот же канал с backoff.
**Ограничения**: максимум 3 попытки; backoff 1s/5s/15s.
**Ошибки**: `retry-budget-exhausted` при исчерпании попыток.
```

## 3. Файлы в `new/` (новый артефакт целиком)

- Имя файла = slug (с префиксом: `fn-`, `cmp-`, `scn-`, …).
- YAML-фронтматтер обязателен. `status` нового артефакта в `new/`:
  - артефакты слоёв (fn/cmp/scn/api/data/…) — `status: draft`;
  - новое решение `adr-` — `status: proposed` (фаза предложения; в `accepted` переводит человек после принятия, не при создании); локальное `dr-` — `status: accepted` сразу (фазы предложения нет).
- Содержимое — по шаблону `masterspec/templates/tpl-*.md`, HTML-комментарии удалить.
- В change.md §2.2 — строка с этим slug; в §5 — продублирован список для ревью.

### 3.1. Машинные форматы api/data (`.yaml`/`.json`) — режим компаньона
Контракт `api` и схема `data` могут существовать в машинном виде (OpenAPI/AsyncAPI/JSON Schema), нужном внешним процессам (кодоген, валидация, mock). Тогда артефакт — **ПАРА файлов с одним slug, рядом в одной директории**:
- `<slug>.yaml` (или `.json`) — сама машинная спека (её потребляют внешние процессы);
- `<slug>.md` — **компаньон, обязателен**: masterspec-фронтматтер (`type`, `slug`, `factory`, `status`, для api — `scope`), назначение, ограничения и ссылка `Машинная спека: <slug>.yaml`.

Правила:
- В `new/` кладутся **ОБА файла**. `apply-change` копирует пару в целевую директорию.
- Индексация, `verify` и определение статуса идут по **компаньону** (`.md` несёт фронтматтер); машинный файл метаданных masterspec не содержит.

**ДОБАВЛЕНИЕ новой пары** (артефакта ещё нет): строка в §2.2 ADDED, оба файла в `new/`; `apply-change` копирует пару, проверяя отсутствие коллизии целевого `<slug>.md`.

**ИЗМЕНЕНИЕ существующей пары** (артефакт уже есть):
- *машинный sidecar* меняется ЗАМЕНОЙ ЦЕЛИКОМ — строка в §2.1 MODIFIED (slug + путь `<slug>.yaml`), новая версия кладётся в `new/<slug>.yaml`. `apply-change` перезаписывает существующий sidecar (целевой ДОЛЖЕН существовать — это MODIFIED, а не ADDED с проверкой коллизии). Diff-блоки §4 по `.yaml`/`.json` НЕ применяются — частичный merge структурированных форматов запрещён намеренно (риск некорректного слияния);
- *компаньон `.md`* меняется обычными diff-блоками §4 (или, при крупной переработке, `replace-section`).

При мерже `apply-change` копирует/заменяет файл(ы) в целевой директории (по `type:` через `artifact-routing.md`) и проставляет финальный статус: артефакты слоёв → `actual` (merge PR уже был согласованием); `adr-`/`dr-` сохраняют свой решенческий статус.

## 4. Запреты
- **Не теряй фокус change.** Правка > 5 артефактов в разных слоях — подумай о разбиении. Исключение: согласованное архитектурное решение (ADR + несколько `cmp-`/`scn-`).
- **Не добавляй новый артефакт diff-блоком** — только через `new/<slug>.md`.
- **Не нарушай дисциплину слоёв в теле diff-блока** (`masterspec/references/layer-discipline.md`).
- **Не правь YAML-фронтматтер целевого файла diff-блоком** (миграция типа/slug — через remove + new). `updated:` `apply-change` обновляет сам.

## 5. Валидация change.md перед сохранением

`evolve` (шаг 0) перед записью проверяет:
1. Шапка заполнена, статус = `На согласовании`.
2. §1 заполнен (цель + приоритет + слои).
3. §2 — хотя бы одна непустая таблица.
4. Для каждой строки §2.1: если путь `.md` — есть diff-блок в §4; если путь машинного sidecar (`.yaml`/`.json`) — diff-блока НЕТ, есть файл-замена `new/<slug>.yaml` (sidecar-replace, §3.1).
5. Для каждой строки §2.2 — файл `new/<slug>.md` создан, YAML-фронтматтер валиден.
5a. Для машинной пары (api/data с sidecar) — компаньон `<slug>.md` И машинный `<slug>.yaml`/`.json` присутствуют в `new/`, и машинный файл парсится как валидный YAML/JSON. Битый/отсутствующий sidecar ловится здесь, на создании change, а не в конце мержа.
6. §7 заполнен или `Нет изменений.`
7. §8 критерии приёмки на месте.
8. Дисциплина слоёв проверена для каждого MODIFIED и ADDED.

Нарушение — не сохранять, показать проблемы пользователю.
