# Канонический формат и алгоритм reindex для `00-masterspec-index.md`

Этот справочник — единственный источник истины для структуры индекса фабрики. Используется:
- kernel-режимом `reindex` при полной регенерации индекса;
- `masterspec-apply-change` после применения diff-блоков и копирования `new/*` (запускает reindex автоматически);
- ревью индекса при smoke-тестировании.

Ручной правке в `00-masterspec-index.md` подлежат только разделы §1 «Паспорт» и §7 «Белые пятна и открытые вопросы». Всё остальное — производная, переписывается reindex'ом.

---

## 1. Канонические разделы

Нумерация стабильная: число определяется типом артефакта, а не порядком появления. Если в фабрике нет артефактов определённого типа — раздел полностью отсутствует (и его номер не занимает пустую строку). Когда артефакты появляются — раздел получает тот же номер, что в таблице ниже.

### 1.1. Требования (§3.x)

| § | Заголовок | Типы |

## 2. Порядок строк внутри раздела

- По алфавиту значения `slug:` из фронтматтера.
- Исключение — §3.2 (функции): сначала группировка по `block:` (поддиректория), поддиректории — по алфавиту; внутри блока — по алфавиту slug. Если у функции нет `block:` — она попадает в сам раздел до подзаголовков.
- Исключение — §6 (ADR): по алфавиту slug (обычно это `adr-NNN-...`, что даёт хронологический порядок).

---

## 3. Формат строки

Каждая строка артефакта:

```
 <marker> `<path-from-factory-root>` # <comment>
```

- `<marker>` — один символ по правилу:
  - `+` если `status: actual`
  - `-` если `status: draft` ИЛИ если поля `status:` во фронтматтере нет (в частности, для `00-glossary.md`, у которого статус как правило не проставляют).
  - `?` если файла нет (применимо, только если артефакт упомянут в индексе явно — обычно через §2.2 change.md в состоянии «планируется»)
- `<path-from-factory-root>` — от корня фабрики, в backtick'ах.
- `<comment>` — свободный текст после `#`.

### 3.1. Источник комментария

Порядок приоритета при генерации:
1. Если в предыдущей версии `00-masterspec-index.md` по тому же `<path>` уже был комментарий — сохраняется дословно.
2. Иначе берётся H1-заголовок целевого файла (`# <текст>` в первой строке после фронтматтера). Если в заголовке есть шаблон `: <подзаголовок>` — берётся часть после `:`.
3. Иначе — пустой комментарий (строка заканчивается на пути).

### 3.2. Capabilities под компонентом (§4.1 только)

После строки с `cmp-*` следуют строки возможностей — по одной на каждую `### cap-<name> — <описание>` внутри тела файла компонента:

```
 + `02-specifications/01-components/cmp-foo.md` # foo
   · cap-bar — описание
   · cap-baz — описание
```

- Отступ `   · ` (3 пробела + точка-буллет + пробел).
- Текст после `·`: `cap-<name> — <описание из заголовка>`. Если описания нет — только `cap-<name>`.
- Порядок — по алфавиту `<name>`.
- Между компонентами — одна пустая строка.

## 4. Шаблон итогового файла

```markdown
---
type: masterspec-index
factory: <slug>
updated: YYYY-MM-DD
---
# Индекс фабрики: <slug>

## 1. Паспорт
<— сохраняется дословно из предыдущей версии —>

---

## 2. Общие артефакты
 + `00-glossary.md` # <комментарий, если файл есть>

---

## 3. Слой требований

### 3.1. Паспорт АС/ФП
 <строки>

### 3.2. Функции АС/ФП

#### <block-1>
 <строки>

#### <block-2>
 <строки>

### 3.N. <заголовок из §1.1>
 <строки>

---

## 4. Слой спецификаций

### 4.N. <заголовок из §1.2>
 <строки>

---

## 5. Слой кодовой базы (LLD)

### 5.N. <заголовок из §1.3>
 <строки>

---

## 6. Решения (ADR)
 <строки>

> В индекс попадают только сквозные `adr-` (из `04-decisions/`). Локальные `dr-` НЕ индексируются отдельной строкой: они привязаны к артефакту-владельцу, лежат на его слое и грузятся только когда нужно обоснование. Это намеренно — чтобы индекс отражал truth текущего состояния, а не каждое локальное «почему».

---

## 7. Белые пятна и открытые вопросы
<— сохраняется дословно из предыдущей версии —>

---

### Легенда
 + done — файл создан, содержание актуально
 - draft — файл создан, содержание неполное
 ? not started — файл не создан
 · (точка) — возможность внутри компонента, не отдельный файл
```

Разделитель `---` ставится только между слоями 3↔4↔5↔6↔7 и после §2. Между подразделами внутри слоя (например, 3.1 и 3.2) — не ставится.

Пустые разделы в финальный файл не выводятся: если, скажем, нет `dict-*`, раздел «### 3.7. Справочники» отсутствует целиком, а §3.8 остаётся с номером 3.8 (не смещается вверх).

## 5. Алгоритм reindex (псевдокод)

```
1. Прочитать текущий 00-masterspec-index.md (если есть):
   - frontmatter (factory)
   - §1 «Паспорт» — сохранить в переменную passport_section
   - §7 «Белые пятна» — сохранить в переменную gaps_section
   - все строки с путями — построить map path → comment_text

2. Просканировать фабрику:
   - Glob("01-requirements/**/*.md"), ("02-specifications/**/*.md"),
     ("03-codemap/**/*.md"), ("04-decisions/**/*.md"), ("00-glossary.md")
   - Для каждого файла:
     - прочитать YAML-фронтматтер
     - извлечь: type, slug, status, block?, scope?
     - если это cmp-* — дополнительно распарсить заголовки `### cap-*` из тела
     - если нет фронтматтера или нет type — предупреждение, пропускаем файл в индексе

3. Распределить по разделам по таблицам §1.1–§1.4:
   - type + scope (для api) → (section_number, section_title)
   - для function с block: дополнительно группировать
   - для cmp собрать caps[]

4. Для каждой группы отсортировать по slug (функции — сначала по block, потом по slug).

5. Определить marker каждой строки:
   - status == 'actual' → +
   - status == 'draft' → -
   - файл не существует (из §2.2 change.md ADDED в статусе «планируется») → ?
   (В reindex этот кейс не используется — он появляется только при создании change (evolve), когда файл ещё не создан.
    reindex всегда работает по реально существующим файлам.)

6. Для каждой строки определить comment:
   - взять comment_text из map (шаг 1) по path
   - иначе извлечь из H1 файла (первая строка `# ...` после фронтматтера)
   - иначе пусто

7. Сгенерировать итоговый markdown по шаблону §4:
   - заголовок, frontmatter (updated = сегодня)
   - passport_section (вставить как есть)
   - §2 с glossary (если есть)
   - §3, §4, §5, §6 — по таблицам с пропуском пустых разделов
   - gaps_section (вставить как есть)
   - легенду (всегда одна и та же)

8. Записать результат в 00-masterspec-index.md (полная перезапись).
```

## 6. Граничные случаи

- **Пустая фабрика** (нет ни одного артефакта): reindex всё равно выводит §1, §2 (без glossary, если его нет), разделы 3–6 полностью отсутствуют, §7 и легенда.
- **Файл без фронтматтера**: пропускается в индексе, выводится предупреждение в stderr / в отчёт reindex. Не валит процесс.
- **Дубли slug'а** (два файла с одинаковым `slug:`): reindex включает оба в индекс, но добавляет строку-предупреждение в отчёт. Исправление — через evolve (S6).
- **Файл в неправильной директории** (например, `scn-*.md` лежит в `04-decisions/`): reindex сортирует по `type:`, а не по пути. В индекс попадёт в правильный раздел, но путь будет «чужой». Это ловится скиллом `verify`.
- **Тип, которого нет в таблицах §1.1–§1.4**: файл пропускается, предупреждение в отчёт. Принятие нового типа — через расширение этого файла + `artifact-routing.md`.

## 7. Защита от потери данных

- `passport_section` и `gaps_section` сохраняются ДОСЛОВНО. Если они не найдены в старом индексе (или старого индекса нет) — в §1 и §7 вставляются шаблонные заглушки:

  §1 Паспорт:
  ```
  - Фабрика: <slug> (<описание>)
  - Владелец: <?>
  - Дата создания индекса: YYYY-MM-DD
  - Статус описания: <?>
  ```

  §7 Белые пятна:
  ```
  <Наполняется вручную. reindex сохраняет этот раздел дословно.>
  ```

- Старый файл индекса не удаляется до успешной записи нового (атомарная замена).

---

## 8. Что reindex НЕ делает

- Не создаёт новых артефактов.
- Не меняет фронтматтеры существующих артефактов.
- Не исправляет неправильные пути (файл остаётся там, где он лежал; только в индексе получит правильный раздел).
- Не добавляет/удаляет строки в §1 и §7 (кроме дат в YAML-фронтматтере индекса).
