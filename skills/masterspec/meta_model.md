# Мета-модель описания фабрики

Черновой стандарт для описания фабрики в виде набора markdown-артефактов.

Документ покрывает **три уровня**:
- **слой требований** — описание фабрики на уровне АС/ФП и её функций;
- **слой спецификаций** — описание фабрики на уровне компонентов и их кооперации;
- **слой кодовой базы** — маппинг спецификаций на конкретную реализацию в коде.

---

## 1. Назначение

Эта мета-модель нужна, чтобы:
- разложить фабрику на устойчивый набор артефактов;
- сделать устройство системы обозримым для аналитика, разработчика и агента;
- дать единый формат хранения знаний, который можно постепенно заполнять и поддерживать в актуальном состоянии;
- позволить агенту быстро навигировать от требования к коду и обратно.

Ключевой принцип: **сначала фиксируется устройство фабрики как объекта управления, потом поверх него описываются изменения** (изменения отслеживаются через git).

---

## 2. Объекты мета-модели и дисциплина слоёв

### 2.1. Слой требований — «что и зачем»

Отвечает на вопросы, ВНЕ зависимости от реализации:
- что за фабрика / АС / ФП перед нами;
- какие функции она выполняет (use-case: актор → система → результат);
- какие нефункциональные требования на неё наложены;
- как функции связаны между собой;
- какие внешние правила и ограничения действуют;
- какова концептуальная модель данных предметной области.

**Запрещено в этом слое:**
- имена компонентов, сервисов, модулей;
- имена таблиц, коллекций, индексов, полей;
- языки, фреймворки, библиотеки, СУБД, брокеры, протоколы (кроме случаев, когда конкретная технология является внешним бизнес-ограничением — например, регуляторное требование);
- классы, методы, файлы, URL, эндпоинты;
- ссылки вниз на slug’и слоёв спецификаций и кода.

**Разрешено:** имена внешних систем/акторов (они вне фабрики), бизнес-термины из глоссария, измеримые величины.

### 2.2. Слой спецификаций — «как устроено взаимодействие»

Отвечает на вопросы:
- из каких компонентов состоит фабрика;
- какую функциональность даёт каждый компонент;
- в каком порядке компоненты кооперируются при выполнении функции;
- какими алгоритмами реализуются правила работы;
- какие API (внутренние и внешние) и какие данные обеспечивают работу фабрики;
- каков профиль нагрузки в разрезе сценариев.

**Запрещено в этом слое:**
- код, псевдокод, имена классов/методов/функций;
- имена файлов, пути, пакеты, namespace;
- конкретные таблицы, колонки, типы СУБД, миграции, ORM-классы;
- упоминания конкретных технологий (если только это не фиксируется в ADR как принятое решение);
- ссылки вниз на slug’и кодового слоя (`cmap-`, `trace-`, `dmap-`).

**Разрешено:** компоненты описываются логически (ответственность, границы, интерфейсы); данные — как логические сущности с атрибутами; API — как контракты (логические операции, вход/выход, ошибки, SLA).

### 2.3. Слой кодовой базы (LLD) — «где в коде»

Отвечает на вопросы:
- в каких репозиториях и на каком стеке реализована фабрика;
- какой компонент из спецификации где живёт в коде (пакеты, классы, entry points);
- какой сценарий какой цепочкой вызовов реализуется;
- какие концептуальные сущности в каких таблицах/коллекциях хранятся.

**Ключевые свойства слоя:**
- **Генерируемый** — создаётся и обновляется агентом. Человек ревьюит, но не ведёт вручную.
- **Эфемерный** — устаревает с каждым коммитом, должен быть дешёвым в регенерации.
- **Навигационный** — это индекс, а не повторное описание кода. Каждая ссылка — `file:line` или `file:symbol`.
- **Связанный вверх** — каждый code unit ссылается на slug из спецификации.

### 2.4. Направление ссылок (снизу вверх)

Это твёрдое правило. Ссылки между слоями идут ТОЛЬКО снизу вверх:

```
код         →   спецификации   →   требования
(cmap, trace,     (cmp, cfn, scn,      (as, fn, nfr,
 dmap)             alg, api, data,      rules, context,
                   cd, lp)              fd, cdm)
```

- Артефакт требований НЕ ссылается на спецификации и код.
- Артефакт спецификаций ссылается вверх на требования (и в пределах своего слоя).
- Артефакт кода ссылается вверх на спецификации (и на требования при необходимости).

Внутри одного слоя ссылки могут идти в любом направлении по смыслу.

### 2.5. Что где живёт

| Содержимое | Слой | Артефакт |
|---|---|---|
| Use-case (актор → система → результат) | требования | `fn-` |
| Критерии приёмки | требования | `fn-` |
| Измеримые SLA/SLO/perf | требования | `nfr-` |
| Бизнес- и инженерные правила | требования | `rules-` |
| Внешние системы и потоки | требования | `context-` |
| Связи между функциями | требования | `fd-` |
| Сущности предметной области | требования | `cdm-` |
| Ключевые справочники/классификаторы (малые, стабильные) | требования | `dict-` |
| Приёмочный тест (чёрный ящик под AC функции) | требования | `tc-acc-` |
| Ответственность компонента, границы | спецификации | `cmp-` |
| **Возможности (capabilities) компонента** | спецификации | разделы `cap-*` внутри `cmp-` |
| **Порядок вызова компонентов/возможностей** | спецификации | `scn-` |
| Пошаговая логика / ветвления / инварианты | спецификации | `alg-` |
| Контракты взаимодействия | спецификации | `api-` |
| Логические сущности данных | спецификации | `data-` |
| Профиль нагрузки | спецификации | `lp-` |
| Интеграционный тест (под сценарий) | спецификации | `tc-int-` |
| Конкретные файлы, строки, классы, таблицы | код | `cmap-`, `trace-`, `dmap-` |

### 2.6. Сквозные артефакты

Не привязаны к конкретному слою:
- **глоссарий** — единый словарь терминов предметной области;
- **решения (ADR)** — архитектурные решения с обоснованием выбора.

## 3. Базовая иерархия хранения

```text
masterspec/
  00-masterspec-index.md
  00-glossary.md
  01-requirements/
    01-system/
    02-functions/
      <block>/                  # группировка по функц. блокам (опционально)
    03-nfr/
    04-rules/
    05-landscape/
    06-data-model/
    07-dictionaries/
    08-test-cases/
  02-specifications/
    01-components/              # внутри каждого cmp- — раздел «Возможности» (cap-*)
    02-scenarios/
    03-algorithms/
    04-apis/
      internal/
      external/
    05-data/
    06-diagrams/
    07-load-profiles/
    08-test-cases/
    09-ui-views/
  03-codemap/
    00-repo-map.md
    01-component-maps/
    02-scenario-traces/
    03-data-maps/
  04-decisions/
```

### Правила

- `00-masterspec-index.md` — карта фабрики: что уже описано, чего ещё не хватает.
- `00-glossary.md` — единый словарь терминов.
- В `01-requirements/` лежат артефакты слоя требований (уровень АС/ФП).
- В `02-specifications/` лежат артефакты слоя спецификаций (уровень компонентов). Функциональность компонента (возможности) не выделяется в отдельные файлы — это разделы внутри `cmp-`.
- В `03-codemap/` лежат артефакты слоя кодовой базы — генерируются агентом, не ведутся вручную.
- В `04-decisions/` лежат ADR.
- Функции АС/ФП при большом количестве группируются по функциональным блокам в подпапках.
- API разделены на `internal/` и `external/`.
- Спецификации API и схемы данных могут быть в `.md`, `.yaml` или `.json`.
- Тест-кейсы: приёмочные — в слое требований (`tc-acc-`), интеграционные — в слое спецификаций (`tc-int-`). Юнит-тесты не выделяются в артефакты.
- Изменения (AS-IS / TO-BE / diff) отслеживаются средствами git.

## 4. Соглашения по оформлению

### 4.1. YAML-фронтматтер

Каждый артефакт начинается с YAML-фронтматтера:

```yaml
---
type: component
slug: cmp-<slug>
factory: <factory-slug>
status: draft            # draft / actual / deprecated
owner: <team or person>
updated: YYYY-MM-DD
---
```

Артефакты слоя кодовой базы дополнительно содержат `generated: true`.

Инструкции по заполнению встроены в шаблон в виде HTML-комментариев `<!-- ... -->`. **При заполнении комментарии удаляются.**

### 4.2. Формат ссылок

Связи указываются по slug: `-> slug`.

Ссылки идут **снизу вверх** (см. 2.4). В разделе «Связи» артефакта перечисляются только slug’и вышестоящих слоёв и своего слоя.

Префиксы slug:

| Префикс | Слой | Тип артефакта |
|---|---|---|
| `as-` | требования | паспорт АС/ФП |
| `fn-` | требования | функция |
| `nfr-` | требования | НФТ |
| `rules-` | требования | правила |
| `context-` | требования | диаграмма окружения |
| `fd-` | требования | функциональная диаграмма |
| `cdm-` | требования | концептуальная модель данных |
| `dict-` | требования | справочник / классификатор |
| `tc-acc-` | требования | приёмочный тест |
| `cmp-` | спецификации | компонент (внутри — возможности `cap-*`) |
| `scn-` | спецификации | сценарий |
| `alg-` | спецификации | алгоритм |
| `cd-` | спецификации | компонентная диаграмма |
| `nav-` | спецификации | карта навигации между ui-view |
| `api-` | спецификации | API |
| `data-` | спецификации | схема данных |
| `lp-` | спецификации | профиль нагрузки |
| `tc-int-` | спецификации | интеграционный тест |
| `ui-view-` | спецификации | экран / карточка / форма UI |
| `cmap-` | кодовая база | component map |
| `trace-` | кодовая база | scenario trace |
| `dmap-` | кодовая база | data map |
| `adr-` | сквозной | решение |

Возможности компонента (`cap-*`) — не отдельный файл, а разделы внутри `cmp-`. Ссылка из сценария: `-> cmp-<slug>/cap-<name>`.

### 4.3. Ссылки на код

Только в слое кодовой базы:

```
path/to/file.kt:42              # конкретная строка
path/to/file.kt:ClassName       # класс/объект
path/to/file.kt:method()        # метод/функция
```

В слоях требований и спецификаций таких ссылок быть не должно.

Для обобщения типов code units используется поле `kind`:

| Группа | Значения kind |
|---|---|
| Структурные | `class`, `interface`, `struct`, `enum`, `trait`, `module`, `package` |
| Поведенческие | `function`, `method`, `handler`, `coroutine`, `job`, `listener`, `middleware` |
| Данные | `table`, `view`, `index`, `stored-proc`, `migration`, `collection`, `cache` |
| Интеграция | `endpoint`, `topic`, `queue`, `event`, `config` |

## 5. Карта отношений между артефактами

Читай как «А описан на основе Б»; стрелка `→` означает направление ссылки (снизу вверх).

```text
Требования
  АС/ФП (as)
    имеет функции (fn, внутри — use-case)
    ограничивается НФТ (nfr)
    подчиняется правилам (rules)
    имеет окружение (context) и функц. связи (fd)
    имеет концептуальную модель данных (cdm)
    содержит справочники (dict, опираются на cdm)
  Приёмочные тесты (tc-acc) → fn, dict, cdm

Спецификации → ссылаются вверх на требования
  Компонент (cmp) → fn, as
    внутри — возможности (cap-*): каждая → используется какими fn
  Сценарий (scn) → fn  (внутри: порядок вызова cmp/cap)
  Алгоритм (alg) → scn, cmp, rules
  API (api) → cmp  (внутри: контракт)
  Схема данных (data) → cdm
  Профиль нагрузки (lp) → nfr, scn
  Компонентная диаграмма (cd) → cmp
  Интеграционные тесты (tc-int) → scn, cmp

Код → ссылается вверх на спецификации
  Repo map (repo-map) → cmp (через cmap)
  Component map (cmap) → cmp, cap-*, scn
  Scenario trace (trace) → scn
  Data map (dmap) → cdm, data

Сквозные
  Глоссарий — определяет термины для всего
  ADR — обосновывает развилки в любом слое
```

## 6. Онтология артефактов

### 6.1. Сквозные артефакты

#### 6.1.1. Глоссарий

- **Имя файла:** `00-glossary.md`
- **Шаблон:** [`templates/tpl-glossary.md`](templates/tpl-glossary.md)
- **Содержание:** таблица терминов предметной области с определениями и указанием типичных ошибок смешения («Не путать с»).

#### 6.1.2. Решения: ADR и DR

Решение хранится там, где живёт то, к чему оно относится, и столько же. Область влияния решения задаёт его место и долговечность.

**ADR — архитектурное решение (сквозное, долгое).**
- **Имя файла:** `04-decisions/adr-NNN-<slug>.md`
- **Шаблон:** [`templates/tpl-adr.md`](templates/tpl-adr.md)
- **Содержание:** контекст, решение, альтернативы, последствия. Для значимых развилок устройства системы, затрагивающих несколько артефактов.

**DR — решение по конкретному артефакту (локальное).**
- **Имя файла:** `dr-<артефакт-slug>-<тема>.md` — ОТДЕЛЬНЫЙ файл рядом с артефактом, на ЕГО слое (решение о функции — в `01-requirements/`, о контракте — в `02-specifications/`). Не раздел внутри артефакта: артефакт описывает truth текущего состояния, dr — обоснование «почему так», и грузится только когда нужно именно обоснование.
- **Шаблон:** [`templates/tpl-dr.md`](templates/tpl-dr.md)
- **Содержание:** контекст, варианты, выбор, обоснование, последствия (облегчённая форма ADR).
- **Связи:** `-> slug` своего артефакта; фронтматтер `decision-scope: <тема>` (например `privacy`, `storage`) — для индекса-среза «все решения по теме» без дублирования содержания.
- **Когда:** решение по функции, по варианту реализации, по развилке узла-решения — то, что касается одного артефакта и не дотягивает до архитектурного.

**Статусная модель решений** (ЕДИНЫЙ источник; отлична от артефактной `draft`/`actual`/`deprecated`):
- **ADR:** `proposed` (предложено) → `accepted` (принято) → `deprecated` / `superseded`.
- **DR:** `accepted` (локальное решение фиксируется сразу, фазы предложения нет) → `superseded`.
- В change через `new/`: новый `adr-` создаётся со `status: proposed`; новый `dr-` — со `status: accepted`. При `apply-change` статус решений НЕ меняется (в отличие от артефактов слоёв, переходящих в `actual`).
- **Адресация `dr-`:** целевой путь (на слое артефакта-владельца) указывается в §2.2 change.md явно — `apply-change` берёт его оттуда, владельца не вычисляет; поле `about:` в `dr-` — для индекса-среза «решения по артефакту».

**Решения хода работы** (почему генератор выбрал X, что добирали дозапросом) — в `route-run` прогона, эфемерны, в фабрику не переносятся.

### 6.2. Слой требований

#### 6.2.1. Паспорт АС/ФП

- **Имя файла:** `01-requirements/01-system/as-<slug>.md`
- **Шаблон:** [`templates/tpl-passport-as.md`](templates/tpl-passport-as.md)
- **Содержание:** назначение, границы, потребители, что умеет / не умеет, внешние взаимодействия, ограничения.
- **Запрещено:** имена компонентов, технологий, баз.

#### 6.2.2. Функция АС/ФП

- **Имя файла:** `01-requirements/02-functions/fn-<slug>.md` или `01-requirements/02-functions/<block>/fn-<slug>.md`
- **Шаблон:** [`templates/tpl-function-as.md`](templates/tpl-function-as.md)
- **Содержание:** use-case функции: цель, инициатор, триггер, предусловия, основной поток в терминах «актор — система», альтернативные и исключительные потоки, результат, постусловия, критерии приёмки.
- **Запрещено:** порядок вызова внутренних компонентов (это — в `scn-`), имена компонентов, классов, полей.

#### 6.2.3. Нефункциональные требования

- **Имя файла:** `01-requirements/03-nfr/nfr-<scope>.md`
- **Шаблон:** [`templates/tpl-nfr.md`](templates/tpl-nfr.md)
- **Содержание:** измеримые требования (performance, SLA/SLO, безопасность, отказоустойчивость, трассируемость, ограничения по данным). Каждое — с ID, числом и ссылкой на функцию.
- **Поле `immutable: true|false`** во фронтматтере фиксирует, является ли порог внешним обязательством (контракт/SLA перед потребителем, регуляторное требование) — `true` — или внутренним целевым показателем, который можно пересмотреть — `false`. Дефолт при отсутствии поля — `false`. Поле читается валидаторами требований для разграничения конфликта (нарушение immutable-НФТ блокирует изменение) и эволюции фабрики.
- **Запрещено:** названия конкретных технологий, компонентов. Если ограничение на техстек действительно продиктовано извне — это `rules-`, не `nfr-`.

#### 6.2.4. Бизнес- и инженерные правила

- **Имя файла:** `01-requirements/04-rules/rules-<domain>.md`
- **Шаблон:** [`templates/tpl-rules.md`](templates/tpl-rules.md)
- **Содержание:** правила, ограничения и принципы принятия решений. Формулируются абстрактно.
- **Поле `immutable: true|false`** во фронтматтере фиксирует, пересматривается ли правило фабрикой. `true` — правило приходит извне (регуляторка, контракт, ключевой инвариант домена) и не может быть изменено изнутри; требование, его нарушающее, — блок (CONFLICT). `false` — обычное правило, установленное продуктом/командой; противоречащее требование — повод для эволюции (EVOLUTION). Дефолт при отсутствии поля — `false`.
- **Запрещено:** имена компонентов, таблиц, классов.

#### 6.2.5. Диаграмма окружения

- **Имя файла:** `01-requirements/05-landscape/context-<factory-slug>.md`
- **Шаблон:** [`templates/tpl-context-diagram.md`](templates/tpl-context-diagram.md)
- **Содержание:** внешние системы, пользователи, направления потоков. Фабрика — чёрный ящик.

#### 6.2.6. Функциональная диаграмма

- **Имя файла:** `01-requirements/05-landscape/fd-<factory-slug>.md`
- **Шаблон:** [`templates/tpl-func-diagram.md`](templates/tpl-func-diagram.md)
- **Содержание:** связи между функциями АС/ФП (какая передаёт управление какой). Только `fn-`.

#### 6.2.7. Концептуальная модель данных

- **Имя файла:** `01-requirements/06-data-model/cdm-<domain>.md`
- **Шаблон:** [`templates/tpl-conceptual-data-model.md`](templates/tpl-conceptual-data-model.md)
- **Содержание:** сущности, атрибуты, связи — без привязки к реализации. Для каждой сущности с жизненным циклом — **обязательный раздел «Состояния и переходы»**: матрица «состояние × событие → результат/ЗАПРЕЩЕНО», без пустых ячеек (пустая ячейка = дыра, ловится `verify` ось 4).
- **Запрещено:** таблицы, колонки, СУБД-типы, индексы.

#### 6.2.8. Справочник / классификатор

- **Имя файла:** `01-requirements/07-dictionaries/dict-<slug>.md`
- **Шаблон:** [`templates/tpl-dictionary.md`](templates/tpl-dictionary.md)
- **Содержание:** название, источник структуры (ссылка на сущность в `cdm-`), таблица значений, режим ведения, инварианты, связи с функциями.
- **Когда заводить:** справочник определён предметной областью, содержит немного значений и меняется редко. Большие/динамические классификаторы сюда не попадают — их объём показывается в `lp-`, структура — в `data-`, конкретное хранение — в `dmap-`.
- **Запрещено:** имена таблиц/колонок БД, СУБД-типы.

#### 6.2.9. Приёмочный тест

- **Имя файла:** `01-requirements/08-test-cases/tc-acc-<slug>.md`
- **Шаблон:** [`templates/tpl-test-acceptance.md`](templates/tpl-test-acceptance.md)
- **Содержание:** чёрный ящик под один или несколько AC функции: предусловия, вход, шаги актор/система, ожидаемый результат, постусловия.
- **Запрещено:** упоминание компонентов, классов, технологий, таблиц.

### 6.3. Слой спецификаций

#### 6.3.1. Компонент (и его возможности)

- **Имя файла:** `02-specifications/01-components/cmp-<slug>.md`
- **Шаблон:** [`templates/tpl-component.md`](templates/tpl-component.md)
- **Содержание:** назначение, ответственность, границы, **раздел «Возможности» (`cap-*`)** — стабильный набор способностей компонента с описанием входа/выхода и ссылками на функции, где каждая используется; зависимости.
- **Возможности (`cap-*`):** не выносятся в отдельные файлы. Каждая — раздел внутри `cmp-`. На возможности ссылаются сценарии и component map через нотацию `cmp-<slug>/cap-<name>`.
- **Запрещено:** код, классы, файлы, пути, конкретные технологии.

#### 6.3.2. Сценарий

- **Имя файла:** `02-specifications/02-scenarios/scn-<slug>.md`
- **Шаблон:** [`templates/tpl-scenario.md`](templates/tpl-scenario.md)
- **Содержание:** **порядок вызова возможностей компонентов** для реализации функции. Каждый шаг: `-> cmp-<slug>/cap-<name>`. Это единственное место, где фиксируется последовательность.
- **Запрещено:** код, классы, пути, имена полей.

#### 6.3.3. Алгоритм

- **Имя файла:** `02-specifications/03-algorithms/alg-<slug>.md`
- **Шаблон:** [`templates/tpl-algorithm.md`](templates/tpl-algorithm.md)

#### 6.3.4. Спецификация API (внутренний / внешний)

- **Имя файла:** `02-specifications/04-apis/internal/api-<slug>.md|yaml|json` или `external/...`
- **Шаблон:** [`templates/tpl-api.md`](templates/tpl-api.md); для `.yaml`/`.json` — OpenAPI / AsyncAPI.
- **Содержание:** контракт (логические операции, вход/выход, ошибки, ограничения, SLA, идемпотентность).
- **Запрещено в .md-спецификации:** конкретные пути файлов, имена классов-обработчиков.

#### 6.3.5. Схема данных

## 8. Практические правила ведения

1. Один файл — один объект описания.
2. Имя файла стабильное и читабельное.
3. Каждый артефакт начинается с YAML-фронтматтера.
4. Связи — в формате `-> slug`. Направление — снизу вверх (см. 2.4).
5. HTML-комментарии из шаблона — инструкции; при заполнении удаляются.
6. Если знание извлечено из репозитория — пометить источник.
7. Если знание неполное — «Открытые вопросы» или `status: draft`.
8. Диаграмма без текстового файла — недостаточный артефакт.
9. Описание пригодно и для человека, и для агента.
10. `.yaml`/`.json` для API и данных — в индексе указывается фактическое расширение.
11. Изменения отслеживаются через git.
12. Codemap генерируется агентом, не правится вручную.
13. **Минимальный объём.** Не дублируй содержимое между артефактами — ставь ссылку.

## 9. MVP для старта

Минимальный полезный набор:
- `00-masterspec-index.md`, `00-glossary.md`;
- 1 паспорт АС/ФП;
- список функций с критериями приёмки;
- 1 диаграмма окружения;
- 1 концептуальная модель данных;
- ключевые справочники (`dict-`), если предметка на них опирается;
- 3–7 ключевых компонентов — с возможностями внутри;
- 2–5 ключевых сценариев;
- 2–5 ключевых алгоритмов;
- 1 компонентная диаграмма;
- 1 схема данных по главному домену;
- 1–3 API на критичные взаимодействия;
- 1 профиль нагрузки;
- приёмочные тесты под ключевые AC и интеграционные — под ключевые сценарии (по желанию на MVP).

После спецификаций — генерация кодового слоя: 1 repo map, component maps и scenario traces для ключевых, 1 data map для главного домена.
