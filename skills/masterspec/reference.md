# Примеры использования Skills

В этом файле собраны конкретные кейсы применения навыка `f-desc`.

---

## 🧩 Пример 1: Проектирование новой фабрики с нуля (режим design)

**Навык:** `f-desc`
**Запрос пользователя:**
> Спроектируй фабрику уведомлений: принимает запросы из разных систем, доставляет сообщения через SMS/push/email, умеет повторять попытки при сбоях.

**Что произошло:**
- Шаг 1: Скилл определил режим `design` и создал индекс фабрики из `templates/tpl-masterspec-index.md`, зафиксировав скоуп и владельца.
- Шаг 2: На слое требований заполнены паспорт АС (`as-notification-factory`), функции (`fn-send-notification`, `fn-check-delivery-status`) с use-case-структурой, НФТ, правила доставки, диаграмма окружения (C4-context) и концептуальная модель данных.
- Шаг 3: На слое спецификаций выделены компоненты (`cmp-api-gateway`, `cmp-notification-router`, `cmp-template-engine`, `cmp-channel-adapter-sms`) с возможностями `cap-*`, описаны сценарии со ссылками `-> cmp-<slug>/cap-<name>`, алгоритм выбора канала, внутренние и внешние API, схема данных.
- Шаг 4: Добавлены приёмочные тесты (`tc-acc-*`) и интеграционные (`tc-int-*`), зафиксированы ADR по ключевым решениям.

**Результат:**
Получено полное непротиворечивое описание фабрики с bottom-up трассировкой. Разработчики получили спецификации без путаницы с требованиями, тестировщики — два уровня тестов с чёткими границами, архитектор — ADR по спорным вопросам. Экономия на ревью ≈ 40% за счёт отсутствия дублирования между артефактами.

## 🧩 Пример 2: Восстановление описания из разрозненных документов (режим recover)

**Навык:** `f-desc`
**Запрос пользователя:**
> У нас есть ТЗ в Word, презентация архитектуры и переписка в почте по фабрике расчёта лимитов. Собери описание по мета-модели, помечай пробелы.

**Что произошло:**
- Шаг 1: Скилл прочитал исходные материалы, извлёк сущности и разложил их по слоям: функциональность → требования, компоненты → спецификации, имена таблиц/сервисов → пометил для codemap.
- Шаг 2: Создал артефакты из шаблонов, заполнил только то, что подтверждено источником. Непокрытые области помечены статусом `?` в индексе и в разделе «Белые пятна».
- Шаг 3: Для каждого пробела сгенерирован конкретный вопрос («не определён SLA по времени расчёта», «нет контракта с сервисом скоринга»).

**Результат:**
Команда получила структурированную базу знаний вместо разрозненных документов плюс список из 12 открытых вопросов для интервью со стейкхолдерами. Время на подготовку к архкомитету сократилось с недели до двух дней.

## 🧩 Пример 3: Генерация codemap по репозиторию (режим codemap)

**Навык:** `f-desc`
**Запрос пользователя:**
> Есть описание фабрики нотификаций в спецификациях. Проанализируй репо и свяжи компоненты/сценарии/схемы данных с кодом.

**Что произошло:**
- Шаг 1: Скилл прочитал `02-specifications/`, собрал список компонентов с их `cap-*`, сценариев и доменов данных.
- Шаг 2: Выполнил grep/glob по репозиторию, нашёл модули, соответствующие компонентам, и сопоставил `cap-<name>` с конкретными классами/функциями по `file:line`.
- Шаг 3: Сгенерировал `03-codemap/00-repo-map.md`, `cmap-*.md` (маппинг capabilities → код), `trace-*.md` (цепочка вызовов сценария), `dmap-*.md` (сущности → таблицы/ORM). Все файлы помечены `generated: true`.

**Результат:**
Разработчик, впервые открывший репозиторий, за 20 минут понял, какой сценарий какими классами обслуживается. При рефакторинге compass по codemap показал, что изменение затронет `cap-route-by-channel`, а значит — функцию `fn-send-notification`.

## 🧩 Пример 4: Аудит полноты описания (режим audit)

**Навык:** `f-desc`
**Запрос пользователя:**
> Проверь описание фабрики платежей на полноту и дисциплину слоёв.

**Что произошло:**
- Шаг 1: Скилл прошёл по чек-листу из `SKILL.md` §2 (9 пунктов: frontmatter, bottom-up-ссылки, чистота слоёв, покрытие функций сценариями, наличие capabilities и т.д.).
- Шаг 2: Обнаружил нарушения: в `fn-process-payment.md` упомянут класс `PaymentProcessor` (утечка слоя кода в требования); сценарий `scn-refund` не имеет привязки к `fn-*`; два компонента без capabilities.
- Шаг 3: Выдал отчёт с файлами, номерами строк и предлагаемыми правками, плюс обновил «Белые пятна» в индексе фабрики.

**Результат:**
Команда получила конкретный план доработок на 4 часа вместо абстрактного «нужно улучшить документацию». Архитектор утвердил описание после первой итерации правок.

## 🧩 Пример 5: Reverse engineering существующей системы (режим reverse)

**Навык:** `f-desc`
**Запрос пользователя:**
> Есть рабочий монолит без документации. Восстанови описание по коду.

**Что произошло:**
- Шаг 1: Скилл сначала построил codemap снизу — прошёл по модулям, классам, таблицам, API и собрал черновые компоненты с их capabilities.
- Шаг 2: По трассам вызовов (controllers → services → repositories) восстановил сценарии; по сигнатурам API — функции (`fn-*`); по таблицам — концептуальную модель данных.
- Шаг 3: Сформулировал вопросы к бизнес-владельцу для подтверждения функций и НФТ (то, что нельзя увидеть из кода: SLA, приоритеты, бизнес-правила).

**Результат:**
Получен черновик описания на всех трёх слоях за один день работы, 80% которого подтверждено без вопросов. Позволило команде начать рефакторинг с пониманием границ компонентов вместо действий «на ощупь».
