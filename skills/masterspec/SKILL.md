# masterspec — kernel-скилл описания фабрик

Ты помогаешь описывать фабрики (автоматизированные системы) по мета-модели masterspec. Все справочные файлы лежат рядом в директории скилла.

Для **изменений** в уже описанной фабрике этот скилл не вызывается — см. раздел 8 «Change workflow».

---

## 1. Главный принцип: три слоя

Описание фабрики строится в три слоя. Каждый отвечает на свой вопрос:

| Слой | Вопрос | Что описывает |
|---|---|---|
| **Требования** (`01-requirements/`) | ЧТО даёт система и как себя ведёт | Полезность, внешнее поведение, ограничения |
| **Спецификации** (`02-specifications/`) | КАК устроено взаимодействие частей | Компоненты, их кооперация, контракты |
| **Кодовая база** (`03-codemap/`) | ГДЕ в коде это реализовано | Файлы, классы, строки, таблицы |

**Ссылки идут снизу вверх.** Жёсткие запреты по слоям, направление ссылок и список что где живёт — в `references/layer-discipline.md`. Читай его перед первым Write/Edit и на каждом ревью.

Полная мета-модель (расширенная версия, с обоснованием и примерами) — `meta_model.md`.

## 3. Чек-лист перед сохранением любого артефакта

Перед `Write` проверь по порядку:

1. **Слой определён.** Куда кладу — `01-requirements/`, `02-specifications/`, `03-codemap/` или `04-decisions/`? Справка — `references/artifact-routing.md`.
2. **Содержимое соответствует слою.** Применил запреты из `references/layer-discipline.md`.
3. **Ссылки идут снизу вверх.** Из этого файла не указаны slug'и нижележащих слоёв.
4. **Фронтматтер заполнен.** `type`, `slug`, `factory`, `status`, `updated` (+ `generated: true` для codemap).
5. **Имя файла = slug.**
6. **Все HTML-комментарии `<!-- ... -->` удалены** (это были инструкции шаблона, не содержимое).
7. **Нет дублирования** с уже созданными артефактами. Если возникает желание повторить — поставь ссылку.
8. **«Открытые вопросы»** проставлены там, где знание неполное.
9. **Индекс обновлён** (`00-masterspec-index.md`). В режиме `design`/`recover`/`codemap`/`reverse` — допускается ручное добавление строк по шаблону из `references/index-canonical.md`. В остальных потоках (`apply-change`, регулярная сверка) предпочтительно запустить режим `reindex` — он перегенерирует §3–§6 полностью по файлам фабрики.

Если любой пункт не выполнен — не сохраняй, доделай.

## 4. Режимы работы

Определи режим по аргументу `$0` или по запросу пользователя. Если неясно — спроси.

### 4.1. design — проектирование новой фабрики

Пользователь описывает бизнес-задачу, ты раскладываешь её в артефакты.

1. Прочитай `meta_model.md` (разделы 2–3) и `references/layer-discipline.md`.
2. Создай `00-masterspec-index.md` (шаблон: `templates/tpl-masterspec-index.md`, пример: `examples/00-masterspec-index.md`).
3. **Слой требований** (в такой последовательности):
   1. паспорт (`as-`) — одной фразой «что это и где границы»;
   2. функции (`fn-`) — по одной на каждую полезность, с use-case внутри;
   3. НФТ (`nfr-`) — измеримые требования, привязанные к функциям;
   4. правила (`rules-`) — бизнес- и инженерные ограничения;
   5. диаграмма окружения (`context-`) и функциональная диаграмма (`fd-`);
   6. концептуальная модель данных (`cdm-`) — сущности домена;
   7. справочники (`dict-`) — только ключевые, малые, стабильные;
   8. приёмочные тесты (`tc-acc-`) — под критерии приёмки функций.
4. **Слой спецификаций** (только после того, как требования согласованы):
   1. компоненты (`cmp-`) — внутри каждого раздел «Возможности» (`cap-*`), перечисляющий стабильные способности компонента;
   2. сценарии (`scn-`) — порядок вызова возможностей компонентов для реализации функции (ссылки `-> cmp-<slug>/cap-<name>`);
   3. алгоритмы (`alg-`) — пошаговая логика внутри сценария/компонента;
   4. API (`api-`), данные (`data-`), компонентная диаграмма (`cd-`), профиль нагрузки (`lp-`);
   5. интеграционные тесты (`tc-int-`) — под сценарии.
5. **ADR** (`adr-`) — при каждой архитектурной развилке, где был реальный выбор.
6. **Глоссарий** — пополняй при каждом новом термине.
7. **Индекс** обновляй после создания каждого артефакта.

Для каждого артефакта: прочитай `templates/tpl-*.md`, скопируй, заполни. HTML-комментарии — инструкции; **удаляй после заполнения**.

### 4.2. recover — восстановление из документов

Пользователь даёт Confluence, PDF, Swagger, презентации.

1. Прочитай все источники, составь карту: фрагмент → тип артефакта и слой.
2. Создай индекс-скелет.
3. Создавай артефакты от требований к спецификациям. В каждом укажи источник.
4. Всё неоднозначное — в «Открытые вопросы» со статусом `draft`.
5. Собери глоссарий.

**Не домысливай.** Неоднозначность — зафиксируй и спроси.

### 4.3. codemap — анализ репозитория

Пользователь даёт репозиторий, ты генерируешь слой кодовой базы.

1. Изучи структуру: языки, фреймворки, модули.
2. Создай `00-repo-map.md`.
3. Определи компоненты, создай `cmap-*.md`.
4. Создай `trace-*.md` по сценариям.
5. Создай `dmap-*.md` по entity/миграциям.
6. Обнови индекс.

Все codemap-артефакты: `generated: true`. Указывай реальные `file:line`.

### 4.4. audit — аудит покрытия

1. Прочитай `00-masterspec-index.md`.
2. Проверь: файлы существуют? фронтматтер корректен? разделы заполнены?
3. **Проверь дисциплину слоёв** (`references/layer-discipline.md`): в требованиях нет спецификационной лексики; в спецификациях нет кода.
4. Проверь связи: каждый `-> slug` ведёт на существующий артефакт? ссылки идут снизу вверх?
5. Найди «висящие» артефакты.
6. Сформируй отчёт, предложи порядок доработки.

### 4.5. reverse — reverse engineering из кода

1. Выполни шаги codemap.
2. На основе кода предложи артефакты спецификаций и требований.
3. Все артефакты — `status: draft`, «Открытые вопросы» = «Восстановлено из кода».
4. Создай индекс.

### 4.6. reindex — перегенерация `00-masterspec-index.md`

Полностью перезаписывает разделы §3–§6 индекса по реальному содержимому фабрики. Разделы §1 «Паспорт» и §7 «Белые пятна и открытые вопросы» сохраняются дословно. Режим не меняет артефакты фабрики — только индекс.

Когда запускать:
- автоматически — `masterspec-apply-change` вызывает его на §8 своего пайплайна;
- вручную — когда индекс вручную дрейфанул (пустые секции, перепутанная нумерация, расхождения с файлами);
- после больших `codemap`/`reverse`-прогонов, где создано много артефактов сразу.

Алгоритм:

1. Прочитай `references/index-canonical.md` — полную спецификацию формата и алгоритма.
2. Прочитай текущий `00-masterspec-index.md` (если есть). Сохрани §1, §7 и словарь `path → comment` из строк артефактов.
3. Просканируй `00-glossary.md` и все `.md` внутри `01-requirements/`, `02-specifications/`, `03-codemap/`, `04-decisions/`. Для каждого прочитай YAML-фронтматтер: `type`, `slug`, `status`, `block?`, `scope?`. Для `cmp-*` дополнительно распарсь заголовки `### cap-<name> — <описание>`.
4. Распредели по секциям из `index-canonical.md` §1. Функции (`fn-*`) сгруппируй по `block:`. Для API раздели на §4.4/§4.5 по `scope:`.
5. Внутри каждой группы отсортируй по `slug` (функции — сначала по `block`, потом по `slug`).
6. Для каждой строки:
   - marker: `+` (actual), `-` (draft);
   - comment: из сохранённого словаря (если был), иначе — H1-заголовок целевого файла.
7. Сгенерируй итоговый markdown по шаблону §4 из `index-canonical.md`. Пустые секции пропусти (но номера остальных не сдвигай — §4.9 всегда «Интеграционные тесты», даже если §4.7 пуста).
8. Перезапиши `00-masterspec-index.md`, проставь `updated:` = сегодня.
9. Отчёт: список файлов без фронтматтера (пропущенные), дубли `slug:` (если есть), типы, не известные `index-canonical.md`.

Не вызывает: добавление/изменение артефактов, создание новых файлов, правку фронтматтеров.

## 5. Ключевые правила ведения

Кратко; полная версия — `meta_model.md`, раздел 8.

1. **Один файл = один объект.**
2. **Имя файла = slug.**
3. **YAML-фронтматтер обязателен.** Codemap: + `generated: true`.
4. **Связи по slug:** `-> slug`. Ссылки на код допустимы только в codemap-слое, формат `file:line`.
5. **Неполное знание не прячется.** `status: draft` + «Открытые вопросы».
6. **Всегда обновляй индекс.** Формат — `references/index-canonical.md`. Ручное редактирование допустимо, но авторитетен режим `reindex` (перегенерация §3–§6 по файлам).
7. **Новый термин = запись в глоссарий.**
8. **Изменения через change-workflow** (см. §8), не через папки as-is/to-be.
9. **Минимальный объём.** Не повторяй одно и то же в разных артефактах — ставь ссылку.

## 6. Взаимодействие с пользователем

- **Итеративно.** Начни с MVP (см. `meta_model.md`, раздел 9), покажи, получи обратную связь.
- **При проектировании** задавай конкретные вопросы: «Какие внешние потребители?», «Какой SLA?». Не переходи к спецификациям, пока требования не согласованы.
- **При восстановлении** показывай, что извлёк и откуда, перед оформлением.
- **При анализе кода** сначала покажи repo map, потом генерируй детали.
- **Противоречия** — в «Открытые вопросы», спроси пользователя.

## 8. Change workflow (как вносить изменения)

**Не редактируй артефакты фабрики напрямую.** Любое изменение оформляется через change-скиллы с ревью. Этот скилл (kernel) редактирует артефакты только в режимах `design`/`recover`/`codemap`/`reverse` — когда артефактов либо ещё нет, либо они целиком генерируются из кода.

### 8.1. Карта скиллов

| Скилл | Когда запускать |
|---|---|
| `masterspec` (этот) | Проектирование новой фабрики с нуля; recover/codemap/audit/reverse |
| `masterspec-explore` | Structured research кодовой базы параллельными read-only субагентами (target=factory-spec или factory-change) |
| `masterspec-propose` | Создать `change.md` — предложение на изменение артефактов фабрики |
| `masterspec-design` | Создать `design.md` + `tasks.md` — план реализации change в коде (dev-design, не архитектурный) |
| `masterspec-implement` | Выполнить `tasks.md` с верификацией (сборка/тесты/линтер) |
| `masterspec-apply-change` | Влить `change.md` + файлы из `new/` в артефакты фабрики (мультиартефактный мерж) |
| `masterspec-archive-change` | Переместить завершённый change в `changes/archive/YYYY-MM-DD-<name>/` |

### 8.2. Жизненный цикл изменения

```
masterspec (design) ─┐
                      ▼
                   propose → (design?) → implement → apply-change → archive-change
                      ▲                                   │
                      └──────── explore (read-only) ──────┘
```

1. Нет ни одной фабрики в `masterspec/` → `masterspec` в режиме `design`.
2. Нужно изменить артефакты уже описанной фабрики → `masterspec-propose` (создаёт `changes/<name>/change.md` со статусом `На согласовании`).
3. Ревью и согласование → merge PR, статус вручную меняется на `Согласовано`.
4. Сложный CR (затрагивает код) → `masterspec-design` создаёт `design.md` + `tasks.md`. Для простого CR (правка одного AC, текстовая доработка) — шаг пропускается.
5. Реализация кода → `masterspec-implement` выполняет задачи из `tasks.md`, ставит статус `В реализации`, по завершении запускает верификацию.
6. Влитие в артефакты фабрики → `masterspec-apply-change` мержит `change.md` (diff-блоки) и файлы из `new/` в соответствующие директории фабрики, ставит статус `Реализовано`.
7. Архивация → `masterspec-archive-change` перемещает change в `archive/YYYY-MM-DD-<name>/`.

### 8.3. Статусы `change.md`

| Статус | Кто ставит | Когда |
|---|---|---|
| На согласовании | `masterspec-propose` | Change создан, идёт ревью в PR |
| Согласовано | Аналитик вручную | PR с change.md вмержен, change одобрен |
| В реализации | `masterspec-implement` | Первый запуск implement, идёт кодинг по tasks.md |
| Реализовано | `masterspec-apply-change` | Изменения вмержены в артефакты фабрики |
| Архивировано | `masterspec-archive-change` | Change перемещён в `archive/` из статуса `Реализовано`. При архивации с warning (другой статус) — физически перемещено, но статус в шапке сохранён для аудита |

Обратные переходы (например, возврат из `В реализации` в `Согласовано` для правок) — только вручную.

### 8.4. Правила change-workflow

- **Артефакты фабрики (`masterspec/01-*/02-*/03-*/04-*`) редактируются только через `apply-change`.** Ручные правки вне режимов kernel-скилла — запрещены.
- **Откат `apply-change` — через git**: `git checkout HEAD -- masterspec/`. Никаких `.bak`-файлов.
- **Формат изменений гибридный**: мелкие правки — diff-блоки в `change.md`; крупные (новый артефакт целиком) — файлы в `changes/<name>/new/<slug>.md`. Выбор — см. `references/change-conventions.md` и `masterspec-propose/references/change-format.md`.
- **Верификация в `implement` обязательна** — запуск сборки/тестов/линтера перед объявлением реализации завершённой.
- **Маппинг триггеров**: «добавь/поменяй/убери в фабрике X» → `propose`. «Опиши фабрику Y с нуля» → `masterspec` в режиме `design`.
