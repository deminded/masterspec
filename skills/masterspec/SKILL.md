# masterspec — kernel-справочник

Я — СПРАВОЧНИК мета-модели. Я не «делаю» (не порождаю, не правлю, не проверяю) — я держу знание, которое читают скиллы-операции и человек. Всё «делающее» — отдельные скиллы-глаголы (см. §4).

> Принцип набора: kernel = знание; операции = скиллы-глаголы; вариации одного назначения = ПАРАМЕТР скилла, а не новый скилл.

---

## 1. Главный принцип: три слоя

| Слой | Вопрос | Что описывает |
|---|---|---|
| **Требования** (`01-requirements/`) | ЧТО даёт система и как себя ведёт | Полезность, внешнее поведение, ограничения |
| **Спецификации** (`02-specifications/`) | КАК устроено взаимодействие частей | Компоненты, кооперация, контракты |
| **Кодовая база** (`03-codemap/`) | ГДЕ в коде реализовано | Файлы, классы, строки, таблицы |

**Ссылки идут снизу вверх.** Жёсткие запреты по слоям и направление ссылок — `references/layer-discipline.md`. Полная мета-модель — `meta_model.md`.

## 3. Чек-лист перед сохранением артефакта слоя

Применяется к артефактам слоёв (`01-`/`02-`/`03-`/`04-`). Служебные файлы (`route-run-<ts>.md`, `change.md`) живут вне слоёв и имеют свой формат — этот чек-лист к ним не применяется. Перед `Write`:

1. **Слой определён** — `01-requirements/` / `02-specifications/` / `03-codemap/` / `04-decisions/`. Справка — `references/artifact-routing.md`.
2. **Содержимое соответствует слою** — запреты из `references/layer-discipline.md`.
3. **Ссылки идут снизу вверх** — нет slug'ов нижних слоёв.
4. **Фронтматтер заполнен** — `type`, `slug`, `factory`, `status`, `updated` (+ `generated: true` для codemap). Исключение: singleton-файлы с фиксированным именем (`00-masterspec-index.md`, `00-glossary.md`, `03-codemap/00-repo-map.md`) поле `slug` НЕ несут. Артефакты слоёв: `status` ∈ `draft`/`actual`/`deprecated`. Решения ведут свой цикл (ADR: `proposed`→`accepted`→`deprecated`/`superseded`; DR: `accepted`→`superseded`) — единый источник статусной модели `meta_model.md §6.1.2`.
5. **Имя файла = slug.**
6. **HTML-комментарии шаблона удалены.**
7. **Single-source** — нет дублирования с другими артефактами; повтор → ссылка. Один факт живёт в одном месте.
8. **«Состояния и переходы»** заполнены для cdm-/data- (матрица состояние×событие без пустых ячеек).
9. **Значимый выбор записан** — если был реальный выбор из ≥2 вариантов, есть decision record: `adr-` (сквозной, в `04-decisions/`) или `dr-` (локальный, рядом с артефактом на его слое). Выбор без записи = немое решение.
10. **«Открытые вопросы»** там, где знание неполное.
11. **Индекс обновлён** — `00-masterspec-index.md` приведён в соответствие ПОЛНОЙ перегенерацией по `references/index-canonical.md` (единственный способ; точечные правки строк §3–6 не применяются).

Любой пункт не выполнен — не сохраняй, доделай.

## 4. Карта скиллов-операций (что когда запускать)

kernel сам не запускается «в режиме» — выбери скилл-глагол:

| Скилл | Вызов | Назначение |
|---|---|---|
| `explore` | 👤 запуск, 🤖 субагенты внутри | разведка кодовой базы под фабрику/зону изменения; собирает изолированный фокус-набор. Нужен только при наличии кода |
| `derive layer=req\|spec` | 👤 | породить слой: из бизнес-запроса → требования (req), из требований → спецификации (spec). Оркестратор слоя; при отсутствии фабрики инициализирует её |
| `evolve entry=req\|rule\|ext` | 👤 | изменить описанную фабрику (правка узла ИЛИ добавление узла); точка входа — требование / правило / внешний контракт. Сам заводит change |
| `gen type=<артефакт>` | 🤖 (вызывает derive/evolve) | сгенерировать ОДИН артефакт (fn/cmp/scn/api/data/…), 1 субагент со своей ролью и изолир. контекстом |
| `verify scope=req\|spec\|change` | 🤖 (→ 👤 на гейт) | вычитка по осям O1–O5 (+O0/O6/O7 для spec); конфиги-данные |
| `recover source=docs\|code` | 👤 | восстановить описание фабрики из документов или кода |
| `apply-change` | 👤 (после merge PR) | влить change в артефакты фабрики + обновить индекс |
| `archive-change` | 👤 | переместить завершённый change в archive/ |

**Фокус-набор** — изолированный срез: бизнес-описание ровно одного элемента + релевантный срез вышестоящих артефактов (а не вся фабрика). Его готовит оркестратор (в `context=full`) или planner-субагент (в `context=lean` — `references/patterns/context-isolation.md`) и подаёт в `gen`, чтобы субагент не тонул в лишнем контексте.

**Создание change** — это шаг 0 внутри `evolve` (отдельного скилла нет). Формат change.md — `references/change-format.md`, шаблон — `templates/tpl-change.md`. Устаревший `propose` помечен `deprecated` и в карту не входит.

**Граница набора:** реализация в коде (`impl-plan`, `implement`) — отдельный набор скиллов кодинга, вне этого справочника.

## 5. Ключевые правила ведения

1. **Один файл = один объект.**
2. **Имя файла = slug.**
3. **YAML-фронтматтер обязателен** (codemap: + `generated: true`).
4. **Связи по slug:** `-> slug`, снизу вверх. Ссылки на код — только в codemap, формат `file:line`.
5. **Неполное знание не прячется** — `status: draft` + «Открытые вопросы».
6. **Изменения — через `evolve` → `apply-change`, не правкой артефактов напрямую** (кроме генерации с нуля через `derive`/`recover`).
7. **Новый термин = запись в глоссарий.**
8. **Single-source** — не повторяй одно в разных артефактах, ставь ссылку.

## 6. Паттерны процесса (references, общие для скиллов-операций)

Не скиллы — несущие принципы, которые скиллы-операции применяют. Лежат в `references/patterns/`:

- **`decision-node.md`** — узел-РЕШЕНИЕ: где есть реальная развилка, не исполняй, а дай дивергенцию вариантов → выбор → decision record (ADR). Парный к узлу-ИСПОЛНЕНИЮ.
- **`element-workflow.md`** — узел-ИСПОЛНЕНИЕ: планировщик → исполнитель(и) → ПРИЁМЩИК (сводит редакции в одну, single-source, на месте).
- **`enforcement.md`** — гейты как мандаты, не советы: hard-gate блокирует переход; «Согласовано» ставит человек (merge PR), не агент. Жёсткость дозируется.
- **`verification-axes.md`** — 5 осей вычитки (предмет verify): статика (элемент/слой/межслой) · динамика (прогон содержания + прогон годности для следующего слоя) · негатив (состязательный прогон).
- **`context-isolation.md`** — изоляция контекста оркестратора (режим `context=lean`): оркестратор = диспетчер по файлам (держит план + пути + сводки, не содержимое); тяжёлое чтение делегируется planner/gen/verify-субагентам через файлы. Для моделей с ограниченным контекстом.

## 7. Структура фабрики в проекте пользователя

```
masterspec/
├── 00-masterspec-index.md
├── 00-glossary.md
├── 01-requirements/    # 01-system/ 02-functions/ 03-nfr/ 04-rules/ 05-landscape/ 06-data-model/ 07-dictionaries/ 08-test-cases/
├── 02-specifications/  # 01-components/ 02-scenarios/ 03-algorithms/ 04-apis/{internal,external} 05-data/ 06-diagrams/ 07-load-profiles/ 08-test-cases/ 09-ui-views/
├── 03-codemap/         # 00-repo-map.md 01-component-maps/ 02-scenario-traces/ 03-data-maps/  (generated: true)
├── 04-decisions/       # adr- — сквозные решения (локальные dr-* лежат рядом с артефактами на их слоях)
├── changes/
│   ├── <name>/         # активный change: change.md, .research/, new/, route-run-<ts>.md
│   └── archive/        # завершённые change (sibling активных, НЕ вложен в <name>/)
│       └── YYYY-MM-DD-<name>/
└── .work/              # эфемерная рабочая зона режима context=lean: plan-<layer>.md, .focus/, verify-report (вне боевого дерева, как .research/)
```

Полная вложенная раскладка с подпапками и правилами — `meta_model.md §3` (единственный источник структуры; здесь — обзор).
