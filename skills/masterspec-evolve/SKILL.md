# masterspec-evolve — изменить фабрику

Точечность (не перелопачивать всё) и полнота (не забыть правок) одновременно — оба полюса закрывает точный граф влияния. Читай мета-модель и дисциплину слоёв в kernel (`../masterspec/`), паттерны — в `../masterspec/references/patterns/`.

## Точка входа (любой узел любого слоя)
- `entry=req` — корень в слое требований (новая/изменённая функция или правило). Распространение вниз.
- `entry=rule` — корень = конкретное правило; обратные рёбра ИМЕННО его (фильтр по ребру, не весь блок правил).
- `entry=ext` — корень = листовой внешний контракт. Распространение вверх по исходящим рёбрам (кто вызывает + требование-родитель). «Требования не меняются» — гипотеза к проверке, не факт.

## Два режима (определяются корнем)
- **ПРАВКА** — узел уже существует (`--root=<slug>`). Зона = обход по ОБРАТНЫМ рёбрам (ссылки идут снизу вверх, изменение бежит против них: правишь узел — затронуты ссылающиеся на него).
- **ДОБАВЛЕНИЕ** — нового узла ещё нет (`--root=new:<имя>`). После создания узла — forward-каскад ВНИЗ: породить обязательные нижележащие артефакты (новый атрибут сущности → правки fn/api/data, что его выводят; новая fn → scn/api/data/tc-acc; новое требование → элементы нижних слоёв). Это под-derive с корнем на новом узле.

## Метод
0. **Заведи change.** Создай `changes/<name>/` и `change.md` по **каноническому 8-секционному формату** (`../masterspec/references/change-conventions.md`; детальный формат секций и diff-блоков — `../masterspec/references/change-format.md`, шаблон — `../masterspec/templates/tpl-change.md`) — именно этот формат потом парсит `apply-change`, отступать нельзя. В change.md: что меняем, зачем, якоря скоупа (корневые slug'и и затронутые пути). `new/` внутри — карантин черновиков новых/правленых артефактов. Отдельного скилла для создания change нет — это делает evolve.
1. **Корень и режим** — узел приложения (`--root`); определи ПРАВКА или ДОБАВЛЕНИЕ.
2. **Зона.**
   - ПРАВКА → impact-зона по обратным рёбрам ссылок.
   - ДОБАВЛЕНИЕ → forward-набор обязательных нижележащих артефактов (см. состав слоёв в `../masterspec/meta_model.md`); создаются через `gen`.
3. **Контекст.** Если у фабрики есть код — `explore target=factory-change` по anchor (зона/новый узел). Если кода нет — `explore` не нужен: обратный/прямой индекс ссылок строй `Grep` по `-> ` в артефактах, держи его явно, не в уме.
4. **Точечные правки/создание (по формату change).** Правки СУЩЕСТВУЮЩИХ артефактов оформляются как **diff-блоки в `change.md` §4** (типы `modify-bullet`/`replace-section`/`add-subsection`; формат — `../masterspec/references/change-format.md`). НОВЫЕ артефакты — отдельными файлами в `changes/<name>/new/<slug>.md`. Именно эти две формы потом читает `apply-change` — больше он ниоткуда не берёт. Развилку с реальным выбором — в узел-решение (`patterns/decision-node.md`), не молча.
5. **Приёмщик** (`patterns/element-workflow.md`) — зону писали несколько исполнителей → сведи редакции в одну (single-source) на месте.
6. **Полнота каскада:**
   - **каскад-вниз-к-верификации:** новое/изменённое правило или альтернативный поток порождает приёмочный критерий (правка/добавление/подъём → AC);
   - **обе стороны контракта:** вход + ответ-успех + ответ-отказ, порождённый изменением;
   - **forward-полнота (для ДОБАВЛЕНИЯ):** каждый обязательный нижележащий артефакт порождён (нет узла без необходимого потомка);
   - **scope-fence:** каждому соседу по графу — явный вердикт «не затронут, потому что…»;
   - **правило немого вердикта:** узел на обратном ребре без правки И без вердикта = непройден;
   - **правило немого подъёма:** бизнес-эффект, поднятый флагом, но без материализации каскада вверх (правка верхнего узла + scope-fence + AC) = непройден. Подъём вверх = под-каскад с корнем на верхнем слое.
7. **Проверка-вверх (entry=ext и побочные эффекты):** техническое (не поднимается) vs бизнес-значимое (поднимается в требования). Критерий — «меняет ли узел доменный ИНВАРИАНТ», а не «на каком уровне стека выглядит». Разрешай развилку операциональным тестом из базы (есть ли источник нового значения, нарушается ли инвариант); вердикт условный, если зависит от допущения. Любое новое обязательное значение в контракте обязано указать узел-владелец источника.
8. **route-run** — `changes/<name>/route-run-<ts>.md` (шаблон `../masterspec/templates/tpl-route-run.md`): вход · режим · зона/набор · правки · scope-fence · подъёмы · открытые развилки · метрики (точность: тронуто/каскад; полнота: забытое; немые вердикты/подъёмы).
9. **Вычитка** — `verify scope=change`.
10. **Hard-gate** (`patterns/enforcement.md`) — выход = черновик; применение в фабрику только человеком: ревью + merge PR → `apply-change`.

## Выход
- `changes/<name>/change.md` (правки существующих артефактов = diff-блоки §4) + `new/` (новые артефакты) — черновик + scope-fence (что не тронуто и почему);
- `route-run-<ts>.md` с отчуждаемыми метриками;
- открытые развилки (узлы-решения) и условные вердикты подъёма — человеку.
