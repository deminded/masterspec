# masterspec-evolve — изменить фабрику

Точечное изменение: не перелопачивать всё (точность), но и не забыть правок (полнота). Оба полюса закрывает точный граф влияния. Читай мета-модель и дисциплину слоёв в kernel (`../masterspec/`), паттерны — в `../masterspec/references/patterns/`.

## Точка входа (любой узел любого слоя)
- `entry=req` — корень в слое требований (новая функция, изменение правила). Распространение вниз по обратным рёбрам.
- `entry=rule` — корень = конкретное правило; обратные рёбра ИМЕННО его (фильтр по ребру, не весь блок правил).
- `entry=ext` — корень = листовой внешний контракт. Распространение вверх по исходящим рёбрам (кто вызывает + требование-родитель). «Требования не меняются» — это гипотеза к проверке, не факт.

## Метод
1. **Корень** — узел приложения изменения (`--root`).
2. **Impact-зона** — обход по ОБРАТНЫМ рёбрам ссылок (ссылки в модели идут снизу вверх, изменение бежит против них: правишь узел — затронуты ссылающиеся на него). Используй `explore target=factory-change` по anchor. Обратный индекс зависимостей держи явно, не в уме.
3. **Точечные правки** — только в зоне, в карантин `changes/<name>/new/`. Развилку с реальным выбором — в узел-решение (`patterns/decision-node.md`), не молча.
4. **Приёмщик** (`patterns/element-workflow.md`) — если зону писали несколько исполнителей, сведи редакции в одну (single-source) на месте.
5. **Полнота каскада:**
   - **каскад-вниз-к-верификации:** новое/изменённое правило или альтернативный поток порождает приёмочный критерий (правка ИЛИ подъём → AC);
   - **обе стороны контракта:** вход + ответ-успех + ответ-отказ, порождённый изменением;
   - **scope-fence:** каждому соседу по графу — явный вердикт «не затронут, потому что…»;
   - **правило немого вердикта:** узел на обратном ребре без правки И без вердикта = непройден;
   - **правило немого подъёма:** бизнес-эффект, поднятый флагом, но без материализации каскада вверх (правка верхнего узла + scope-fence + AC) = непройден. Подъём вверх = под-каскад с корнем на верхнем слое.
6. **Проверка-вверх (entry=ext и побочные эффекты):** техническое (не поднимается) vs бизнес-значимое (поднимается в требования). Критерий — «меняет ли узел доменный ИНВАРИАНТ», а не «на каком уровне стека выглядит». Разрешай развилку операциональным тестом из базы (есть ли источник нового значения, нарушается ли инвариант); вердикт условный, если зависит от допущения. Любое новое обязательное значение в контракте обязано указать узел-владелец источника.
7. **route-run** — `changes/<name>/route-run-<ts>.md`: вход · зона · правки · scope-fence · подъёмы · открытые развилки · метрики (точность: тронуто/каскад; полнота: забытое; немые вердикты/подъёмы).
8. **Вычитка** — `verify scope=change`.
9. **Hard-gate** (`patterns/enforcement.md`) — выход = черновик; применение в фабрику только человеком: ревью + merge PR → `apply`.

## Выход
- `changes/<name>/new/` — правки (черновик) + scope-fence (что не тронуто и почему);
- `route-run-<ts>.md` с отчуждаемыми метриками;
- открытые развилки (узлы-решения) и условные вердикты подъёма — человеку.
