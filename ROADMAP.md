### 🔴 S2. `apply-change` не блокируется при отсутствии git

**Где**: [masterspec-apply-change/SKILL.md](masterspec-apply-change/SKILL.md) §4 + [merge-workflow.md](masterspec-apply-change/references/merge-workflow.md) §1.1.

**Проблема**: когда фабрика лежит в директории без `.git/`, `git status --porcelain masterspec/` возвращает `fatal: not a git repository` и выходит с успешным кодом. Текущая логика трактует «пустой вывод» как «чисто» и пропускает. Safety net «откат через git» в этом сценарии отсутствует.

**Следствие**: в таком проекте `apply-change` необратим — любая ошибка в diff-блоках ломает фабрику без возможности восстановления.

**Предложение**:
- Перед `git status` проверять `git rev-parse --is-inside-work-tree`. Если ответ не `true` — блок с сообщением «фабрика не в git-репозитории; либо инициализируй git, либо явно согласись на работу без отката через флаг `--no-git-rollback` в аргументе скилла».
- Добавить `--no-git-rollback` как явный opt-in для smoke-тестов и документации (но с предупреждением в выводе).
- Тест: повторить прогон №1 в директории без `.git/` и убедиться, что без флага скилл выходит.

**Найдено в**: прогон №1, шаг 2 «предпроверка git» — `fatal: not a git repository` был проигнорирован.

### ✅ S3. Финальная валидация «файл реально получил правку» — ЗАКРЫТО

**Решение**: §9 в [merge-workflow.md](masterspec-apply-change/references/merge-workflow.md) разбит на три подраздела:

- **§9.1 Smoke-check** — прежние 4 структурных проверки (число файлов, ADDED в индексе, REMOVED нет в дереве, нет обратных ссылок). Провал → откат через `git checkout HEAD -- masterspec/`.
- **§9.2 Verification** — для каждой строки §2.1/§2.2/§2.3 change.md отдельный алгоритм по типу правки: `modify-bullet` проверяется по первой + последней непустой строке `ПОСЛЕ:` (компромиссный порог, устойчив к нормализации whitespace); `replace-section` / `add-subsection` — первая строка `ПОСЛЕ:` в границах раздела; `prepend-to-file` — голова файла совпадает с головой `ПОСЛЕ:`; ADDED — файл есть, YAML валиден, нет `<!--`; REMOVED — файла нет. Строки, пропущенные пользователем в §6-конфликте, хранятся в памяти сессии и исключаются из verification. Особый случай пустого `ПОСЛЕ:` обработан явно (§9.2.5).
- **§9.3 Вердикт** — `U == 0` → статус `Реализовано`; `U > 0` → AskUserQuestion с тремя опциями: `rollback` / `override` / `leave`. Тихий переход в `Реализовано` при `unconfirmed > 0` теперь явно помечен в [apply-change/SKILL.md Guardrails](masterspec-apply-change/SKILL.md) как баг.

[apply-change/SKILL.md](masterspec-apply-change/SKILL.md) §11 и §13 «Вывод» обновлены под трёхступенчатый отчёт `confirmed / skipped_by_user / unconfirmed`.

**Валидация на фабрике**: требуется следующий прогон. До фактического тестирования S3 остаётся «закрыто по коду, не закрыто по эмпирике» — аналог S5, случай `---`. Включить в прогон №8.

### Graphify как опциональная карта кода — дизайн v2.1 (два круга Sol-ревью 16.07, PR #20)
Контекст: пакет ssot-tools сберовских коллег (разбор 16.07) поверх публичного graphify (MIT, tree-sitter, 36 языков, локально без LLM). Пилоты 16.07: доменная математика переносима (mailtg-bridge cohesion 0.89; site — 3 домена 0.91–0.97, режет по функции, не по каталогам); скрипт коллег берём как рецепт математики, не код (JVM-гейт искусственный; расхождение THRESHOLD код↔дока).

**Два инварианта дизайна.**
1. **Опциональность:** нет graphify / нет графа → каждый скилл работает по-старому без деградации функции. Граф не коммитится (gitignore), строится локально `graphify update .`.
2. **Parity: граф меняет стоимость и порядок, но не полноту.** Операционально — два класса потребителей с разными оракулами:
   - *gate-потребители* (coverage, verify, все expected-set'ы): графу ЗАПРЕЩЕНО участвовать — expected строится только из спеки; parity абсолютен по построению;
   - *discovery-потребители* (sync стадия 1, codemap hint, шардирование explore): гарантия полноты лежит ТОЛЬКО на детерминированном ядре без графа (локаторный маппинг; нарезка по каталогам; существующий tool-приоритет); граф добавляет кандидатов/подсказки СВЕРХ ядра, каждая помечена источником и проходит тот же verify/gate.
   Parity-оракулы (машинные): (а) детерминированное ядро on/off идентично bit-for-bit; (б) explore: объединение шардов = срезу владельца (set-diff пуст) в обоих режимах; (в) codemap: expected-set gate зелёный в обоих режимах. Граф = discovery/cache, НЕ источник истины: запись в артефакт всегда подтверждается чтением кода; наведение/верификация остаются за Serena/LSP/чтением.

#### Нормативный блок 1 — versioned graph adapter + матрица режимов
- Единственная точка чтения графа — `masterspec/scripts/graph-adapter.py` (stdlib); скиллы graph.json напрямую не парсят. Контракт входа версионирован: `schema_version`, обязательные поля узла (id, source_file, file_type) и ребра (source, target, relation); неизвестная версия / отсутствующие поля → отказ адаптера, не догадка.
- Исполнимый контракт адаптера (фиксируется тактом 0 ДО потребителей): операции CLI — `check` (валидация graph+manifest → статус), `meta` (манифест + coverage), `slice --root <r>`, `neighbors --of <locator> [--depth k]`, `edges --relation <list>`; выход — JSON на stdout `{schema_version, status: ok|fallback|error, reason?, data}`; exit codes: 0 = ok, 2 = fallback-состояние (auto-потребитель работает по-старому и печатает причину), 3 = жёсткая ошибка (режим on); диагностика — JSON-строками в stderr. Разрешение узлов: адаптер отдаёт готовые локаторы v2 (source_file + symbol из label) — id графа потребитель не видит. Манифест пишет НАШ build-wrapper `graph-build.sh` (вызывает graphify per root, затем пишет манифест).
- Freshness/trust-манифест `graph.meta.json`: graphify_version, schema_version, revisions: [{root, commit, dirty}] по-репно, roots[] (канонизированы realpath, отсортированы), excludes[], digest (sha256 graph.json), files_seen/files_parsed по расширениям. parsed < seen (mixed-language) → `partial: true` — потребители знают, что граф видит не всё.
- Матрица режимов `graph=off|auto|on` — единая семантика у всех потребителей:
  - `off` — граф не читается, даже если существует;
  - `auto` (дефолт) — ЛЮБОЕ отклонение (нет бинаря/графа, unknown schema, digest mismatch, stale: built_at_commit ≠ HEAD, dirty — флаг сборки ИЛИ живой `git status` корня на момент чтения, partial для потребителя полноты: отклонение только при заявленном требовании полноты, иначе — предупреждение в выводе) → заметный fallback: работа по-старому + обязательная строка `graph: fallback (<причина>)` в выводе и route-run. Средозависимой тишины нет: результат тот же (parity), видимость деградации обязательна;
  - `on` — то же отклонение → явная ошибка, стоп; переопределение только явными флагами `--allow-stale` / `--allow-dirty` / `--allow-partial`.
- Политика рёбер — confidence-first (уточнено по живому graphify 0.9.17 в такте 0): рёбра несут поле `confidence: EXTRACTED|INFERRED` — гейт доверия ведётся по НЕМУ; именной список INFERRED-классов (semantically_similar_to, conceptually_related_to, shares_data_with, part_of, rationale_for) — fallback для графов без поля; ребро без confidence с именем вне известных словарей трактуется консервативно как inferred. INFERRED по умолчанию отрезаны; включение — явный флаг, каждое такое ребро несёт пометку inferred, использовать как доказательство запрещено (навигационная подсказка). contains/method — по запросу для навигации. Словарь relation живого graphify шире пяти имён (imports_from, uses): EXTRACTED-факты с незнакомыми именами наружу ПРОХОДЯТ (это код-факты); именной allowlist остаётся у потребителей (математика доменов, блок 3), не у гейта доверия адаптера.
- Monorepo/vendor/generated/symlink: roots[]/excludes[] фиксируются в манифесте; дефолтные excludes — vendor/, node_modules/, build/, dist/, target/ + generated-маркеры (путь `/generated/`, заголовок `@generated`); симлинк, ведущий вне roots, не разыменовывается (узел помечается external). Monorepo: граф строится на root, адаптер отдаёт срез по запрошенному root.
- **Multi-repo (дефолт playbook: N реп кода + репа аналитики рядом):** граф и манифест — НА КАЖДУЮ репу кода независимо (commit/dirty/digest свои: свежесть и stale судятся по-репно); адаптер принимает список пар (граф, манифест) по roots и адресует срез по репе. Project-root (база путей всех локаторов) фиксируется явно в машинной секции repo-map; принадлежность локатора репе — longest-prefix-матч по каноническим roots из той же секции (НЕ «первый сегмент пути»: держит вложенные checkout/worktree и кодовую репу в корне). Это же правило резолва использует ось O_C1 и стадия 1 sync (diff и HEAD у каждой репы свои). Cross-repo рёбер в графе НЕТ по природе (tree-sitter не видит сетевых вызовов и чужих пакетов) — межрепные связи живут на слое спеки (api-/scn-/context-) и в fallback-механике (grep/Serena по-репно); домены блока 3 режутся внутри одной репы, поверх реп не строятся; blast-radius sync — per-repo по графу этой репы, межрепное расширение — по спеке, не по графу.
- Негативные тесты (без них инвариант не введён): stale graph / unknown schema / dirty worktree / нет бинаря / битый digest / symlink наружу / mixed-language — в `auto` каждый даёт fallback С ПОМЕТКОЙ, в `on` — явную ошибку. Parity-тесты: фикстура + прогон потребителя on/off — множества результатов совпадают.
- Оговорка по структурной валидации (эмпирика такта 0, живой graphify 0.9.17): dangling endpoint ребра — НЕ порча схемы, а штатное поведение graphify (`imports os` без материализации узла stdlib); адаптер и build-wrapper такие рёбра сохраняют (пропуская с warning-диагностикой при выдаче), deviation не поднимают — иначе auto-режим падал бы на любом реальном коде с импортами. Фильтр excludes удаляет ребро только если его endpoint-узел был вырезан фильтром, а не когда узел никогда не существовал. Той же природы узлы БЕЗ source_file (внешние символы: builtins/stdlib — str, Path, Exception; эмпирика пилота mailtg: 17 из 1991): не порча схемы, а нелокализуемые узлы — вне локаторных выдач, с info-диагностикой, не deviation.
- Семантика «нет бинаря» в матрице: отсутствие graphify — жёсткая ошибка ПОСТРОЕНИЯ (graph-build.sh, exit 3) и отклонение для решения «строить ли граф»; ЧТЕНИЮ валидного свежего графа бинарь не нужен — адаптер сообщает `binary_available: false` (check/meta + stderr warning), deviation не поднимает: свежесть уже гейтится stale-чеком, а перестроить протухший всё равно нечем — это и всплывёт как stale.
- **Режим ПОСТРОЕНИЯ в скиллах** (решение Евгения tg-7378): `build=never|ask|auto`, дефолт `ask` (спросить пользователя перед запуском graphify; `auto` — строить молча, только при чистом worktree). Строить ТОЛЬКО по roots КОДА, переданным скиллу (playbook: roots=<репы кода>); репозиторий аналитики графом не покрывается НИКОГДА — guard: root с маркером фабрики (`masterspec/00-masterspec-index.md` в корне) отвергается с явным сообщением, даже если пользователь передал его в roots по ошибке. Негативный тест на guard обязателен (graphify экстрактит «код» из fenced-блоков md — граф по аналитике был бы мусором, тихо отравляющим discovery).

#### Нормативный блок 2 — code-locator v2 (два вида локаторов, не смешивать)
- **Provenance артефакта** (фронтматтер `provenance:`, ставит recover) — область происхождения, стабильность важнее точности: `code:<path>` | `code:<path>#<symbol>`. `:line` в provenance запрещён (дрейфует с каждым коммитом); существующие `code:<path>:line` — предмет миграции.
- **Локаторы-доказательства** в теле cmap/trace/dmap — точность важнее: `<path>#<symbol>` (определение) | `<path>#<symbol>:<line>` (строка внутри символа — анонимные/лямбды) | `<path>:<line>` (только где символа нет: конфиг, SQL, разметка — с фиксацией в белых пятнах).
- Грамматика едина и формальна (для чекера): `locator := path ['#' symbol] [':' line]`; `path` — POSIX-путь от project-root, `#`, `:` и пробел в пути запрещены; `symbol` — `[A-Za-z_$][A-Za-z0-9_$.]*` с опциональным суффиксом `()`; `line` — целое ≥ 1; пустые части невалидны. Разделитель `#`: двоеточие перегружено формами v1 (`path:42`, `path:Class`). Классификация чекером: v2 ⇔ содержит `#`; v1-line ⇔ хвост `:<цифры>`; v1-symbol ⇔ хвост `:<не-цифры>`; остальное — невалид. Такт 2a — это ПРАВКА НОРМЫ, не только чекер: meta_model §4.3 (+§2.3, §4.1 provenance) переписываются на v2 с dual-read-оговоркой на окно миграции.
- **Dual-read:** все читатели локаторов (verify O_T4-резолвинг, индексация, sync) принимают v1 И v2 на всё окно миграции.
- **Отдельный чекер** `masterspec/scripts/check-code-locator.py` (детерминированный, в ряд с check-operational-envelope): парсит все локаторы 03-codemap/ и provenance-полей, классифицирует v1/v2/невалидный; file-резолвинг (файл существует) — FAIL, symbol-резолвинг — best-effort warning через grep, НЕ через граф (чекер не зависит от опциональной зависимости — parity).
- Подтакты: **(2a)** правка нормы + грамматика + чекер dual-read (v1 на этом такте НЕ дельта — только отчётная классификация); **(2b)** writers (gen/recover/codemap) пишут v2 для новых локаторов + деприкация v1: с этого такта v1-локатор в 03-codemap/ и `code:<path>:<line>` в provenance = BLOCKER-строки детектора (чекер = детектор дельты migration-контура, migration-класс change `> Класс: migration`). #### Нормативный блок 3 — explore: домены ТОЛЬКО как шардирование внутри существующих owners
- Роли, владение коллекциями и группировка НЕ меняются (`masterspec-explore/references/research-orchestration.md §3`): домены — не новая ось декомпозиции и не новые роли. Граф применяется в одной точке: когда срез layer-role owner'а превышает потолок, шардирование ЭТОГО среза идёт по функциональным доменам из domains.json вместо эвристики «по каталогам». Выход — штатные `<role>-<batch-id>.yaml`. Fallback: текущая нарезка. Parity: состав ролей, коллекций и покрытие идентичны on/off.
- Скрипт `masterspec/scripts/graph-domains.py` (stdlib, детерминированный bit-for-bit; пилотная версия проверена 16.07). Математика — single source в `references/graph-domain-math.md`, скрипт ей следует; НЕГАТИВНЫЙ тест на расхождение код↔дока (урок коллег: THRESHOLD 0.30 в коде vs 0.15 в доке):
  - collapse узлов по source_file; префильтр по кодовым расширениям (graphify экстрактит код из fenced-блоков md); языкового гейта НЕТ;
  - relation allowlist {references, calls, imports, inherits, implements}; отрезаны {contains, method} и весь INFERRED; вес пары = Σ weight рёбер (default 1.0);
  - направление рёбер игнорируется (A→B и B→A суммируются в неупорядоченную пару, кратные рёбра суммируются); activity(a) = Σ w(a,·) по всем инцидентным парам; cosine-нормализация norm = w/√(activity_a·activity_b), округление 4 знака (бит-стабильность), нулевой знаменатель ⇒ пара отбрасывается; карантин shared: fan_in (число РАЗЛИЧНЫХ ссылающихся классов) ≥ cutoff (20) ИЛИ cross-module ≥ 50% модулей (module map — опциональный явный файл [{name, prefix}], без него действует только fan_in); кластеры = связные компоненты norm > threshold (union-find, объединение в отсортированном порядке ключей); порядок компонентов до tie-break: (size desc, first member asc); absorb компонент < min_size (3) к компоненту с максимальным суммарным coupling (tie-break: индекс asc), нулевой coupling → shared с reason=absorbed_to_shared; cohesion = internal/(internal+external), рёбра к shared вне знаменателя; коллизии domain ID — суффиксы -2, -3 в порядке (size desc, first member asc). Все пороги — явные CLI-флаги с задокументированными дефолтами.
- Контракт `domains.json`: schema_version; source {digest graph.json, built_at_commit}; params (все пороги); domains[] {id, files[], package_prefixes[], cohesion, size}; shared {files[], reason по файлу}; coverage {files_in_graph, assigned, shared, unassigned[]} — непопавший файл виден поимённо, не теряется молча. Шард получает files[] домена как owned_paths + shared как shared_readonly (только чтение, во все шарды). Lifecycle: локальный untracked-кэш (gitignore, пересоздаётся; протух digest → пересчёт) — в ОТЛИЧИЕ от codemap-слоя, который ревьюится человеком и живёт в git («эфемерность» codemap в мета-модели — про скорость устаревания, не про untracked).
- Stable domain IDs: id = kebab-slug лексикографически первого package_prefix (коллизия → суффикс -2). Детерминирован между прогонами; меняется при переименовании пакета — приемлемо для эфемерного кэша (зафиксировать выбор в dr- при такте).

#### Нормативный блок 4 — sync и coverage
- **Sync строго через change-lifecycle** (blocker ревью снят конструкцией): sync НИКОГДА не пишет в 01-*/02-*/03-* — его выход только черновики в `changes/<sync-...>/`, дальше штатный конвейер verify layer=change → human-gate (merge) → apply-change. Четыре разделённые стадии:
  1. детерминированное обнаружение кандидатов (скрипт): git diff диапазона коммитов → маппинг изменённых файлов на артефакты по локаторам v2 (cmap/trace/provenance); при графе — blast-radius по EXTRACTED-рёбрам глубины k (default 1), fallback — импортёры через grep/Serena; выход sync-candidates.json, каждый кандидат несёт источник `locator|blast-graph|blast-grep` и причину. Гарантия полноты — ТОЛЬКО на локаторном ядре (источник locator, детерминизм bit-for-bit on/off); blast-расширения любого источника — эвристические подсказки сверх ядра, в критерий полноты не входят;
  2. LLM-классификация кандидатов (затронут содержательно vs rename/refactor без смены поведения) → черновик change по change-conventions;
  3. verify layer=change (штатный);
  4. human-gate → apply-change (штатные).
- **Свежесть — своя ось** (blocker ревью: O_T5 не трогать — это соответствие tc-flt текущим scn/api, другой предмет). Новая ось `O_C1 свежесть codemap` (layer=code) с проверяемым предикатом: каждый codemap-артефакт несёт во фронтматтере `verified_revision: [{root, commit}]` по-репно (пишет генератор/verify); артефакт stale ⇔ существует его локатор, чей файл изменён или исчез по `git diff <verified_revision>..HEAD --name-only` СВОЕЙ репы (резолв репы — правило блока 1). Blast-radius в предикат НЕ входит (он для черновиков sync, не для оси). Подключение layer=code — нормативная правка verification-axes (сейчас перечисляет req|spec|change), отдельным пунктом такта coverage gate. Severity major, не hard-gate; механика подсчёта = стадия 1 sync в режиме отчёта (без черновиков).
- **Coverage сверху вниз** — expected-set из спеки с явными ключами множеств: (cmp-slug, cap-name) → строка «Маппинга возможностей» в cmap-<cmp>; (scn-slug, шаг) и (scn-slug, OE-ID) → шаг/OE-строка в trace-<scn>; (cdm-/data-сущность) → строка в dmap-<domain> — dmap в expected ОБЯЗАТЕЛЕН (§6.4.4). НЕ «каждый символ кода документирован» — противоречит норме «навигационный индекс, не инвентарь». Гейт: детерминированный set-diff expected − actual = ∅ по каждому из трёх множеств (в ряд с pre-gate покрытия verification-axes); обратное направление (анти-призрак) — резолвимость локаторов, закрыта чекером блока 2. Направление «код → где документирован» гейтом НЕ является; при графе допустим warning-отчёт о крупных незадокументированных узлах (с оговоркой partial при mixed-language).
- **Acceptance-метрики пилота** (форма фиксируется сейчас, пороги — при нарезке такта): (а) recall стадии 1 на ручной разметке диапазона коммитов mailtg-bridge + доля false-positive; (б) доля черновиков, проходящих verify layer=change без ручной доводки; (в) инвариант «ни одной записи вне changes/» — детерминированная проверка git status; (г) parity по оракулам инварианта 2: локаторное ядро стадии 1 идентично on/off bit-for-bit, артефактный состав change после verify совпадает.
