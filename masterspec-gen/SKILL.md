# masterspec-gen — сгенерировать один артефакт

Узел-исполнение (`../masterspec/references/patterns/element-workflow.md`). Я генерирую РОВНО ОДИН артефакт указанного типа в изолированном контексте — не вижу всю фабрику, только поданный фокус-набор. Чужое — только через ссылку.

## Вход (изолированный фокус-набор)
Готовит оркестратор/explore: бизнес-описание ровно этого элемента + релевантный срез вышестоящих артефактов (для fn — результаты OE-интервью+cdm+rules+context+глоссарий; для cmp — fn; для api — cmp+data+релевантные OE) + шаблон. НЕ подаётся остальная фабрика.

**Формат фокус-набора:** связный markdown — бизнес-описание элемента, процитированные (не пересказанные) срезы вышестоящих артефактов и ссылки `-> slug`, плюс какой `tpl-<type>` заполнять. Это НЕ сырой YAML-агрегат explore. Оркестратор передаёт его gen-субагенту как содержимое задачи (Task-промпт) или файлом, путь к которому даёт `from=<путь>`.

**Куда писать (`target_path`/`target_mode`):** в lean-режиме приходят из плана (`patterns/context-isolation.md §Адресация`). Если не заданы — выводи по умолчанию: путь — из `artifact-routing.md` по `type` (api — по `scope`, fn — по `block`); режим — машинный api/data (`.yaml`/`.json`) → `sidecar` (+ компаньон `.md`), новый артефакт → `file`, изменение существующего `.md` → `diff` (выдай diff-блок для `change.md §4`, не пиши файл напрямую). Пиши ровно туда и так — не «по догадке».

**Guardrails и цикл приёмки:** при `guardrails≠off` conformance-пункты активных правил (`../masterspec/references/patterns/guardrails.md`) входят в приёмочные критерии элемента; `loop=on` включает пятишаговый цикл с blind-verify и severity-гейтом (`../masterspec/references/patterns/generation-loop.md`; дефолт off). Применённые правила — в route-run («Guardrails applied»).

## Метод
1. Прочитай шаблон `../masterspec/templates/tpl-<type>.md`, скопируй структуру.
2. Заполни по мета-модели. Дисциплина слоёв — `../masterspec/references/layer-discipline.md` (запреты для слоя элемента).
3. Тип-специфика.
   - **Требования:** `as` — назначение, границы, потребители, что умеет и не умеет; `fn` —
     обязательны альтернативные/исключительные потоки и все восемь OE для external-I/O либо одна
     строка internal-only из
     `../masterspec/references/operational-envelope.md`: `APPLICABLE` с основанием и AC, `N/A` только с причиной,
     нехватка знания только `OPEN` (не домысливать); каждый критерий приёмки — проверяемый оракул
     given-when-then; `nfr` — числовая метрика результата с квалификаторами и основанием
     `-> fn-<slug>/OE-LOAD`; `rules` — формулировка абстрактная (без компонентов/таблиц), проставлен
     `immutable`; `context` — для OE-DELIVERY полный внешний путь, предпосылки/ack/silent failure;
     `cdm` — раздел «Состояния и переходы» (матрица состояние×событие без пустых ячеек, у перехода
     триггер и момент) + тип каждой связи (composition/aggregation/association/inheritance,
     бизнес-смысл); `dict` — значения и источник структуры (ссылка на сущность `cdm`); `tc-acc` —
     под конкретные AC и APPLICABLE OE, с provenance fixture, fidelity и границей наблюдения.
   - **Спецификации:** `cmp` — раздел возможностей (`cap-*`); `scn` — порядок вызова возможностей,
     каждое ветвление с ≥2 явными исходами (нет молчаливого else) + владелец воплощения каждой
     APPLICABLE OE; `api` — обе стороны контракта (вход / ответ-успех / ответ-отказ) и типы, для
     async — направление produce/consume, для внешнего OE-DELIVERY — ack stages/silent failure/
     reconciliation; `data` — состояния и логические ограничения; `alg` — пошаговая логика,
     ветвления (каждое условие с явной веткой «иначе»), инварианты; `lp` — только из OE-LOAD,
     отделяет общую популяцию от активного подмножества и задаёт бюджет работы; `ui-view` (для
     систем с UI) — экран/представление: состав, источники данных, переходы между состояниями;
     `nav` — карта навигации между `ui-view` (входы/выходы экранов); `tc-int` — ссылка на OE-маппинг
     владельца `scn-`, честная fidelity и остаточный риск без второй OE-таблицы; `tc-flt` — один
     каталог на scn, детерминированно из точек `scn→api`, без домысла результатов, со стабильной
     связью строк с `tc-int` и глубиной по criticality.
   - **Все fn/tc:** обязательна `criticality: high|medium|low`. `tc-acc`/`tc-int` следуют
     `../masterspec/references/test-case-standard.md`: повелительное действие, testData,
     верифицируемый expectedResult; для ошибок — отдельная проверка логов + `OE-EVIDENCE`.
4. Ссылки `-> slug` только вверх/в слое.

## Дозапрос (≤2 раунда)
Не хватает фокус-набора (упомянута сущность вне поданного среза, неясное правило) — НЕ выдумывай. Запрос «не хватает X» идёт оркестратору, который маршрутизирует по единому контракту (`../masterspec/references/patterns/context-isolation.md §Дозапрос`): нужен факт из КОДА → `explore`; нужен срез ДОКУМЕНТА/артефакта (в lean) → `planner` дописывает `.focus`; нет ни там, ни там → открытый вопрос человеку. Лимит 2, потом «Открытый вопрос».

Отдельный случай — НЕКОРРЕКТНЫЙ структурный вход (поданный машинный вход не соответствует ожидаемой схеме: битый sidecar/IR, нарушенная структура фокус-набора): это НЕ дозапрос (добирать нечего) и не повод домыслить корректную форму — gen возвращает единичный `BLOCKER: вход не соответствует схеме <X>`. Чинит вход тот, кто его подал, не gen.

## Самопроверка перед отдачей
Прогон своего артефакта по оси 1 (`patterns/verification-axes.md`): пустой обязательный пункт →
доделать или пометить «Открытый вопрос». Для `fn` сначала сверить внешний инициатор/канал: external-I/O
имеет точное множество восьми OE и четыре поля каждой грани; internal-only — одну N/A-строку без
скелета. `N/A` без причины запрещено. Это НЕ замена verify слоя — лишь элементная гигиена.

**Evidence-трассировка (forward-антигаллюцинация).** Каждое непустое содержательное утверждение артефакта должно иметь основание в фокус-наборе (требование/вышестоящий срез/бизнес-описание). Заполнено правдоподобно, но БЕЗ основания в поданном входе — это дефект (выдумка), а не наполнение: либо найди основание (при нехватке — дозапрос §Дозапрос), либо пометь «Открытый вопрос». «Звучит разумно» ≠ «следует из требований». Это самопроверка автора (artifact-internal), не независимый контроль.

**Структурная валидность вывода (детерминированно, до отдачи).** Форма вывода проверяется дешёвой сверкой, не на глаз: машинный вывод (sidecar `.yaml`/`.json`) парсится без ошибок и НЕ содержит markdown-ограждений (```); структура артефакта соответствует `tpl-<type>` (обязательные разделы на месте и не переименованы). Это проверка ФОРМЫ — отдельно от evidence (содержание) и осей verify (слой); ловит структурно-битый вывод до того, как он уйдёт дальше.

## Выход
Зависит от `target_mode`:
- `file` (дефолт для нового) → `<type>-<slug>.md` (`status: draft`);
- `diff` (изменение существующего `.md`) → diff-блок для `change.md §4` (файл напрямую НЕ пишется);
- `sidecar` (машинный api/data) → пара: компаньон `<slug>.md` + машинный сайдкар. Формат — по природе контракта (OpenAPI/AsyncAPI для api, JSON Schema для data, расширяемо WSDL/protobuf/…): `../masterspec/references/patterns/sidecar-formats.md`. Компаньон несёт `sidecar_format:`/`sidecar:` во фронтматтере.
Плюс список дозапросов и открытых вопросов → оркестратору в route-run.

## Развилка внутри генерации
Если по ходу встретился реальный ВЫБОР (несколько жизнеспособных вариантов модели/контракта/алгоритма) — не решай молча. Эскалируй оркестратору как развилку: он развернёт узел-решение (`../masterspec/references/patterns/decision-node.md`) и зафиксирует decision record. Молчаливый выбор = немое решение, потом его переоткроют.
