# migration-rules — правила переразложения формы (legacy → schema-first)

Справочник скилла `masterspec-migrate`. **Legacy-форма** — артефакт в старой форме: `api`/`data` с
контрактом/схемой в теле `.md` без машинного сайдкара; `scn`/`alg` без атрибута `notation`/`form`.
**Целевая форма** — schema-first (api/data → сайдкар) и нотационная (scn → `notation`, alg → `form`).
«Legacy»/«целевая» — про ФОРМУ артефакта, не про номер `meta_model_version` (версию ставит
сертификация по состоянию, не migrate).

Содержит: таблицы маппинга фронтматтера, правила пометки MIGRATE-TODO, конвертацию логических типов,
источники неоднозначности, определение `notation` (§9) и `form` (§10) по фактическому содержимому.

| `slug` | `slug` | без изменений |

| `factory` | `factory` | без изменений |
| `status: actual` | `status: draft` | downgrade обязателен (есть MIGRATE-TODO → human-gate) |
| `status: draft\|deprecated` | без изменений | |
| `owner`, `updated` | без изменений | |
| *(отсутствовало)* | `sidecar_format` по ПРИРОДЕ | `openapi-3.1` (sync HTTP/REST) \| `asyncapi-2.x` (event-driven) \| иной стандарт (protobuf/WSDL/SDL) под транспорт; транспорт БЕЗ готового стандарта (MTProto RPC и т.п.) — сайдкар НЕ создавать, `# MIGRATE-TODO: стандарт под транспорт не определён`. `patterns/sidecar-formats.md`, «нотацию не выдумывай» |
| *(отсутствовало)* | `sidecar: <slug>.<ext>` | имя машинного файла (если сайдкар создаётся) |
| *(отсутствовало)* | `produced_by: migrate` | проставить |
| `provenance` (recover-поле, легаси-блок) | `provenance: # MIGRATE-TODO: ...` (скаляр) | канон — СКАЛЯР `provenance: docs:<…> \| code:<path>` (`../../masterspec/meta_model.md §4.1`), не блок; только если исходник несёт намёк на происхождение (потребляемый внешний контракт, `scope: external`) — migrate НЕ реконструирует источник, ставит `provenance: # MIGRATE-TODO: provenance (скаляр docs:/code:) уточнить — legacy: "<цитата>"` |

> **boundary НЕ проставляется.** Реестр boundary (intra/inter/perimeter) и security-guardrails —
> отдельный такт (Фаза 5). До него migrate не заводит `boundary` ни значением, ни MIGRATE-TODO.

### Тело api

| Раздел legacy | Куда |
|---|---|
| «Операции / события» | сайдкар (operationId + summary) |
| «Контракт» (вход/выход/ошибки) | сайдкар (schemas) |

Остаётся в компаньоне `.md` (схемой не выражается): назначение; маппинги полей (YAML-блок, §4);
ограничения (SLA/идемпотентность/ретраи); связи.

## 2. Фронтматтер data

| Поле legacy | Целевая форма | Правило |
|---|---|---|
| `type: data-schema` | `type: data-schema` | без изменений |
| `slug`, `factory`, `owner`, `updated` | без изменений | |
| `status: actual` | `status: draft` | downgrade обязателен |
| `status: draft\|deprecated` | без изменений | |
| *(отсутствовало)* | `sidecar_format: json-schema-2020-12` | добавить |
| *(отсутствовало)* | `sidecar: <slug>.schema.json` | добавить |
| *(отсутствовало)* | `produced_by: migrate` | проставить |

### Тело data

| Раздел legacy | Куда |
|---|---|
| «Сущности» | сайдкар (`$defs`) |
| «Ключевые атрибуты» | сайдкар (`properties`) |
| «Связи между сущностями» | сайдкар (`$ref`, `array`) |

Остаётся в компаньоне: назначение; стейт-машина (таблица «Состояния и переходы», §5); логические
ограничения (составная уникальность, TTL, state-зависимые обязательности); связи.

> **Плоская data без ER-сущностей** (config-схема — таблица «ключ / тип / значение по умолчанию»,
> без сущностей и связей): свойства идут прямо в `properties` сайдкара. Структура «Сущности /
> Ключевые атрибуты / Связи» — типовой случай доменной data, не единственный; стейт-машины у config
> обычно нет.

## 3. Конвертация логических типов → JSON Schema

| Логический тип legacy (prose) | JSON Schema 2020-12 |
|---|---|
| «строка» / «текст» | `type: string` |
| «целое» | `type: integer` |
| «число» / «дробное» / «сумма» / «дробь» | `type: number` — НЕ сужать до `integer`: голое «число» не гарантирует целочисленность |
| числовое, но не ясно целое или дробное (термин не «целое» и не из строки выше) | `# MIGRATE-TODO: целое или дробное? — legacy: "<цитата>"` |
| «флаг» / «булево» / «признак» | `type: boolean` |
| «дата-время» / «timestamp» | `type: string, format: date-time` |
| «дата» | `type: string, format: date` |
| «UUID» / «GUID» (явно) | `type: string, format: uuid` |
| «идентификатор» (формат не указан) | `type: string` — `format` НЕ подставлять (Telegram id, Message-ID и т.п. — не UUID); неоднозначный формат → `# MIGRATE-TODO: формат идентификатора` |
| «URI» / «URL» | `type: string, format: uri` |
| «список» / «массив» | `type: array` |
| «объект» / «структура» | `type: object` |
| не указан / только проза | `.schema.json`: `"x-migrate-todo": "тип не указан — legacy: \"<цитата>\""` (поле без `type`). `.openapi.yaml`: `# MIGRATE-TODO: тип не указан — legacy: "<цитата>"` |

**Запрещённые СУБД-типы** (в спецификации быть не должно; если встретились — MIGRATE-TODO):
`VARCHAR`, `TEXT`, `INT4`, `BIGINT`, `FLOAT8`, `JSONB`, `TIMESTAMP`, `BOOL`, `SERIAL` и аналоги.

## 4. Конвертация источников маппингов

| Источник legacy (prose) | `source.kind` | `source.ref/path/value` |
|---|---|---|
| `-> data-<slug> (поле X)` | `artifact` | `ref: -> data-<slug>`, `path: $.X` |
| `-> api-<slug> (поле X)` | `artifact` | `ref: -> api-<slug>`, `path: $.X` |
| «контекст приложения», «сессия», «конфиг», «текущий пользователь», «текущее время» | `context` | `path: $.<описание>` |
| «константа», литеральное значение | `constant` | `value: <литерал>` |
| Неоднозначно / смешанный источник | — | `# MIGRATE-TODO: kind неясен — legacy: "<цитата>"` |

### Трансформации

| Трансформация legacy | `transform` / `logic` |
|---|---|
| Нет / identity | `transform` опустить |
| Простое из whitelist (`$uppercase`, `$string`, `$number`, `$round`) | `transform: "<JSONata>"` |
| Словесная с логикой (условия, циклы, ≥2 источника) | `logic: # MIGRATE-TODO: JSONata/alg — legacy: "<цитата>"` |
| `$map`, `$reduce`, `$filter`, `$sort`, lambda, рекурсия | `logic: -> # MIGRATE-TODO: выносить в alg-` |

## 5. Стейт-машина: извлечение из матрицы

> **Только для legacy 2D-матрицы.** Если стейт-машина исходника УЖЕ в целевой ТАБЛИЧНОЙ форме
> (`| Состояние | Событие | → Результат (или ЗАПРЕЩЕНО) | Момент |` — как в `templates/tpl-data-schema.md`,
> раздел «Состояния и переходы»), она уже в целевой форме: перенести as-is, §5-извлечение не
> применять. Ниже — для legacy, где источник оформлен двумерной матрицей.

Канонический источник (legacy) — **матрица**: строки = состояния, столбцы = события, ячейки =
результат перехода или «ЗАПРЕЩЕНО» (мета-модель §6.3.5). Это НЕ список переходов — двумерная таблица.
Целевая форма — не 2D-матрица, а РАЗВЁРНУТАЯ таблица: одна строка на пару (состояние, событие).

Пример матрицы:

```
|              | оплачен         | отменён         |
|--------------|-----------------|-----------------|
| черновик     | → активный      | ЗАПРЕЩЕНО       |
| активный     | ЗАПРЕЩЕНО       | → отменён       |
```

Алгоритм:

1. **Состояние**: метки строк матрицы — дословно, в колонку «Состояние».
2. **Событие**: заголовки столбцов матрицы — дословно, без нормализации, в колонку «Событие».
   Составное («Отмена (клиент/система)») — as-is, НЕ разбивать. Неясный/пустой заголовок →
   `<!-- MIGRATE-TODO: имя события неясно — legacy: "<текст>" -->`.
3. **→ Результат (или ЗАПРЕЩЕНО)**: каждая ячейка матрицы → одна строка целевой таблицы: «→
   состояние» переносится как есть; «ЗАПРЕЩЕНО» → буквально `ЗАПРЕЩЕНО`; пустая ячейка →
   `<!-- MIGRATE-TODO: ячейка пуста --> ` (НЕ трактовать как ЗАПРЕЩЕНО).
4. Декартово произведение состояний × событий покрыто полностью (матрица по convention полна) —
   ровно одна строка целевой таблицы на пару, ни одна не пропускается.
5. **Момент** (при чтении / по расписанию / при записи): если источник называет момент перехода
   (в матрице рядом или прозой) — переносится в колонку «Момент»; не назван — `<!-- MIGRATE-TODO:
   момент перехода не указан -->`.

Результат для примера:

| Состояние | Событие | → Результат (или ЗАПРЕЩЕНО) | Момент |
|---|---|---|---|
| черновик | оплачен | → активный | <!-- MIGRATE-TODO: момент перехода не указан --> |
| черновик | отменён | ЗАПРЕЩЕНО | — |
| активный | оплачен | ЗАПРЕЩЕНО | — |
| активный | отменён | → отменён | <!-- MIGRATE-TODO: момент перехода не указан --> |

## 6. Правила пометки MIGRATE-TODO

**Обязательно помечать:** скаляр `provenance` (намёк на происхождение есть, источник неизвестен
migrate — §1); тип поля (не указан/прозой); секция ошибок (прозой без структуры); `source.kind`
(неоднозначен); `operation` маппинга (не определяется); `transform`/`logic` (словесная логика);
заголовок столбца-события (неясное/пустое имя, §5); строка целевой стейт-таблицы (пустая ячейка
результата, §5); СУБД-тип; тело «Поведение» scn (holding route — §9 п.4); участник scn без slug
(§9); внешний BPMN (§9); правило alg procedural без «иначе» (§10); `**Hit policy:**` decision-table
(не читается — §10); строка decision-table (комбинация не покрыта — §10).

**Никогда НЕ угадывать:** тип поля по умолчанию; имя операции/событие (не нормализовать молча);
провенанс (не из имени файла); `notation` при неоднозначной прозе (фиксировать `yaml-graph` holding,
кандидат только в TODO — §9); `pull-rules` из прозы (§9); участник → cmp/cap по совпадению имён; Hit
policy `unique` по умолчанию (§10); ветку «иначе» (§10).

**Формат пометки — по типу файла:**
- `.openapi.yaml` / YAML-фронтматтер / YAML-блок `.md`: `поле: # MIGRATE-TODO: <что> — legacy: "<цитата>"`
- `.schema.json` (JSON без `#`): `"x-migrate-todo": "<что> — legacy: \"<цитата>\""`; дублировать в report.
- Тело `.md` (вне фронтматтера/fenced — основной формат scn/alg): HTML-комментарий рядом с местом
  неоднозначности: `<!-- MIGRATE-TODO: <что> — legacy: "<цитата>" -->` (не одним общим в конце файла).

## 7. Источники неоднозначности (типовые)

`scope: external` (provided и consumed в одном значении); маппинг «поле ← алгоритм» (kind + нет
JSONata); маппинг без операции (один JSON Pointer в нескольких operationId); контракт прозой (нет
типов/кодов); составной столбец матрицы; пустая ячейка (≠ ЗАПРЕЩЕНО, ≠ переход); «сумма»/«идентификатор»
(точность/формат не указаны); проза без нумерации (порядок неявен — важен или нет?); участник обиходным
именем; «если X» без «иначе»; «Правила» похожи на классификатор, но не таблица; ссылка на внешний BPMN.

## 8. Статусная дисциплина

| Ситуация | Статус |
|---|---|
| Есть ≥1 MIGRATE-TODO | `status: draft` обязательно |
| Нет MIGRATE-TODO (теоретически) | `status: draft` — human-gate всё равно обязателен |
| После human-gate закрыты все TODO + verify прошёл | Владелец вправе `status: actual` |
| Исходник уже `status: deprecated` | ОСТАЁТСЯ `deprecated` — migrate его НЕ поднимает в `draft`; переразлагается только форма, статус не трогается |

migrate **никогда** не ставит `actual` и **никогда** не штампует `meta_model_version` (это сертификация
по состоянию — `../../masterspec/references/migration.md`).

## 9. Сценарии: определение notation по содержимому

Legacy-scn — «Последовательность шагов» (нумерованный happy path) + «Ветвления» (прозой). Legacy не
различал нотаций. **На практике большинство legacy-сценариев — проза без диаграмм.** Диаграммный путь
редкий, но проверяется первым — он однозначен.

### Порядок детекции (первое совпадение решает)

1. **Fenced `plantuml` с sequence-конструкциями** (`participant`/`actor`, стрелки) **ИЛИ** fenced
   `mermaid` с `sequenceDiagram` → `notation: sequence`, диаграмма as-is, участники → секция
   «Участники». Mermaid-`sequenceDiagram` проверяется здесь, ДО шага 2 — иначе рискует уйти в workflow.
2. **Fenced `mermaid` с `flowchart`/`graph`** (не `sequenceDiagram` — тот уже отсёкся шагом 1) →
   `notation: workflow`, as-is, узлы → «Участники».
3. **Строгая markdown-нумерация + полные ветвления — основной практический случай.** Признаки:
   «Последовательность шагов» строго `1.`/`2.`/`3.` (один пункт = одно действие; слитный compound-пункт
   = один узел); каждое условие «Ветвлений» имеет ОБА исхода описанными («если X — иначе Y»), ни одна
   ветка не опущена. Прозаические связки («затем», «после этого») без markdown-нумерации — НЕ основание
   (см. п.4). → `notation: yaml-graph`, детерминированная конвертация (правило ниже).
4. **Нет строгой нумерации ИЛИ есть неполное ветвление — штатный holding route (не деградация).**
   Действие: граф НЕ строится (ни целиком, ни частично); проза переносится **as-is** в «Поведение»;
   `notation` = фиксированное holding-значение `yaml-graph` (дефолт реестра, не подбор «по смыслу» —
   таблица «тип поведения → нотация» предназначена для `gen`, при миграции НЕ консультируется);
   почему `yaml-graph` — единственная нотация без обязательной машинно-парсимой формы тела (O1 не
   требует парсимого блока, в отличие от sequence/workflow/bpmn), значит прозаическое тело не даёт
   синтаксически невалидный артефакт; несоответствие полноты ловится позже O4a под human-gate; рядом —
   `<!-- MIGRATE-TODO: переоформите по нотации (кандидат: <X>) -->`, кандидат ТОЛЬКО в TODO и report,
   во фронтматтер не попадает.
5. **`bpmn` не порождается никогда** — legacy не имел машинных `.bpmn`. Ссылка на внешний BPMN → as-is +
   `<!-- MIGRATE-TODO: legacy ссылался на внешний BPMN — решите: sidecar-пара (notation: bpmn +
   sidecar_format/sidecar) или другая нотация -->`; `notation` определяется по п.4.

### Запрет на синтез pull-rules

`pull-rules` **никогда не выводится** из прозы детерминированно, даже если текст похож на
параллельную/вытягивающую логику. Причина: `pull-rules` — отдельная предметная модель (участник тянет
что/откуда/при каком условии, атрибут конкуренции, арбитраж пересечений, инварианты потока), этих
категорий в прозе как явных фактов нет. Восстановить их детерминированно = домыслить структуру. Похоже
на pull-логику → правило 4 (as-is + TODO с рекомендацией рассмотреть `pull-rules`), не автовыбор.

### Правило конвертации шагов → yaml-graph (только для случая 3)

- Каждый нумерованный шаг → один узел; текст дословно (допустима грамматич. нормализация формы).
- Ссылка `-> cmp-<slug>/cap-<name>` (`-> api-<slug>` для внешнего) при шаге — инлайн в узел as-is.
- Шаг без ссылки — см. «Семантический инвариант» (не выдумывать маппинг).
- «Если X — A, иначе — B» → узел-развилка с двумя рёбрами к A и B, оба исхода дословно.
- «Если X — A» БЕЗ «иначе» **в этом маршруте не встречается**: детекция (п.4) отсеивает такой сценарий
  в holding route ДО конвертации. Граф с недостающей веткой `else` и TODO вместо неё **не строится**.
- Один шаг, слитно описывающий несколько внешних вызовов, → **один** составной узел, как в legacy
  (границы вызовов не зафиксированы явно, разделение — домысел). Наблюдение — в report, TODO не нужен.

### Семантический инвариант — участники → cmp/cap

Ссылка `-> cmp-`/`-> cap-` при шаге/участнике — переносится as-is. Участник прозой без slug («служба
доставки», «менеджер очереди») → `<!-- MIGRATE-TODO: участник '<имя>' не резолвится в cmp/cap -->`.
Маппинг на конкретный slug **не подбирать** — даже интуитивно очевидное совпадение имён (участник
«Роутер уведомлений» и `cmp-notification-router`) не даёт права на автопривязку; решение — за владельцем.

## 10. Алгоритмы: определение form по содержимому

Legacy-alg — «Правила» пронумерованным перечнем словами (legacy не различал форм). На практике
большинство — процедурная проза; табличные классификаторы (именно как таблица «условие → результат»)
реже.

### Критерий «уже таблица решений»

`form: decision-table` ТОЛЬКО если «Правила» уже оформлены markdown-таблицей с колонками-условиями и
последней колонкой-результатом — таблица уже есть, migrate её переразлагает в DMN, не синтезирует форму.

Иначе — `form: procedural`. Безопасный дефолт, не домысел: проза переносится as-is.

**Кандидат в decision-table** (классификатор по смыслу, но НЕ таблицей — в т.ч. перечислимый набор
взаимоисключающих условий прозой «если A и B — X; если A и не B — Y») → **не конвертировать**:
`form: procedural` + `<!-- MIGRATE-TODO: кандидат в decision-table — рассмотрите переоформление -->`.
Даже очевидный классификатор — синтез формы, если не оформлен таблицей в самом legacy.

### Конвертация в DMN-таблицу (когда критерий выполнен)

- Каждое условие legacy → колонка входа; результат → последняя колонка. Каждая строка → строка таблицы,
  содержание без изменения.
- **Hit policy** проставляется ТОЛЬКО если очевидна: строки взаимоисключающи и полны (или есть explicit
  default) → `unique`; из текста читается приоритет по порядку («сначала…, иначе…») → `first`; явный
  приоритет, отличный от порядка строк → `priority`; иначе → `**Hit policy:** <!-- MIGRATE-TODO: hit
  policy не очевидна — unique/first/priority? -->`. НЕ проставлять `unique` по умолчанию: неверная hit
  policy меняет семантику разрешения конфликтов. Незакрытый TODO вместо Hit policy — блокер O1 при verify.
- Непокрытая комбинация входов (нет строки и нет default) → `<!-- MIGRATE-TODO: комбинация <условия>
  не покрыта -->` (НЕ добавлять default-строку от себя — изобретение решения).

### Полнота procedural

Правило «если X — <действие>» без явной «иначе» (или без «иначе — невозможно, потому что…») →
`<!-- MIGRATE-TODO: ветка "иначе" не описана — не домысливать -->`, ветка не дописывается.

### Запреты

- Не подбирать Hit policy «на глаз» — пустое-с-TODO лучше неверного.
- Не превращать процедурную прозу в decision-table, даже если похоже на классификацию.
- Не добавлять default-строку/ветку «иначе» от себя.
