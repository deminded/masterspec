# Edge cases: нестандартные ситуации

Справка по ситуациям, которые не укладываются в прямой линейный прогон шагов.

## Отсутствует design.md

`design.md` — **опционален**. Для простых CR (однострочная правка валидации, текстовая доработка, изменение одного параметра конфига) он может отсутствовать.

- Если `design.md` есть — прочитай на шаге 5 и используй как контекст.
- Если нет — пропусти шаг 5, используй только `change.md` и `tasks.md`.
- Если на шаге 7 упёрся в **архитектурный вопрос** (новый модуль, новая сущность, новое взаимодействие между компонентами), а `design.md` отсутствует — поставь паузу и предложи запустить скилл `masterspec-impl-plan`. Не угадывай решение.

## Отсутствует tasks.md

**Блокер.** Без `tasks.md` скиллу нечего реализовывать. Сообщи пользователю об ошибке, предложи сначала запустить скилл `masterspec-impl-plan` (или `masterspec-propose`, если change вообще не создан). Выйди.

## Частично выполненные задачи (fluid workflow)

Скилл можно вызывать, когда часть задач уже отмечена `- [x]`.

- Не переотмечай уже выполненные.
- Начни с первой встреченной `- [ ]` в пронумерованной секции (см. `runbook.md`, шаг 6).
- Покажи актуальный прогресс: «Прогресс: N/M задач выполнено» перед стартом.
- Если статус change.md = `В реализации` — ничего не меняй, просто продолжай.

## Задача непонятна

Поставь паузу перед выполнением. Спроси:

- Какой конкретно артефакт должен получиться?
- Какой критерий «готово» — есть ли acceptance в design.md / change.md?
- Какие файлы трогать?

Не пытайся «догадаться» и реализовать приблизительно. Лучше пауза на 30 секунд, чем код, который потом придётся удалять.

## Реализация выявила проблему в design.md

Если `design.md` есть и в нём ошибка / неполнота — поставь паузу. Предложи два варианта:

1. Поправить `design.md` вручную и продолжить.
2. Пересоздать секцию через `masterspec-impl-plan`, если правка существенная.

Не правь `design.md` сам в рамках этого скилла — он предназначен для реализации, не для проектирования.

## Реализация упёрлась в архитектурный вопрос, design.md отсутствует

Поставь паузу. Не выдумывай архитектуру. Предложи запустить скилл `masterspec-impl-plan` — он дополнит change файлом `design.md`, после чего можно вернуться в `masterspec-implement`.

## Конфликт tasks.md ↔ change.md

Если описание задачи в `tasks.md` противоречит сценарию в `change.md` (например, tasks.md говорит «добавить поле X», а change.md описывает поведение для поля Y) — **не выполняй**. Поставь паузу, попроси пользователя разрешить конфликт: либо обновить change.md, либо переформулировать задачу, либо пересоздать change через `masterspec-propose`.

## Статус change.md некорректный

См. `runbook.md`, раздел «Статусы change.md». Для значений, не входящих в матрицу (опечатка, отсутствует поле) — AskUserQuestion с подтверждением перехода в `В реализации`.

## Верификация технически невозможна

Нет окружения, нет доступа к внешним сервисам, CI недоступен и т. п. Действия:

1. Не объявляй change завершённым.
2. В `Output On Completion` вместо списка зелёных команд напиши: «Автоматическая верификация не выполнена — <причина>».
3. Не ставь в tasks.md галочку напротив «верификация» (если она там есть в нечисловой секции — её трогать и так нельзя).

## Прерывание пользователем

Если пользователь прервал в середине задачи:

- Если задача реализована частично и не работает — верни `- [ ]` в tasks.md.
- Если реализована и компилируется — оставь `- [x]`.
- В следующий вызов скилл продолжит с первой `- [ ]` (fluid workflow).

## Что скилл НЕ делает

- Не меняет `change.md`, кроме строки `> **Статус**:`.
- Не меняет `design.md`.
- Не меняет чеклист верификации, статус выполнения, лог изменений в `tasks.md` — только чекбоксы `- [ ]` / `- [x]` в пронумерованных секциях.
- Не создаёт новых артефактов фабрики (в `masterspec/01-*/02-*/03-*/04-*`).
- Не архивирует change — это работа `masterspec-archive-change`.
- Не вливает change в артефакты фабрики — это работа `masterspec-apply-change` (мерж diff-блоков + копирование `new/`).
